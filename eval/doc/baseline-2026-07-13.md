## 1. 执行摘要

cogent 评测体系已从「4 个 Go 单轮任务」升级为**六维度 native 种子集 + 四个主流基准 Adapter** 的分层体系。
本报告是**首份覆盖全套件的合并基线**，锚定两类可信数据：

1. **harness 正确性基线**（本机可复现、零 LLM 成本）：native 可解性双校验 **9/9**、评测层单测 + 四个 Adapter 的
   fixture/oracle 自证**全绿**——证明「加载 → 隔离 → 判定 → 归档」这条评测管线在不花钱前提下端到端正确。
2. **agent 能力基线**（真实 LLM 跑分，引用既有 + 待规模化）：native 满分 **9/9**（deepseek-v4-pro）、
   polyglot go 子集 **3/3**；全量规模化跑分（跨 Adapter）为运行侧一次性成本（见 §5）。

> 定位：**规模由 Adapter 承载**（EVAL_SPEC §4.5「hard 体量多经 Adapter 引入」），native 负责六维度覆盖。
> 数据集齐备时全套件可达 **600+ 可跑样本**（§2）。

## 3. harness 正确性基线（零 LLM 成本，本机实测全绿）

不花 LLM/无 Docker/无网络即可复现，锚定「评测管线本身正确」。实测于 2026-07-13（go1.26 via `GOTOOLCHAIN=auto`）：

| 校验 | 结果 | 复现命令 |
| --- | --- | --- |
| native 可解性双校验 | **9/9 通过**（初始必失败 / oracle 必通过） | `bash eval/bin/eval_selfcheck.sh` |
| 评测层 + 四 Adapter 单测/fixture | **全绿** | `GOTOOLCHAIN=auto go test ./internal/eval/...` |
| swebench fixture oracle | **go + python(pytest) 双绿** | `go test ./internal/eval/adapter/swebench -run TestAdapterOracleFixture` |
| terminalbench fixture oracle | **通过** | `go test ./internal/eval/adapter/terminalbench -run TestAdapterOracleFixture` |
| polyglot 六语言 oracle | 需 `POLYGLOT_DIR`（历史六语言全绿） | `POLYGLOT_DIR=... go test ./internal/eval/adapter/polyglot -run TestOracleSolutionsPass` |

> 这些 fixture/oracle 用**数据集参考解代替 agent 产出**、走真实 `adapter+verifier` 代码路径，
> 在不花 LLM 钱前提下证明数据集加载、工作区隔离、隐藏测试 pristine、判定与归档全部正确。

## 5. 规模化跑分模板（供 keyed 运行回填）

统一入口 `cogent eval run`（省成本用便宜模型 + 收紧 Budget）。**LLM key 仅走环境变量**（不落盘）：

```bash
export GOTOOLCHAIN=auto
# deepseek 必须显式设端点，否则默认打 OpenAI → 401 空转烧预算
export COGENT_LLM_BASE_URL=https://api.deepseek.com/v1 COGENT_MODEL=deepseek-chat

# native 全量（含运行期三任务），并发 3
cogent eval run --n-concurrent=3 --max-iterations=8 --max-cost=1 --out=eval-artifacts/native/report.md

# polyglot 取子集
cogent eval run --dataset=polyglot --polyglot-dir=~/.cache/cogent-eval/polyglot-benchmark \
  --language=go --limit=10 --n-concurrent=2

# swebench Lite 子集（跑完自动导出 predictions.jsonl 交官方 sb-cli/Docker 判定）
cogent eval run --dataset=swebench --swebench-file=lite.jsonl --swebench-repos=/repos --limit=20

# terminalbench 子集（纯脚本类任务；容器依赖类需官方 Harbor）
cogent eval run --dataset=terminalbench --terminalbench-dir=/path/to/terminal-bench --limit=10

# 回归对比（检出退化退出码 3）
cogent eval compare --base=baseline.json --head=report.json --fail-on-regress
```

预期指标口径见 EVAL_SPEC §6.6（Task Success / Pass@1 / Pass@k / Avg Iterations-to-Green / Avg Cost / Early-Stop / Injection Resist / 分档通过率）。

## 6. 已知局限与后续

- **规模化 agent 跑分是运行侧一次性成本**：需数据集（polyglot clone / SWE-bench 导出 / Terminal-Bench clone）、
  多语言工具链或 Docker、以及 LLM 预算；本报告已锚定 harness 正确性 + 既有 agent 基线，规模化数值随 keyed 运行回填。
- **`repo-root` 自宿主任务**（add_find_files_tool / session_root_span）运行器首版不跑（需干净 clone/worktree）。
- **Terminal-Bench 模式 B** 仅适用于纯文件系统/脚本类任务；依赖容器的任务需官方 Harbor（模式 A，需 Docker）。
- **后续 E5-2**：夜间/手动 CI 跑分（非门禁，质量基线），配 `eval compare` 防「改好一个改坏三个」。
