## cogent run 一秒退出问题（已修复 ✅）
- 现象：`cogent run` 打印 banner + `you>` 后立刻退出（无用户输入）。
- **真正根因**：`rawInput.fill()` 中 `os.File.Read` 在 VMIN=0/VTIME=1 终端超时返回 n=0 时，Go runtime 会附带 `io.EOF`；`fill()` 未特判这个 pseudo-EOF，直接冒泡给 `readByte` → `decodeKey` 返回错误 → `readLine` 误判 raw 不可用 / 之前版本直接返回 `ok=false` → REPL 退出。
- **修复**：
  1. `lineeditor.go` `fill()`：对 `m==0 && err==io.EOF` 视为正常 VTIME 超时（`return false, nil`），只有 non-EOF 真实错误才冒泡。
  2. `lineeditor.go` `readLine()`：改为三态返回 `(line, ok, usable)`，区分"用户主动退出"与"环境不支持 raw"。
  3. `prompter.go` `inputReader.next()`：`usable=false` 时懒回退到逐行读取（降级），而非退出 REPL。
- 验证：expect 真实 pty 下 raw 行编辑正常工作，输入 `exit` 逐字符回显 + 干净退出；非 TTY 管道 `printf 'exit\n'|cogent run` 走逐行路径也正常。

## REPL SIGINT 管理重构（已完成 ✅）
发现上面的双击 Ctrl-C 逻辑只在 raw mode 内有效（捕获 0x03 字节），但 `readLine()` 结束后 termios 恢复 cooked mode 时按 Ctrl-C 仍会产生系统 SIGINT → `main.go` 的 `signal.NotifyContext` cancel ctx → 程序直接退出。

修复方式：`commands.go` 的 `inputLoop` 新增本地 SIGINT 管理：
- `signal.Notify(sigCh, SIGINT)` 接管 SIGINT（覆盖 main.go 的 NotifyContext）
- 等待输入前 `drainSignals(sigCh)` 排空残留信号
- 模型执行使用 `runTurnInterruptible`：per-turn ctx + goroutine 监听 sigCh → cancel 本轮
- SIGINT 仅中断当前 turn（打印 `[interrupted]`），REPL 循环继续
- 只有 raw mode 内双击 Ctrl-C 或 `exit` 命令才真正退出
- 验证：expect 4 项测试通过（单次不退出、有文本清行、双击退出、模型执行中 Ctrl-C 中断 turn 后 REPL 继续）。
- **追加修复**：发现 `signal.Notify` 是多播的——`inputLoop` 的 Notify 和 `main.go` 的 `NotifyContext` 同时收到 SIGINT，后者仍会 cancel 顶层 ctx 导致退出。修复：在 `inputLoop` 开头加 `signal.Reset(syscall.SIGINT)` 解除 NotifyContext 对 SIGINT 的监听，让 inputLoop 独占管理 SIGINT。现在实测 `kill -INT` 也只中断当前 turn 不退出 REPL。
