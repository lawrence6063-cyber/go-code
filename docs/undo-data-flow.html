Cogent Undo 功能 数据流说明文档

基于 243a1b4 提交（add undo），梳理 /undo 一键回退功能的完整数据流。

1. 功能概览

Undo 为 REPL 模式提供了「一键撤销上一轮」能力，覆盖三个层面的状态回退：

工作区回退 snapshot

基于 git stash create/apply，精确恢复到上一轮 agent 执行前的文件状态。

  • 每轮开始前拍摄工作区快照（stash SHA）
  • undo 时先 git checkout -- . && git clean -fd，再 git stash apply
  • 非 git 仓库时跳过文件恢复，仅回退消息

消息历史回退 engine

从 engine.msgs 中截断上一轮追加的所有消息，使 LLM 不再"记得"已撤销的改动。

  • 每轮开始前记录 len(e.msgs) 作为截断点
  • undo 时 e.msgs = e.msgs[:snap.msgIndex]
  • 连续 undo 支持逐轮回退（栈式快照）

会话持久化 session

向 transcript 追加 "undo" 事件，使 cogent resume 能正确重建回退后的状态。

  • undo 事件 payload 含 RevokedUUIDs
  • resume 重建时遇 undo 事件，截断对应 UUID 之后的消息
  • 落盘失败仅告警，不阻断 undo

2. 核心数据结构

2.1 turnSnapshot — 快照栈帧

每轮对话开始前压入 engine.turnSnapshots 栈，undo 时弹出并据此回退。

// internal/engine/engine.go
type turnSnapshot struct {
    msgIndex int    // 该轮开始前 e.msgs 的长度（截断点）
    stashID  string // 该轮开始前的工作区快照标识（git stash SHA，可空）
    lastUUID string // 该轮开始前的 session 事件链尾 UUID
}
字段类型说明来源
msgIndexint消息切片截断下标len(e.msgs)
stashIDstringgit stash create 返回的 SHA；无改动时为空Snapshotter.Take()
lastUUIDstringsession 事件链尾 UUID，用于 undo 事件定位截断点e.lastUUID

2.2 undoPayload — 持久化事件载荷

// internal/engine/undo.go
type undoPayload struct {
    RevokedUUIDs []string `json:"revoked_uuids"` // 被撤销的事件 UUID 列表
}

2.3 UndoResult — 返回给 CLI 的结果

// internal/engine/engine.go
type UndoResult struct {
    Summary        string // 被撤销轮次的摘要（user 消息前 50 字符）
    HasFileChanges bool   // 是否恢复了工作区文件改动
    RemovedCount   int    // 被移除的消息数量
}

// engine.Run 中的关键调用顺序
func (e *engine) Run(ctx context.Context, task string) (<-chan types.StreamEvent, error) {
    // ...
    go func() {
        defer close(out)
        safeGo(e.onPanic(ctx, out), func() {
            e.takeTurnSnapshot(ctx)       // ← 步骤 2-4：采集快照
            userMsg := types.Message{Role: types.RoleUser, Text: task}
            e.msgs = append(e.msgs, userMsg)
            e.record(ctx, userMsg)        // ← 步骤 6：持久化
            e.step(ctx, out)              // ← 步骤 5：ReAct 循环
        })
    }()
    return out, nil
}

4. 回退数据流：/undo 执行路径

1

用户输入 /undo → handleUndo

inputLoop 检测到 "/undo"，调用 handleUndo(ctx, eng)

2

engine.Undo 弹出快照栈顶

snap := e.turnSnapshots[len-1]，然后 e.turnSnapshots = e.turnSnapshots[:len-1]

3

截断消息历史

4

回退 session 事件链指针

e.lastUUID = snap.lastUUID — 使后续新事件接在 undo 截断点之后

5

Snapshotter.Restore 恢复工作区

若 snapshotter != nil && IsGitRepo()，执行 discardAll（checkout + clean）再 git stash apply snap.stashID

6

recordUndoEvent 追加 undo 事件

向 session transcript 追加 {Type:"undo", Payload:{RevokedUUIDs:[snap.lastUUID]}}，ParentUUID 指向当前 e.lastUUID

7

CLI 打印撤销摘要

handleUndo 根据 UndoResult 打印：摘要文本 + 是否恢复文件 + 移除消息数

// engine.Undo 核心逻辑
func (e *engine) Undo(ctx context.Context) (*UndoResult, error) {
    if len(e.turnSnapshots) == 0 {
        return nil, ErrNothingToUndo
    }
    snap := e.turnSnapshots[len(e.turnSnapshots)-1]
    e.turnSnapshots = e.turnSnapshots[:len(e.turnSnapshots)-1]

    removed := e.msgs[snap.msgIndex:]
    summary := buildUndoSummary(removed)     // 第一条 user 消息前 50 字符
    e.msgs = e.msgs[:snap.msgIndex]          // 截断消息
    e.lastUUID = snap.lastUUID               // 回退事件链指针

    hasFileChanges := false
    if e.snapshotter != nil && e.snapshotter.IsGitRepo() {
        if err := e.snapshotter.Restore(ctx, snap.stashID); err != nil {
            slog.Warn("undo: workspace restore failed", "err", err)  // 解耦：仅告警
        } else {
            hasFileChanges = true
        }
    }
    e.recordUndoEvent(ctx, snap)             // 持久化 undo 事件

    return &UndoResult{Summary: summary, HasFileChanges: hasFileChanges, RemovedCount: len(removed)}, nil
}

5. Resume 重建数据流

当用户执行 cogent resume <session-id> 时，undo 事件在消息重建中被正确处理：

1

session.Load 加载全部事件

从 transcript 文件读取全部 session.Event，包含 user/assistant/tool_result 和 undo 事件

2

rebuildMessages 逐事件重建

维护 uuidIndex map[string]int，将每个消息事件追加到 entries

3

遇到 undo 事件 → 截断

解析 undoPayload.RevokedUUIDs[0]，在 uuidIndex 中找到截断位置，entries = entries[:idx]

4

filterUnresolvedToolUses 修复配对

截断可能破坏 tool_use/tool_result 配对，剥离无结果的 tool_use 和孤立的 tool_result

5

注入 system prompt + continue

重建的消息列表接在 system prompt 之后，再注入 "Continue from where you left off." 续跑

// rebuildMessages 中 undo 事件处理核心
for _, ev := range events {
    lastUUID = ev.UUID
    if ev.Type == "undo" {
        var payload undoPayload
        json.Unmarshal(ev.Payload, &payload)
        if len(payload.RevokedUUIDs) > 0 {
            cutUUID := payload.RevokedUUIDs[0]
            if idx, ok := uuidIndex[cutUUID]; ok {
                entries = entries[:idx]  // 截断到被撤销 UUID 之前
            }
        }
        continue
    }
    // 正常消息事件 ...
    uuidIndex[ev.UUID] = len(entries) + 1
    entries = append(entries, msgEntry{msg: msg, uuid: ev.UUID})
}

9. 边界与容错策略

无快照可撤 ErrNothingToUndo

turnSnapshots 为空时（刚启动 / 已全部撤销），返回 ErrNothingToUndo，CLI 打印"没有可撤销的轮次"。

非 Git 仓库

Snapshotter.IsGitRepo() 返回 false 时跳过文件恢复，仅回退消息历史。CLI 提示"本轮无文件改动"。

工作区恢复失败

Snapshotter.Restore 出错时仅 slog.Warn 告警，消息历史仍正常回退。消息回退与文件回退解耦，避免状态不一致扩大。

Session 落盘失败

recordUndoEvent 中 session.Append 失败仅告警，不阻断 undo 操作本身（与现有 session 容错策略一致）。

Resume 遇损坏 undo 事件

rebuildMessages 解析 undo payload 失败时 slog.Warn 并 continue，不影响后续事件重建。

设计原则：三层回退（消息、文件、持久化）互相解耦，任何一层失败不影响其他层完成。核心不变量是 e.msgs 截断必须成功（内存操作），文件和持久化均为增强而非正确性前提。
