01架构升级全景

Phase 2 的执行路径是 engine.executeTools → 逐个 runOne（简单 for 循环）。Phase 3 在中间插入了一个可独立单测的调度层 orchestrate，并补全了 sandbox 的完整执行能力：

关键取舍：为什么要分层？

设计动机：orchestrate 只管"怎么排列和调度"，不感知具体工具怎么执行、不感知 StreamEvent；per-tool 执行（mode 校验、事件发送、未知工具规范化、tool.call span）仍在 engine 的 runOne，通过回调 RunFunc 注入。

好处：① 可独立单测（无需 fake LLM/fake channel）；② 依赖方向单向（engine → orchestrate，不反向）；③ 两侧行为不漂移——engine 的 runOne 是唯一定义"执行一个工具"的地方。

02分批算法 PartitionBatches

模型一次可能返回多个 tool_use（如同时读 3 个文件 + 改 1 个文件）。PartitionBatches 做一次线性扫描，把连续的"并发安全"工具合并成并发批，其余各自独占串行批：

internal/orchestrate/orchestrate.go// PartitionBatches 把工具调用序列切分为并发/串行批：向前扫描，
// 连续的并发安全工具合并进同一并发批，其余各自独占一个串行批。
func PartitionBatches(blocks []types.ToolUseBlock, pool tool.Pool) []Batch {
    var batches []Batch
    var concurrent []types.ToolUseBlock
    flush := func() {
        if len(concurrent) == 0 { return }
        batches = append(batches, Batch{Concurrent: true, Blocks: concurrent})
        concurrent = nil
    }
    for _, b := range blocks {
        if isConcurrencySafe(b, pool) {
            concurrent = append(concurrent, b)
            continue
        }
        flush()
        batches = append(batches, Batch{Concurrent: false, Blocks: []types.ToolUseBlock{b}})
    }
    flush()
    return batches
}

示例：模型返回 [read, read, write, read] → 分批为 [c2, s1, c1]（c=并发批+大小，s=串行批）。

fail-closed：未知即不安全

isConcurrencySafefunc isConcurrencySafe(block types.ToolUseBlock, pool tool.Pool) bool {
    if pool == nil { return false }     // nil 池 → 全串行
    t, ok := pool.Get(block.Name)
    if !ok { return false }              // 未知工具 → 串行
    return t.IsConcurrencySafe(block.Input)
}
工程要点：fail-closed 把"忘记标记并发安全"从潜在 bug 变成"退化为全串行"（正确但慢），而不是"默认并发"（快但可能写竞态）。最坏情况只是性能退化，永远不会出数据一致性问题。

03并发执行 Run

Run 逐批执行，每批起一个 tool.batch span，按 Concurrent 选择串行或并发：

internal/orchestrate/orchestrate.gofunc Run(ctx context.Context, batches []Batch, run RunFunc, tracer observe.Tracer) []types.Message {
    results := make([]types.Message, 0, total)
    for _, b := range batches {
        if ctx.Err() != nil { break }  // 取消时跳过剩余批次
        results = append(results, runBatch(ctx, b, run, tracer)...)
    }
    return results
}

errgroup + 索引槽位写法

并发批的核心难题：如何并行执行但保序返回？答案是预分配 slots 数组，每个 goroutine 写自己的索引位——无锁、无 append 竞态、结果天然有序。

runConcurrent — errgroup 槽位写法func runConcurrent(ctx context.Context, blocks []types.ToolUseBlock, run RunFunc) []types.Message {
    slots := make([]types.Message, len(blocks)) // 预分配，每人写自己的位
    done  := make([]bool, len(blocks))
    g, gctx := errgroup.WithContext(ctx)
    for i := range blocks {
        i := i                                  // 闭包捕获循环变量
        g.Go(func() error {
            if gctx.Err() != nil { return gctx.Err() }
            slots[i] = run(gctx, blocks[i])     // 写自己的索引位 → 无竞态
            done[i] = true
            return nil
        })
    }
    _ = g.Wait()
    out := make([]types.Message, 0, len(blocks))
    for i := range slots {
        if done[i] { out = append(out, slots[i]) }  // 滤掉未执行的空槽
    }
    return out
}
不变量 — function calling 配对完整性：ctx 取消时，尚未执行的工具不会产出零值 Message{}——通过 done[i] 标记过滤。这保证了回流给模型的 tool_result 数量 ≤ tool_use 数量，API 不会报 400。
为什么 errgroup 不取消兄弟？工具错误已被 engine 的 runOne 规范化为 ToolResult{IsError:true}（非 Go error），所以 errgroup 中的 goroutine 永远 return nil，仅在 ctx 被外部取消时中断。单个只读工具失败不应阻止兄弟完成——它们本来就互不影响。

04engine 接入：3 行改造

engine 侧的变更极小——只是把 for 循环替换为 orchestrate 调用 + 注入回调：

    ctx context.Context,

工程要点（STAR）：「Phase 2 执行是串行 for 循环；Phase 3 我把执行职责分离到 orchestrate 包，engine 只需注入一个回调。改造仅 3 行有效代码变更，但获得了并发提速 + 可独立单测的调度核心，且零行为回归——因为 per-tool 逻辑一字未改。」

05sandbox 四层纵深防御

沙箱的安全模型是"不依赖模型善意"的确定性防线，分四层递进：

1
路由（ShouldSandbox）

决定命令是否需要受限环境。Enabled=false 或命中 ExcludedCommands → 继承宿主环境；否则走受限 PATH/HOME/无密钥。

2
危险命令拦截（isDangerous）

内建 IsDangerousCommand（rm -rf /、mkfs、fork bomb、curl|sh 等）+ 配置 DeniedRules 扩展。命中 → 返回 ErrDangerousCommand。无论是否受限环境均生效。

3
受限执行（runCommand）

exec.CommandContext(ctx+timeout, "bash", "-c", cmd)、Dir=WorkRoot、ShouldSandbox 时用最小环境变量。stdout/stderr 分离捕获。

4
执行后清理（scrubBareGitRepoFiles）

exec 前快照顶层；exec 后若 {HEAD, objects, refs} 签名全部为本次新建，移除本次新增的 bare-repo 伪造产物。既有 .git 绝不触碰。

执行后清理：精准打击、零误删

对抗"让 git init --bare . 在工作区伪造 bare repo，后续 git 操作劫持历史"攻击向量：

scrubBareGitRepoFiles 逻辑// 仅当 {HEAD,objects,refs} 签名全部为本次新建时，清理伪造产物
func (s *sandbox) scrubBareGitRepoFiles(before map[string]bool) {
    if !s.bareRepoFreshlyCreated(before) { return }
    for _, name := range bareRepoArtifacts {
        if before[name] { continue }    // 既有条目，绝不删除
        path := filepath.Join(s.cfg.WorkRoot, name)
        if isNewlyCreated(path, before, name) {
            _ = os.RemoveAll(path)
        }
    }
}
安全保证：仅删除本次新建且构成完整 bare-repo 签名的产物。用户既有的 .git/、普通文件（如 HEAD 恰巧存在）一律不碰。签名不完整（如只有 HEAD 无 objects/refs）也不触发清理。

internal/tool/bash.go — Call（重构后）func (t *bashTool) Call(ctx context.Context, input json.RawMessage, _ ProgressSink) (types.ToolResult, error) {
    cmd, err := parseCommand(input)
    if err != nil { return errorResult(err), nil }

    ctx, end := t.tracer.Start(ctx, "sandbox.exec",
        observe.Attr{Key: "command.first", Value: firstWord(cmd)})
    res, execErr := t.sb.Exec(ctx, cmd)
    end(execErr)

    if errors.Is(execErr, sandbox.ErrDangerousCommand) {
        return types.ToolResult{Content: "dangerous command blocked", IsError: true}, nil
    }
    return formatExecResult(res), nil
}
双重兜底：CheckPermission 里先做一次 IsDangerousCommand（Guard 拦截 → deny），即使绕过 Guard（如测试直调 Call），Sandbox.Exec 内还有第二道确定性拦截。安全防线不依赖调用方的善意。

orchestrate_test.go — 并发真并行断言（精简）func TestRun_ConcurrentActuallyParallel(t *testing.T) {
    var inFlight, maxInFlight int32
    run := func(_ context.Context, block types.ToolUseBlock) types.Message {
        cur := atomic.AddInt32(&inFlight, 1)
        // ... CAS 更新 maxInFlight ...
        time.Sleep(10 * time.Millisecond)

    }

    Run(context.Background(), batches, run, testTracer())

        t.Errorf("not parallel")
    }
}

10设计要点回顾

🎯 fail-closed 始终安全

未知工具 → 串行；未知命令 → 受限执行；ExcludedCommands 仅是便利豁免，非安全边界。最坏退化为最安全行为。

📐 依赖方向单向

engine → orchestrate → {tool, observe, errgroup}。orchestrate 不反向依赖 engine、不感知 StreamEvent，可独立单测。

🔒 确定性安全防线

危险命令拦截/路径校验/控制面写禁止不依赖"模型善意"。双重兜底（Guard + Sandbox）确保无论调用路径如何都不越界。

🧩 职责分离 = 可测性

分批归 orchestrate、执行归 engine.runOne、安全归 sandbox。每一层可独立 fake + 表驱动测试，无需搭全栈。

🔗 零值不配对

取消时不产出零值 Message → function calling 配对始终完整 → 不触发 API 400。这是容易被忽视但致命的不变量。

🧹 精准清理不误删

scrub 仅移除满足完整签名的本次新建物。签名不全不删、既有条目不删、用户文件不删。三重保险。
