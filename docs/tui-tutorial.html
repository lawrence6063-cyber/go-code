01一张图看懂：三大数据流域

整个 TUI 可以拆成三条相互独立又首尾相接的数据流。理解了这三条，就理解了全部：

  • 输入域：用户按键 → 解析 → 状态机 → 补全/历史 → 重绘可编辑区，最终吐出一整行文本。
  • 输出域：引擎把执行过程编码成事件流，渲染器把它变成流式文字、彩色 Diff、加载动画。
  • HITL 域：当工具要做危险操作，执行被中断、把决策权交回人手，再按裁决续跑。

02模块地图：谁负责什么

TUI 的实现横跨 cmd/cogent（终端 I/O 与胶水）和 internal（可复用、可单测的纯逻辑）两层。刻意的分层是为了让最难测的终端部分尽量薄。

cmd/cogent/ ├── lineeditor.go # 行编辑器：editorCore 状态机 + lineEditor 终端渲染（883 行） ├── lineeditor_raw.go # enterRaw / restore：进出终端 raw 模式 ├── lineeditor_ioctl_darwin.go # termios ioctl 常量（macOS） ├── lineeditor_ioctl_linux.go # termios ioctl 常量（Linux） ├── streamview.go # streamRenderer：消费事件流 + spinner + 结果折叠 ├── menu.go # runMenu：raw 单选菜单组件（DECSC/DECRC 重绘） ├── prompter.go # inputReader 统一输入 + cliPrompter 权限 HITL ├── statusbar.go # statusBar：模型 / token / 成本常驻栏 └── slashcmd.go # 斜杠命令注册与分发 internal/ ├── completion/ # @ 文件 / 斜杠补全：候选来源、解析、模糊排序 │ ├── provider.go # git ls-files + 5s TTL 缓存 + 模糊分级排序 │ ├── token.go # ParseAtToken / ParseSlashToken：光标处提取补全触发词 │ └── command.go # 内建命令候选 ├── history/history.go # 命令历史：内存 + .cogent/history 落盘 + 反向搜索 └── render/ # 纯渲染函数（无 I/O、易测） ├── stream.go # StreamMarkdown：有状态流式 Markdown-lite 渲染 ├── diff.go # ColorizeDiff：unified diff 语义着色 └── render.go # ANSI 着色原语（Bold/Cyan/Green…）
设计原则：把「与终端耦合的部分」（raw 模式、ioctl、光标控制）压到最薄，把「可以纯逻辑推演的部分」（按键→状态、候选排序、diff 着色）全部下沉到 internal 里做成无 I/O 的纯函数/状态机。耦合越薄，能被单测覆盖的比例越高。

03输入链路：一次按键的旅程

当你在提示符 you> 后敲下一个键，它会经历这样一条流水线，直到你按下回车、整行被提交：

图 2 · 输入链路：从 stdin 字节到提交整行

几个关键点：

  • 解码与逻辑分离：decodeKey 只负责把原始字节（含 ESC 转义序列如方向键）翻译成语义化的 keyEvent；真正的行为决策全在 editorCore。
  • 两个子模式：正常编辑 modeNormal 与反查 modeSearch，handleKey 先按模式分派，逻辑互不污染。
  • 惰性重算补全：任何改动行内容的按键（插入/退格/移动光标/历史切换）都会调 refresh()，它检查光标处是不是 / 或 @ 触发词，决定拉哪种候选。
  • 回车的二义性：下拉激活时回车是「采纳候选」，否则才是「提交整行」——一个键两种语义，靠 e.active 区分。

04核心：editorCore 状态机为什么能被单测

这是整个 TUI 最有含金量的设计。终端交互天然难测——一旦代码里混进了 os.Stdin.Read 和光标转义码，你就只能靠人肉敲键盘验证。Cogent 的做法是把「状态」和「I/O」彻底切开：

editorCore（纯逻辑）

持有 line / cursor / sugg / mode / histIdx 等全部状态。它不碰终端——输入是一个 keyEvent，输出是状态变化 + 一个 editorStatus（继续/提交/中断/EOF）。

lineEditor（薄 I/O 壳）

负责进 raw 模式、读字节、调 decodeKey、把 core 的状态画到终端（drawLine）。它几乎不含判断逻辑，只是搬运工。

可测性收益：因为 editorCore.handleKey 的签名是 keyEvent → editorStatus（纯函数式），单测可以直接喂一串按键事件序列，断言最终的 line/cursor/下拉状态，完全不需要真实终端。连「双击 Ctrl-C 800ms 内退出」这种时间相关逻辑，也通过可注入的 nowFunc 时钟做到确定性测试。这就是为什么简历里敢写「状态机与 I/O 解耦、可纯逻辑单测」。

状态机的一个精巧细节是历史浏览的暂存机制：当你按 ↑ 开始翻历史，第一次会把当前正在编辑的行 stash 起来；一路翻到底再按 ↓ 越过最近一条时，会把 stash 还回来——你的半截输入不会丢。这类"符合直觉"的体验，全是状态机里显式管理出来的。

05补全与历史：@ 文件 / 斜杠命令 / Ctrl-R

@ 文件补全：不为每次击键付出代价

输入 @src/ma 时，ParseAtToken 从光标处向前提取出触发词 src/ma，交给 completion.Provider.Filter。候选来源的关键设计在于缓存：

  • 优先用 git ls-files（含 --others --exclude-standard，即跟踪 + 未跟踪但不含 gitignore），失败或非 git 仓库时回退目录遍历（上限 maxWalkFiles=20000 条防拖慢巨型仓库）。
  • 5 秒 TTL 缓存（cacheTTL=5s）：期内复用内存清单，只在内存里做过滤——避免每敲一个字母就 spawn 一次 git 子进程。这是热路径上的性能意识。
  • 匹配按分级排序：前缀 < 路径段前缀 < 子串 < 子序列，同级再按路径长度、字典序稳定排序。所以 mai 会让 main.go 排在 internal/domain.go 前面。

Ctrl-R 反向增量搜索

按 Ctrl-R 进入 modeSearch，之后每敲一个字符就调 history.SearchBackward 从最近往前找首个包含查询串的历史；再按 Ctrl-R 跳到更早的命中。Esc/Ctrl-G 取消并恢复原行，Enter 采纳命中行。这套交互是 bash/zsh 老用户的肌肉记忆，直接照搬体验。

历史落盘：控制面里的私有数据

图 3 · 历史落盘：内存表 + append-only 文件

历史文件放在 .cogent/ 控制面目录，权限收紧到 0600/0700（你的历史命令可能含敏感路径，不该让同机其他用户读到）。写入是尽力而为的 append——落盘失败也不影响交互继续。

06输出链路：事件流如何变成会呼吸的终端

提交整行后，引擎开始执行，并把过程编码成一串 types.StreamEvent（文本增量、工具开始、工具结果、上下文压缩、完成、错误）。streamRenderer 逐个消费、渲染：

图 4 · 输出链路：事件流 → 分支渲染 → 终端

流式 Markdown：只前进、不回改光标

StreamMarkdown 是有状态的行渲染器：文本块到达时先缓冲，只有凑齐一整行（遇到 \n）才输出该行的 ANSI 渲染结果，不完整的行尾留到下次。它跨块维护代码围栏 ``` 状态，且只向前写、从不回退光标去重绘已输出内容。

为什么这么设计？流式场景下，如果一边收字符一边回改已渲染区域，会和终端自身的换行/滚动打架，造成闪烁与撕裂。「凑整行 + 只前进」用最简单的方式规避了这类竞争——代价是行内 Markdown 要等整行到齐才上色，但这在体感上完全可接受。这是"懂终端渲染的坑"才会做的取舍。

Diff 着色与结果折叠

工具返回若带 diff，走 ColorizeDiff 按语义上色：+ 新增绿、- 删除红、@@ hunk 头青、文件头加粗、上下文灰；超过 maxDiffLines=2000 行则降级为纯文本，避免超大 diff 拖慢渲染热路径。普通文本结果超过 maxResultLines=20 行会折叠并提示剩余行数，防刷屏。

07HITL 链路：危险操作的人工闸门

自主 Agent 敢用的前提，是危险操作能被人拦下。当工具执行触发权限中断，控制权会交回给 cliPrompter——它和 REPL 复用同一个 inputReader（避免多处并发读 stdin 打架）：

图 5 · HITL 权限中断：四态裁决
  • 会话级 always：选 A 后该工具名进入 allow 表，本会话内不再询问（mu 互斥保护，因为 Guard 可能并发调用 Ask）。
  • edit 兜底：选 e 可直接改工具的 JSON 入参；若输入非法 JSON，降级为拒绝并附说明，不会把坏参数塞给工具。
  • reject 带指引：拒绝时可写一句给模型的 guidance，回流后模型据此改道，而非死循环。
  • 无人值守：设 COGENT_YES 时切到 yesPrompter 一律自动批准——爆炸半径由 sandbox 命令拦截、worktree/diff 隔离兜底，所以自动批准仍然可控。
补充：menu.go 的 runMenu 是一个独立的 raw 单选菜单组件（↑↓ 循环、Enter 确认、Esc/Ctrl-G 取消、非 TTY 返回 unavailable 回退文本），用 DECSC/DECRC 保存恢复光标做原地重绘，可用于需要单选交互的场景；而权限 HITL 走的是上面 cliPrompter 的 a/A/e/r 文本决策路径。两者是同一套 raw 交互能力的不同出口。

08渲染细节：为什么"只前进、不回改光标"

行编辑器的 drawLine 用的是相对光标移动（上移 N 行回到输入行 → 跳过提示符 → 清到屏末 → 重绘输入 + 下拉），而不是 DECSC/DECRC（保存/恢复光标绝对位置）。原因很实际：

  • 当输出把终端顶到滚动时，绝对光标位置会失效，DECSC 存的坐标就错位了；相对移动始终相对"当前光标"，天然抗滚动。
  • drawnDropdownLines 记录上次画了几行下拉，这次据此精确上移回退清除——避免残影。

下拉窗口也有约束：候选最多拉 maxSuggest=15 个，但同屏只显示 maxVisibleSuggest=5 行，adjustWindow 保证选中项始终落在可见窗口内。这些是把交互做"顺滑"的琐碎但必要的功夫。

非 TTY 优雅降级

整套 raw 交互只在真终端启用。管道、脚本、测试场景下（非 TTY），inputReader 走后台 Scanner 逐行读取，streamRenderer 走 handlePlain 纯文本路径——行为与旧版一致，保证管道/CI 零回归。甚至"能读 termios 却拒绝进 raw"的受限终端，也会在首次读键时一次性懒回退到逐行模式，而不是让整个 REPL 秒退。

09设计取舍与面试要点

决策取舍 / 理由
手写行编辑器，不套 TUI 框架要的是「状态机可测 + 与引擎无缝集成」，也借此把 raw 模式/termios 吃透；框架反而带来不可控的黑盒。
editorCore 与 I/O 解耦让最难测的终端交互，能用「喂按键序列、断言状态」的纯逻辑单测覆盖——这是最值钱的一点。
流式渲染只前进、不回改光标规避流式重绘与终端滚动的竞争（撕裂/闪烁），代价是行内样式延迟到整行到齐。
补全 git ls-files + 5s TTL 缓存避免每次击键 spawn 子进程，热路径性能意识。
相对光标而非 DECSC/DECRC抗终端滚动，绝对坐标在滚动后会失效。
全链路非 TTY 降级管道/CI/测试零回归，raw 不可用时懒回退不秒退。
面试怎么讲（30 秒收敛）：「我没套 TUI 框架，手写了一套行编辑状态机与终端 I/O 解耦的交互层，所以连 raw 模式行编辑器都能写纯逻辑单测覆盖；流式渲染刻意只前进不回改光标来防重绘撕裂。」然后主动把话题拐回引擎：「这套可见（流式/Diff）、可控（HITL 中断）、可算（token/成本栏）的交互，本质是为了让前面那套自主执行引擎敢在真实仓库上跑——TUI 是让自主 Agent 从『能跑』跨到『敢用』的收尾。」
