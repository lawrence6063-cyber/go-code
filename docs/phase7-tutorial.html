Phase 7 · SubAgent Dispatch & Context Isolation

cogent Phase 7：把"探索子任务"派给隔离的子 Agent

有些子任务（"在大仓里定位某功能实现"）会产生大量中间消息——读十几个文件、grep 一堆匹配。如果都堆进主上下文，很快撑爆窗口、还淹没主线。Phase 7 给 cogent 装上 SubAgent 派发：主 Agent 通过内建 task 工具把子任务交给一个独立上下文的子 Agent，子 Agent 跑完只回流一段结果摘要，主链路保持干净。本阶段最值钱的看点是 依赖破环——如何让 agent→engine→tool 不形成环。

0新增第三方依赖
1新包 internal/agent
0engine/exec 改动行数
0go test -race 告警

核心心智：SubAgent 与主 Agent 共用同一套内核代码（同一个 engine.Engine），区别只在"喂给它的依赖"——独立的消息历史、受限的只读工具池、不落盘的会话。这是"复用而非另起炉灶"的内核抽象红利。

01为什么要 SubAgent？

设想任务："给 X 模块补限流中间件"。模型为了搞清现状，可能要先读 8 个文件、grep 3 次、列 2 个目录——这些探索产生十几条 tool_result，每条几百到几千 token。它们对最终决策有用，但对主对话是噪声：撑大上下文、提前触发压缩、还让后续推理被无关细节干扰。

没有 SubAgent

主上下文 = 系统提示 + 任务 + 15 条探索 tool_result + 推理 + 改动。探索噪声永久驻留，长任务很快超窗。

有 SubAgent

主上下文 = 系统提示 + 任务 + 1 条子 Agent 摘要 + 推理 + 改动。15 条探索消息留在子 Agent 的独立上下文里，跑完即弃。

蓝本对应：Claude Code 的 AgentTool / sidechain——派发独立上下文的子 Agent，子会话写独立 transcript，只把结果摘要回流父 Agent。cogent 用 Go 复刻这一思路，并复用既有内核。

02模块全景与依赖破环 ★

Phase 7 只新增一个包 internal/agent 与一个工具文件 tool/task.go，engine / exec / orchestrate 一行未改。但这里藏着本阶段最关键的工程问题——循环依赖。

问题：一个天然的依赖环

要派发子 Agent，agent 包必须调用 engine.New(...) 起一个子 Engine——所以 agent → engine。而 engine 经 exec.go 依赖 tool（执行工具）——engine → tool。现在 task 工具又要"触发派发"，最直觉的写法是让它 import agent——tool → agent。三条边一拼：

如果天真实现agent ──▶ engine ──▶ tool ──▶ agent   // ✗ 循环！Go 直接编译报错

这不仅是编译错误，更违反 DEV_SPEC §4.4 的硬约束「依赖只能向内」。

解法：把 Spawner 接口下沉到 tool 包

关键洞察——task 工具需要的不是 agent 的实现，而是"能派发"这个能力契约。于是我们把接口定义在 tool 包，让 task 只依赖抽象；agent 返回的具体类型隐式满足该接口（Go 的结构化接口，无需显式 implements，agent 甚至不用 import tool）。

internal/tool/task.go — 接口定义在被依赖的内层// Spawner 派发一个隔离上下文的子 Agent 执行子任务，返回结果摘要。
type Spawner interface {
    Spawn(ctx context.Context, prompt string) (string, error)
}
internal/agent/spawner.go — agent 仅 import engine，隐式满足 tool.Spawnerfunc New(deps engine.Deps) *SubAgent          // 返回具体类型
func (s *SubAgent) Spawn(ctx, prompt) (string, error)  // 方法集匹配接口

依赖方向被彻底捋直，无环、守不变量：

最终依赖方向cmd ──▶ agent ──▶ engine ──▶ tool ──▶ types
 │                                  ▲
 └──── agent.New(...) → tool.NewTask(spawner) ───┘   // cmd 在最外层做 wiring
设计模式归位：这正是依赖倒置原则（DIP）——高层(task)与低层(agent)都依赖抽象(Spawner)，且抽象由被依赖方（内层 tool）定义。同样的手法在标准库随处可见（io.Reader 由消费方定义）。面试时这是一个能讲清"为什么这么放"的好例子。

03上下文隔离的四道保证

"隔离"不是一句口号，而是四个具体、可验证的措施。任何一条漏掉，子 Agent 的探索噪声都可能漏回主链路或造成失控。

#保证怎么做防住什么
1独立消息历史每次 Spawn 都 engine.New(...) 一个全新子 Engine（各自持有 msgs）子任务间互不串扰、不携带父历史
2不落盘派发器强制 Session=nil、SessionID=""不污染主任务的会话 transcript（§8.5）
3受限工具面子池只给只读工具，且不含 task 自身无限递归派发 / 子 Agent 越权改文件
4更紧的护栏子任务 MaxSteps 默认 8（< 主循环 16）；摘要截断到 8KB子任务失控空转烧 token / 回流过大反噬主上下文
第 3 条尤其关键：如果子 Agent 的工具池里也有 task，模型可能不断"派发里再派发"形成爆栈。从工具面物理上抹掉 task，比"叮嘱模型别递归"可靠得多——这是 fail-closed 思维的又一次体现。

04task 工具实现

照 readfile.go 的成熟范式：匿名嵌入 tool.Defaults 拿到 fail-closed 默认，只覆写需要放开的方法。

internal/tool/task.go（节选）type taskTool struct {
    Defaults
    spawner Spawner
}

// 只读：子 Agent 仅装配只读工具池，故派发本身视为只读 → plan/ask 档位也能用
func (t *taskTool) IsReadOnly(json.RawMessage) bool { return true }

// 放行：派发只读子任务无需 HITL（子 Agent 内的高危操作另有自身闸门兜底）
func (t *taskTool) CheckPermission(context.Context, json.RawMessage) (permission.Decision, error) {
    return permission.Decision{Behavior: permission.BehaviorAllow}, nil
}

func (t *taskTool) Call(ctx context.Context, input json.RawMessage, _ ProgressSink) (types.ToolResult, error) {
    // ... 解析 prompt，空/非法直接规范化为 IsError ...
    summary, err := t.spawner.Spawn(ctx, in.Prompt)
    if err != nil {
        return types.ToolResult{Content: fmt.Sprintf("sub-agent failed: %v", err), IsError: true}, nil
    }
    return types.ToolResult{Content: summary}, nil
}
错误规范化惯例：注意 Call 的第二个返回值恒为 nil——派发失败被规范化为 ToolResult{IsError:true} 回流给模型，让它"看到失败、自我修正"，而不是中断整个主循环。这与 Phase 2 起所有工具的错误处理惯例一致。

关于并发安全：task 沿用 Defaults 的 IsConcurrencySafe=false（fail-closed，多个 task 调用串行）。理论上每次 Spawn 都是独立 Engine、共享的 llm.Client 与 tracer 也都并发安全，放开它即可实现"并行探索多个子任务"——这是一个可讲的权衡点，v1 选择保守默认。

05Spawner 与摘要回流

internal/agent/spawner.go（核心）func (s *SubAgent) Spawn(ctx context.Context, prompt string) (summary string, err error) {
    sub := s.deps
    sub.Session = nil   // 隔离保证②：子 Agent 不落盘
    sub.SessionID = ""
    eng, err := engine.New(sub)   // 隔离保证①：全新 Engine = 全新 msgs
    if err != nil { return "", fmt.Errorf("build sub-agent: %w", err) }

    ctx, end := s.tracer.Start(ctx, "agent.spawn")  // 隔离保证④ + trace 串联
    defer func() { end(err) }()

    events, err := eng.Run(ctx, prompt)
    if err != nil { return "", fmt.Errorf("run sub-agent: %w", err) }
    return s.collectSummary(events), nil
}

摘要怎么来？子 Engine 的系统提示本就要求"任务完成时给出简洁总结"，所以子 Agent 的最终助手文本天然就是摘要。collectSummary 把事件流里的 EventText 累积起来，按 UTF-8 边界截断到 8KB：

collectSummary — 完整 range 至 channel 关闭（防泄漏）func (s *SubAgent) collectSummary(events <-chan types.StreamEvent) string {
    var sb strings.Builder
    for ev := range events {          // 必须排空：ctx 取消时 engine 也会 close(out)
        switch ev.Type {
        case types.EventText:  sb.WriteString(ev.Text)
        case types.EventError: /* 附错误说明，仍回流给主 Agent 参考 */
        }
    }
    return truncateSummary(sb.String(), s.maxSummaryBytes)
}
易踩的坑——goroutine 泄漏：for range events 会一直读到 channel 关闭。这正是关键：engine.Run 的内部 goroutine 在 ctx 取消时也会走 defer close(out)，所以排空循环一定会结束。如果偷懒"读到一半 ctx 取消就 return"，子 Engine 的 goroutine 可能阻塞在向已无人消费的 out 发送上——goleak 会立刻抓出来。

06trace 跨边界串联：context 的双重职责

子 Agent 是"任务里的任务"，可观测上理应表现为"trace 里的子树"。cogent 不需要任何额外管线就做到了——因为 context.Context 同时承载取消传播与 trace 传播。

span 树（一次带派发的任务）cogent.session                       # 主任务根 span
└── react.step (step=2)
    ├── llm.stream
    └── tool.batch
        └── tool.call (name=task)      # 主 Agent 调用 task 工具
            └── agent.spawn             # ← Spawn 起的 span，挂在 tool.call 下
                └── react.step (step=0)   # 子 Agent 自己的 ReAct 循环
                    ├── llm.stream         #   全部经 ctx 自动成为子节点
                    └── tool.call (read_file)

Spawn 收到的 ctx 已携带父 tool.call 的 span；它 Start("agent.spawn") 派生新 ctx 传给子 engine.Run，子 Engine 内部的 react.step/llm.stream 再从这个 ctx 派生——无需手动传 trace_id，父子关系自动成立（§8.2 跨边界传播）。同一个 ctx 还意味着：主任务 Ctrl-C 时，子 Agent 的 LLM 调用与 goroutine 一并收手。

07cmd 装配与防递归

所有"组装"都收敛在最外层 cmd，内层包互不感知彼此的实现。

cmd/cogent/commands.go（节选）// 子池：只读工具，刻意不含 task 自身 → 杜绝递归派发
func buildSpawner(llmc llm.Client, prov observe.Provider, workRoot string) tool.Spawner {
    subTools := tool.NewPool(
        tool.NewReadFile(workRoot),
        tool.NewListDir(workRoot),
        tool.NewGrep(workRoot),
    )
    return agent.New(engine.Deps{LLM: llmc, Tools: subTools, Observe: prov, WorkRoot: workRoot})
}

// 主池：task 作只读工具直接入池；复用同一个 llm.Client 让 trace 串起来
spawner := buildSpawner(llmc, prov, workRoot)
tools := []tool.Tool{ ..., tool.NewTask(spawner), ... }
复用而非另起炉灶：buildSpawner 用的还是同一套内建只读工具构造函数、同一个 llm.Client、同一个 observe.Provider。子 Agent 不是"另一个程序"，只是"喂了不同依赖的同一个内核"。

09设计要点回顾

接口下沉破环

Spawner 接口放在 tool 包，agent 隐式满足、无需反向 import —— DIP 的教科书落地。

内核复用

子 Agent = 喂不同依赖的同一个 engine，engine/exec 零改动。

四道隔离

独立 msgs / 不落盘 / 只读子池 / 更紧护栏，条条可验证。

fail-closed 防递归

从工具面抹掉 task 自身，物理杜绝无限派发。

trace 自动串联

ctx 双职责，agent.spawn 子树自动挂接，零额外管线。

摘要回流防膨胀

只回流 ≤8KB 摘要，探索噪声留在子上下文跑完即弃。

STAR（面试话术）：

S：探索类子任务会产生大量中间消息，堆进主上下文会撑爆窗口、淹没主线。

T：隔离探索、只回流结论，且不为此另写一套循环。

A：复用同一 engine 内核派发子 Agent（独立 msgs / 受限只读池 / 不落盘 / 更紧 maxSteps）；为破 agent→engine→tool→agent 的环，把 Spawner 接口下沉到 tool 包，agent 隐式满足；子 Agent 的 span 经 ctx 自动挂为 agent.spawn 子树；排空事件流累积摘要并 8KB 截断，goleak 验无泄漏。

R：复杂任务可拆给子 Agent，主上下文保持干净；新增能力零改内核；依赖方向无环、守 §4.4 不变量。

诚实边界 / Future Work：v1 的 task 串行执行（fail-closed），尚未实现"并行派发多个子任务"；子 Agent 的工具池固定为只读三件套，未做按子任务动态裁剪；摘要直接取最终文本，未来可让子 Agent 产出结构化结论（决策 + 理由 + 待办）以更好地喂回主循环。
