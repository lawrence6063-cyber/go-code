1. Git 是什么 & 为什么用 Git

Git 是一个分布式版本控制系统（DVCS），由 Linus Torvalds 于 2005 年创建，用于管理 Linux 内核源代码。

核心优势

  • 分布式：每个开发者本地都有完整仓库历史，无需联网也能工作
  • 快速：几乎所有操作都在本地完成，毫秒级响应
  • 分支廉价：创建/切换分支几乎零成本，鼓励频繁分支
  • 数据完整性：使用 SHA-1 哈希校验，任何篡改都能被检测
  • 暂存区：精确控制每次提交的内容粒度
Git 与 SVN 最大区别：SVN 是集中式（中央服务器），Git 是分布式（每人都是完整仓库）。

2. 安装与初始配置

安装

# macOS (Homebrew)
brew install git

# Ubuntu / Debian
sudo apt-get install git

# Windows — 下载安装包
# https://git-scm.com/download/win

首次配置（全局）

# 设置用户名和邮箱（每次提交都会记录）
git config --global user.name "Your Name"
git config --global user.email "your@email.com"

# 设置默认编辑器
git config --global core.editor "vim"

# 设置默认分支名
git config --global init.defaultBranch main

# 查看当前配置
git config --list
配置存放位置：~/.gitconfig（全局）、.git/config（项目级）。项目级优先级更高。

3. 核心概念：三区模型

Git 管理文件有三个关键区域：

工作区
Working Directory
你编辑文件的地方
→
git add
暂存区
Staging Area / Index
下次提交的快照
→
git commit
本地仓库
Repository (.git)
永久历史记录

理解类比

  • 工作区 = 你的桌面，当前正在修改的文件
  • 暂存区 = 购物车，挑选好要提交的变更
  • 本地仓库 = 仓库货架，结账后商品入库，形成一次记录

文件状态生命周期

Untracked ──git add──→ Staged ──git commit──→ Committed
     ↑                    ↑                        │
     │                    └──── git add ←──────────┘ (修改后重新暂存)
     └─── 新建文件

4. 基础操作

创建仓库

# 从零初始化
mkdir my-project && cd my-project
git init

# 克隆远程仓库
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # SSH方式

日常工作流

# 1. 查看状态（最常用的命令）
git status

# 2. 添加文件到暂存区
git add file.txt          # 单个文件
git add .                 # 所有变更
git add -p                # 交互式选择部分变更（推荐！）

# 3. 提交
git commit -m "feat: add user login"
git commit                # 打开编辑器写详细信息
git commit -am "fix: typo"  # add + commit（仅限已跟踪文件）

# 4. 查看日志
git log                   # 完整日志
git log --oneline         # 单行简洁模式
git log --graph --oneline --all  # 图形化分支历史
git log -5                # 最近5条
git log --author="Name"   # 按作者筛选
git log -- path/to/file   # 某文件的变更历史
养成好习惯：每次 commit 前先 git status 和 git diff --staged 确认要提交的内容。

Commit Message 规范（推荐 Conventional Commits）

类型(范围): 简短描述

feat:     新功能
fix:      修复 bug
docs:     文档变更
style:    代码格式（不影响逻辑）
refactor: 重构（既非新功能也非修复）
test:     测试相关
chore:    构建/工具/依赖变更

示例：
feat(auth): add OAuth2 login support
fix(api): handle nil pointer in user handler
docs: update README installation steps

5. 撤销与回退

场景命令效果
撤销工作区修改（未 add）git restore file.txt文件恢复到上次提交状态
撤销暂存（已 add，未 commit）git restore --staged file.txt从暂存区移除，工作区保留
修改最后一次提交git commit --amend替换上次提交（可改消息/加文件）
回退到某次提交（保留工作区）git reset --soft HEAD~1撤销 commit，变更留在暂存区
回退到某次提交（清空暂存区）git reset --mixed HEAD~1撤销 commit + add，变更留在工作区
回退到某次提交（全部丢弃）git reset --hard HEAD~1⚠️ 危险！丢弃所有未提交变更
创建一个"反向提交"git revert <commit>安全！不改历史，新增一个撤销提交
git reset --hard 会永久丢失工作区和暂存区的未提交内容！操作前请确认。 如果误操作，可尝试 git reflog 找回。

reset 三种模式图解

              HEAD~1          HEAD (当前)
                │               │
  ──────────────●───────────────●
                          
  --soft:   commit 撤销 → 变更留在 暂存区
  --mixed:  commit 撤销 → 变更留在 工作区 (默认)
  --hard:   commit 撤销 → 变更 全部丢弃 ⚠️

6. 分支管理

为什么要用分支？

分支让你可以在不影响主线代码的情况下开发新功能、修复 Bug、做实验。Git 的分支只是一个指向某次 commit 的指针，创建成本极低。

基础操作

# 查看分支
git branch            # 本地分支
git branch -a         # 所有分支（含远程）

# 创建 & 切换
git branch feature/login       # 创建
git switch feature/login       # 切换（推荐）
git switch -c feature/login    # 创建并切换

# 删除
git branch -d feature/login    # 已合并的分支
git branch -D feature/login    # 强制删除（未合并）

# 重命名
git branch -m old-name new-name

合并（Merge）

# 将 feature 合入 main
git switch main
git merge feature/login

# 快进合并 (Fast-Forward)：如果 main 没有新提交
# main ───●───●───● (feature)  →  main 直接指向 feature 最新提交

# 三方合并 (3-way merge)：如果两边都有新提交，产生合并提交
#       ●───●  (main)
#      /       \
# ●───●         ●  (merge commit)
#      \       /
#       ●───●  (feature)

变基（Rebase）

# 将 feature 的提交"移植"到 main 最新位置
git switch feature/login
git rebase main

# 变基前：
# main:    A─B─C
# feature:   B─D─E

# 变基后（D、E 被重新应用到 C 之后）：
# main:    A─B─C
# feature:       C─D'─E'

Merge vs Rebase 对比

MergeRebase
历史保留真实分支历史，有合并提交线性历史，更整洁
安全性安全，不修改已有提交重写提交历史（SHA 变了）
适用公共分支合并个人分支整理
冲突一次性解决可能每个提交都要解决
黄金法则：永远不要对已经推送到公共仓库的提交做 rebase！

解决冲突

# 冲突发生时，文件中会出现标记：
<<<<<<< HEAD
你的修改
=======
他人的修改
>>>>>>> feature/login

# 解决步骤：
# 1. 手动编辑文件，删除标记，保留正确内容
# 2. git add 已解决的文件
# 3. git commit（merge）或 git rebase --continue（rebase）

# 放弃操作
git merge --abort
git rebase --abort

7. 远程协作

远程仓库管理

# 查看远程仓库
git remote -v

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 修改远程 URL
git remote set-url origin git@github.com:user/repo.git

# 删除远程
git remote remove origin

拉取与推送

# fetch：下载远程变更，不合并
git fetch origin
git fetch --all          # 所有远程

# pull = fetch + merge
git pull origin main
git pull --rebase        # fetch + rebase（推荐，保持线性历史）

# push：推送本地提交到远程
git push origin main
git push -u origin main  # 设置上游，之后直接 git push
git push origin feature/login

# 删除远程分支
git push origin --delete feature/login
本地仓库
← fetch
远程仓库
(GitHub/GitLab)
→ push

协作工作流示例

# 1. 从主分支创建功能分支
git switch main
git pull
git switch -c feature/new-api

# 2. 开发 & 提交
git add .
git commit -m "feat(api): implement user endpoint"

# 3. 推送到远程
git push -u origin feature/new-api

# 4. 创建 Pull Request / Merge Request（在平台上操作）

# 5. 代码审查通过后合并

# 6. 清理
git switch main
git pull
git branch -d feature/new-api

8. 暂存工作：stash

当你正在工作但需要临时切换分支时，git stash 可以"藏起"当前的未提交修改。

# 保存当前工作
git stash                    # 默认保存已跟踪文件的修改
git stash -u                 # 含未跟踪文件
git stash push -m "WIP: login page"  # 带描述信息

# 查看存储列表
git stash list
# stash@{0}: On feature: WIP: login page
# stash@{1}: On main: debugging api

# 恢复工作
git stash pop               # 恢复最近一次，并从列表删除
git stash apply             # 恢复但不删除（可反复使用）
git stash apply stash@{1}   # 恢复指定条目

# 删除
git stash drop stash@{0}    # 删除指定
git stash clear              # 清空所有
stash 是一个栈结构（LIFO），最后存入的最先取出。

9. 标签管理：tag

# 创建轻量标签
git tag v1.0.0

# 创建附注标签（推荐，含作者/日期/消息）
git tag -a v1.0.0 -m "Release version 1.0.0"

# 给历史提交打标签
git tag -a v0.9.0 abc1234

# 查看标签
git tag
git show v1.0.0

# 推送标签到远程
git push origin v1.0.0      # 单个
git push origin --tags       # 所有

# 删除标签
git tag -d v1.0.0                    # 本地
git push origin --delete v1.0.0      # 远程

10. 查看与比较

# diff：比较差异
git diff                    # 工作区 vs 暂存区
git diff --staged           # 暂存区 vs 最后一次提交
git diff HEAD               # 工作区 vs 最后一次提交
git diff main..feature      # 两个分支差异
git diff abc123..def456     # 两个提交差异
git diff --stat             # 仅显示统计信息

# show：查看某次提交详情
git show abc1234
git show HEAD~2             # 倒数第3次提交
git show v1.0.0:path/file   # 某个标签时的文件内容

# blame：查看每一行的最后修改者
git blame file.txt
git blame -L 10,20 file.txt  # 仅看第10-20行

11. 高级技巧

cherry-pick：摘取特定提交

# 将某个提交应用到当前分支
git cherry-pick abc1234

# 摘取多个
git cherry-pick abc1234 def5678

# 只暂存不提交
git cherry-pick --no-commit abc1234

rebase -i：交互式变基（整理提交历史）

# 整理最近3次提交
git rebase -i HEAD~3

# 编辑器中可选操作：
# pick   = 保留该提交
# reword = 保留但修改提交信息
# edit   = 保留但暂停让你修改
# squash = 合并到上一个提交（保留信息）
# fixup  = 合并到上一个提交（丢弃信息）
# drop   = 删除该提交
交互式 rebase 是整理提交历史的利器。在 push 前用它把零碎提交合并成有意义的单元。

reflog：Git 的"后悔药"

# 查看 HEAD 的所有历史移动记录
git reflog

# 输出示例：
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: feat: add feature
# 789abcd HEAD@{2}: checkout: moving from main to feature

# 恢复误删的提交
git reset --hard def5678

reflog 记录了你在本地仓库的每一次 HEAD 变更，即使 commit 被 reset 掉了，90 天内都能通过 reflog 找回。它是你的安全网！

bisect：二分查找 Bug

# 启动二分查找
git bisect start
git bisect bad              # 当前版本有 bug
git bisect good v1.0.0      # 这个版本没有 bug

# Git 会自动 checkout 中间提交，你测试后标记：
git bisect good             # 这个版本没问题
git bisect bad              # 这个版本有问题

# 找到引入 bug 的提交后：
git bisect reset            # 结束，回到原来的分支

12. Git 工作流

Git Flow

  • main — 生产环境代码
  • develop — 开发集成分支
  • feature/* — 功能开发
  • release/* — 发布准备
  • hotfix/* — 紧急修复

适合：版本发布周期明确的项目

GitHub Flow

  • main — 始终可部署
  • feature/* — 从 main 创建
  • 通过 PR 合并回 main
  • 合并后立即部署

适合：持续部署的 Web 项目

Trunk-Based Development

  • 所有人直接在 main（trunk）上工作
  • 使用短生命周期的功能分支（1-2天）
  • 依赖 Feature Flag 控制未完成功能
  • 适合高频发布、成熟 CI/CD 的团队

14. 常见问题与避坑指南

问题解决方案
提交了不该提交的大文件git filter-branch 或 git-filter-repo 重写历史
commit message 写错了git commit --amend（仅限未 push）
merge 搞砸了想撤销git merge --abort（冲突中）或 git reset --hard ORIG_HEAD
不小心在 main 上开发了git stash → 切到新分支 → git stash pop
pull 有冲突不想处理git pull --rebase 或 git merge --abort
想撤销已 push 的提交git revert <commit>（安全方式，不重写历史）
误删分支想恢复git reflog 找到 SHA → git branch recovered <sha>
detached HEAD 状态git switch main 回到分支，或 git switch -c new-branch 保存当前工作
三大高危命令： git reset --hard、git push --force、git clean -fd
操作前务必确认！--force 推送会覆盖他人工作，团队协作中应使用 --force-with-lease。
