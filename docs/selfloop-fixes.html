SELF-LOOP FIXES · 修复说明

自循环实验问题 · 一次性修复说明

针对 docs/selfloop-report.md 暴露的循环机制缺陷，一次性（不分批）完成修复：让客观 verify 成为不可被 reviewer 短路的最终硬闸门、裁决解析鲁棒化、成本护栏闭环、墙钟硬化、reviewer 选型与环境感知工程化，并直接闭合 finish_reason gap。

总览 修复点对照表 ②verify 硬闸门 ③裁决解析 ⑤成本护栏 ④⑥选型与硬化 finish_reason 验证方式

1总览

原实验报告的结论是：循环机制端到端跑通且可观测，但目标未达成——根因是「主观闸门（reviewer）在客观判据（verify）之前持续拒绝」，叠加裁决格式不合规、成本护栏空接线等问题。本次修复按报告逐条对症下药。

核心原则：客观判据必须处于循环终止路径上，且不被任何前置主观闸门短路。「达目标才停」从此回归为「verify 通过才停」，而非「reviewer 满意才停」。

范围：发现①（已就地修复，仅标注）+ 发现②③④⑤⑥ + 实验目标 finish_reason gap。全部改动配套单元测试，go test ./... 全绿、verify.sh 退出码 0。

3发现②：客观 verify 升级为最终硬闸门

这是本次最关键的改造。原流水线是「maker 改 → reviewer 审 → 通过才 Merge → 再跑 verify」的串行双闸门——reviewer 一旦拒绝，worktree 立即被物理丢弃，客观 verify 既跑不到、也看不到改动。

修复前 reviewer 短路客观判据

// orchestrator.go · runPipeline（旧）
if !pr.Approved {
    // reviewer 拒 → 直接返回，跳过 verify
    return Report{Summary: "reviewer rejected"}
}
return o.verify(ctx, goal, out) // 仅通过才验收
  • goal.verify span 数 = 0（从未执行）
  • 退化为「reviewer 满意才停」
  • reviewer 失灵 = 整个目标无法收敛

修复后 verify 并联且不可短路

// orchestrator.go · runPipeline（新）
pr, _ := o.pipeline.Iterate(ctx, task,
        o.makeVerifyAt(goal)) // 注入客观判定
report := o.pipelineReport(pr)
// report.Passed 来自 verify，reviewer 仅作建议反馈
return report
  • 每轮都开 goal.verify span（可观测自证）
  • verify 通过即 达标 + 落盘，reviewer 不能否决
  • verify 失败：合并 reviewer 建议续跑

关键机制

  • 新增 loop.VerifyAtFunc func(ctx, workRoot) verify.Report，由 orchestrator 用 goal.Verifier 适配，扩展 Pipeline.Iterate(ctx, task, verifyAt)。
  • verify.ScriptVerifier 新增 NewSandbox 工厂：按传入 workRoot 构造沙箱，使脚本跑在 maker 改动所在目录（worktree 根），看到真实改动。
  • worktree 流水线：在 ws.Root 执行 verify，通过才 Merge、否则 Discard；diff 回滚版同理（通过保留、失败 git checkout 回滚）。
  • 向后兼容：未配置 --verify 时 verifyAt==nil，回退到 reviewer 裁决作为落盘闸门。

4发现③：裁决解析鲁棒化（fail-closed 不变）

思维链型模型常先输出大段推理，首行不是裁决。解析改为全文扫描结构化裁决标记并取最终裁决，并要求 reviewer 在最后一行输出 VERDICT: APPROVED|REJECTED。仍保持 fail-closed：无明确 APPROVED 标记一律判拒（防自评虚高）。

修复前

func parseVerdict(out) {
  first := firstNonEmptyLine(out)
  if HasPrefix(upper(first), "APPROVED") {
    return Approved
  }
  return Rejected // 思维链首行恒非 APPROVED → 恒拒
}

修复后

func verdictApproved(out) bool {
  // 优先结构化 VERDICT: 标记（取最后一个）
  if v, ok := lastStructuredVerdict(lines); ok {
    return v
  }
  // 回退独立成行 APPROVED/REJECTED
  v, ok := lastBareVerdict(lines)
  return ok && v
}

新增单测覆盖：先推理后 VERDICT: APPROVED、markdown 强调 **VERDICT: APPROVED**、文中提及 approved 但结尾 REJECTED（结构化优先）等场景。

5发现⑤：成本护栏闭环

由于 observe.Meter 只写不可读，新增 cmd/cogent/cost.go：一个装饰 Provider/Meter 的成本计量器，拦截 cogent.tokens 计数，按 token.kind(input/output) 与 llm.model 属性按模型单价累计美元，并实现 loop.CostMeter。

// engine.recordUsage：按 input/output 分计并附属性（支持差异定价）
e.meter.Count("cogent.tokens", u.PromptTokens,
    Attr{"token.kind", "input"}, Attr{"llm.model", e.model})
e.meter.Count("cogent.tokens", u.CompletionTokens,
    Attr{"token.kind", "output"}, Attr{"llm.model", e.model})

// buildOrchestrator：装饰 provider + 注入 Cost，--max-cost 真正生效
costProv := newCostProvider(prov)
loop.New(loop.Deps{ ..., Meter: costProv.Meter(), Cost: costProv.meter })
  • 缺省单价：deepseek-chat 0.27/1.10，deepseek-reasoner 0.55/2.19（美元/百万 token，输入/输出）。
  • 可经 COGENT_PRICE_<MODEL>_INPUT/OUTPUT 覆盖；未知模型用保守回退价（宁可高估，利于早停）。
  • 线程安全（引擎多 goroutine 写、loop 读），cogent.loop.cost_usd 指标随之有值。

6发现④⑥：选型、环境感知与墙钟硬化

reviewer 选型解耦（④）

reviewerModel 缺省返回指令遵循更稳的 deepseek-chat（不再沿用 maker 模型），仍可由 COGENT_REVIEWER_MODEL 覆盖——maker 可用 reasoner 强推理，reviewer 用 chat 稳输出裁决格式。

消除 cwd 幻觉（④）

系统提示注入真实仓库根，明确「shell 与工具已位于仓库根、用相对路径、勿 cd 到 /workspace 等虚构路径」：

// engine/prompt.go
const workspaceHintTemplate = "\nYour shell and tools already run from " +
    "the repository root: %s\nAlways use paths relative to this root; " +
    "do NOT `cd` into invented absolute paths such as /workspace."

墙钟硬化（⑥）

// engine.effectiveLLMTimeout：取单次超时与剩余墙钟的较小值
if dl, ok := ctx.Deadline(); ok {
    if remaining := time.Until(dl); remaining > 0 && remaining < d {
        return remaining // 墙钟将到顶 → 收紧本次调用超时
    }
}

7实验目标：finish_reason 端到端闭合

原实验要 dogfood 闭合的 gap，本次直接落地。
