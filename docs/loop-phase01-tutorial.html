LOOP_SPEC · L0 + L1（P0 核心）· 已落地

从「撞护栏才停」到「达目标才停」

cogent v1 是一台优秀的单次任务执行内核：你点一下，它跑一轮 ReAct（探索→改→执行→验证）就结束。本次落地的 Loop Phase 0+1 在它之上加了一层「外层循环」——你只给目标 + 验收条件 + 预算，它自己反复干、独立判定、不达标带着反馈再来一轮，直到真正达成目标或撞预算才停。

0一句话顿悟：把「判官」请进循环

v1 怎么知道「干完了」？答案有点尴尬——它不知道。它停下来只有几种原因：模型不再要求调用工具了、轮数到顶了、被 Ctrl-C 了。这些都在回答「还能不能跑」，而不是「任务到底完成没有」。模型完全可能自信地说「修好了！」，结果代码根本编译不过。

关键转变：完成是「证明」而非「声明」——由独立于执行体的判官裁决，写代码的 Agent 不给自己打分。
记住三个新词：目标循环（/goal）=外层反复跑；独立判定器（Verifier）=客观判官；预算护栏（Budget）=防失控烧钱的刹车。这一节就讲清这三块怎么拼起来。

1L0 · 编排层地基：一层「套」在 engine 外面的循环

最关键的工程决策是 叠加，而不是重写。v1 的 engine（ReAct 单次执行体）一行都不改，我们在它外面新增一个 internal/loop 包，像套娃一样把 engine 包进去。

依赖严格单向：loop → engine，engine 永远不知道 loop 的存在。这样 v1 内核可独立编译/测试，v2 是一层可裁剪的外挂。

L0 干了两件事

① 定义外层事件 LoopEvent

外层循环向终端单向上抛四类事件：IterationStart（新一轮）、Inner（原样透传内层 engine 的每条输出）、Verify（判定结果）、Finished（最终结局）。终端只是事件的消费者。

② 跑通「单轮闭环」

先不接判官也能转：调一次 engine.Run → 把内层事件透传出去 → 发 Finished。地基稳了，L1 再往里塞「判定 + 预算 + 反馈」。同时埋好 loop.run / loop.iteration 链路 span。

2L1 · 目标循环主流程：双出口的转圈

这是整个 Phase 的心脏。外层循环每一轮做三件事：让 engine 干一轮 → 请判官独立验收 → 据结果决定「停」还是「带反馈再来」。它有两个出口：达标停（成功），或撞预算停（失败兜底）。

两个出口缺一不可：达标停是「成功」，撞预算停是「兜底」——永远不会无限转圈。

对应的真实代码（精简）

internal/loop/orchestrator.gofunc (o *orchestrator) loop(ctx context.Context, goal Goal, budget Budget, ...) LoopResult {
	deadline, cancel := budgetContext(ctx, budget) // 墙钟上限 → ctx 超时
	defer cancel()

	task := goal.Intent // 第一轮：原始目标
	for iter := 0; iter < budget.MaxIterations; iter++ {  // ← 轮数护栏
		if deadline.Err() != nil { return o.result(stopOutcome(ctx), ...) }
		report, stopped := o.iterate(deadline, iter, task, goal, out) // ① 跑一轮 ② 判定
		if stopped { return ... }
		if report.Passed { return o.result(OutcomeAchieved, iter+1, ...) } // ✓ 达目标才停
		if o.overCost(budget) { return o.result(OutcomeBudgetSpent, ...) } // ← 成本护栏
		task = feedbackPrompt(goal.Intent, report) // ③ 未达标 → 把原因拼成下一轮任务
	}
	return o.result(OutcomeBudgetSpent, budget.MaxIterations, ...) // 轮数耗尽兜底
}
读懂这段就读懂了 L1。循环体短到 20 行，因为重活全委托给了 v1 的 engine.Run 和独立判官——这正是「叠加而非重写」的红利。

3独立判定器：写代码的不能给自己打分

「独立」二字是灵魂。如果让写代码的同一个 Agent 自评，等于「让学生自己改卷」——它有强烈的确认偏置，会倾向于说「过了」。所以判官必须独立于执行体。

L1 落地的核心判官是 ScriptVerifier：它把一个验收脚本（如 verify.sh）丢进沙箱执行，退出码 0 = 达标，其余一律未达标。脚本的退出码是确定性的、不经模型解读，天然防注入。

判官与执行体完全解耦：它只输出一个布尔 + 反馈文本，不持有任何写权限，无法被「忽略测试直接报通过」之类的注入劫持。
internal/verify/verify.go// Verifier 判定目标是否达成；实现应确定性、可复现、独立于被测 Agent。
type Verifier interface {
	Verify(ctx context.Context, workRoot, goalIntent string) (Report, error)
}

func (v ScriptVerifier) Verify(ctx context.Context, _, _ string) (Report, error) {
	res, err := v.Sandbox.Exec(ctx, "bash "+v.Script)
	if err != nil { return Report{Summary: ...}, ... } // ← fail-closed：跑不动=未通过
	if res.ExitCode == 0 { return Report{Passed: true, ...}, nil } // ← 退出码 0 才达标
	return Report{Passed: false, Summary: ...}, nil
}
fail-closed 贯穿全程：判官为 nil → 视为永不达标；脚本异常/超时 → 视为未通过；预算零值 → 用保守默认。任何疏漏都退化为「早停」，绝不退化为「空转烧钱」。
一个实战细节：验收脚本要跑 go test，需要宿主的 go 工具链。因此判官用的沙箱设 Enabled=false（继承宿主 PATH），但仍保留危险命令拦截 + 工作目录约束 + 超时——验收脚本是开发者写的可信控制面，与不可信的模型输出隔离对待。

4三重预算护栏：自治的前提是有刹车

一旦 Agent 能「自己反复跑」，最大的新风险就是 失控烧钱（AutoGPT 的经典翻车）。所以 L1 的铁律是：无预算，不启动。Budget 有三道独立刹车，任一触顶即安全停。

🔁

最大轮数 MaxIterations

外层最多重试几轮。默认 8。最硬的兜底——哪怕其它都没设，也绝不会无限循环。

💲

最大成本 MaxCostUSD

累计 LLM 花费上限。默认 $5。经 CostMeter 接口读取累计成本驱动。

⏱️

最大墙钟 MaxWallClock

端到端时间上限。默认 15min。直接转成 ctx 超时，到点取消贯穿内外层。

内外两层护栏叠加成纵深

层级护栏触发即
内层（v1 沿用）maxSteps / 单工具超时 / compact 熔断单轮任务停
外层（v2 新增）MaxIterations / MaxCostUSD / MaxWallClock整条循环停（BudgetSpent）
全局ctx 取消（Ctrl-C / SIGTERM）内外层同时收手（Canceled）
区分两个「轮」：内层 maxSteps 管「一次任务里 ReAct 转几圈」，外层 MaxIterations 管「目标没达成时重试几次」。两层正交，缺一仍可能失控。

5带反馈续跑：复用同一引擎的记忆，零侵入

未达标怎么办？把判官的「为什么没过」拼成一句新任务，再喂给 同一个 engine 实例。这里藏着一个优雅的复用：v1 的 engine 本身就是多轮对话的——同一实例连续 Run 会自动累积消息历史。

不碰内核内部状态，只是「多问一句」。复用 v1 已验证的多轮累积，避免重建上下文的复杂度。
internal/loop/orchestrator.gofunc feedbackPrompt(intent string, r verify.Report) string {
	// 把「原始目标 + 未达标原因 + 证据」拼成下一轮任务
	return "The goal is not yet achieved..." +
		"Original goal: " + intent +
		"Verification feedback (not passed):\n" + r.Summary + ...
}

6怎么跑起来：cogent goal

所有这些能力通过一个新子命令暴露给开发者：

terminal# 给目标 + 验收脚本 + 预算，回车后它自己反复干到达标
cogent goal "修复 parser 的 bug，让所有单元测试通过" \
  --verify=eval/tasks/fix_off_by_one/verify.sh \  # 判官：退出码 0 = 达标
  --max-iterations=8 \                          # 护栏：最多 8 轮
  --max-wallclock=15m                          # 护栏：最多 15 分钟

终端会实时打印外层循环的进展：每一轮的标题、内层 engine 的流式输出、判官的验收结论，最后是归因清晰的终局：

output（示意）=== iteration 1 ===
cogent> [读 sum.go ... 改 sum.go ...]
[verify] NOT PASSED — verification not passed (exit code 1)

=== iteration 2 ===
cogent> [根据上轮测试失败重新定位 ... 修正 off-by-one ...]
[verify] PASSED — verification passed (exit code 0)

=== loop finished: achieved after 2 iteration(s) in 38.2s ===

7收尾：这一步把 cogent 推到了哪

① 达目标才停

从「撞护栏」升级为「独立判官说达标才停」，完成是证明而非声明。

② 零破坏叠加

engine/tool/sandbox 一行没改，全靠 loop+verify 两个新叶子包外挂实现。

③ 预算先行

三重护栏 fail-closed，自治的同时绝不失控烧钱。

用 Boris Cherny 的「三个高度」说：cogent 从「高度 2（你指挥它跑一次）」迈过了「高度 3（你设好目标和预算，它自己跑到达标）」的门槛。你从「司机」变成了「导航员」。
