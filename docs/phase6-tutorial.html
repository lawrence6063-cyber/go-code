cogent Phase 6：让 Agent 拥有「外接工具」的扩展面

前五阶段，cogent 的工具池是封闭的——只认进程内编译进来的内建工具。Phase 6 给它装上 MCP（Model Context Protocol）客户端：通过 os/exec 拉起任意 MCP server 子进程，以换行分隔 JSON-RPC 完成握手与 tools/call，把远端工具以 mcp__<server>__<tool> 命名，fail-closed 融入同一个工具池——内核零改动、安全姿态不降级。

这份教学的特别看点是"零依赖接入外部生态"——亲手实现 MCP stdio 协议，把官方 modelcontextprotocol/go-sdk 挡在主二进制之外；再把"协议契约"沉淀成一套内联的一致性套件 + fake server，验证手写实现的正确性。

01为什么要接 MCP？

一个真正能"自主编码"的 Agent，工具能力必须能横向扩展。把工具硬编译进进程意味着：每新增能力都要改源码、重新发版、重新审计。MCP 把"工具供给"标准化成了一个跨语言协议——server 可以用 Python/Node/Go 任意语言写，cogent 只要会说 MCP，就能把整个开源生态（filesystem/git/fetch/各类数据库……）瞬间纳入工具池。

核心约束是：外部工具默认不可信。cogent 已在 Phase 2 把权限三态裁决 + HITL 装在 Guard 装饰器里——Phase 6 不发明新机制，只是让外部工具同样穿过这道闸门即可。

关键技术决策：自实现 MCP 协议，不进官方 go-sdk

选项代价收益
引入 modelcontextprotocol/go-sdk主 go.mod 新增 ~8 个传递依赖（jsonschema-go、oauth2、segmentio/encoding、x/tools、jwt 等）少写约 400 行代码
自实现最小 stdio 客户端 ✅~400 行 Go（含测试）主模块零新依赖；亲手实现协议可讲清原理；可裁剪到只用的 4 个 RPC
那如何保证手写实现没写错协议？把"连接 → 列工具 → 调用 → 错误结果"的协议契约沉淀成一套一致性套件，再配一个协议兼容的 fake server（由测试二进制自执行充当，无需外部进程），对手写客户端逐条断言——详见第 ⑦ 节。

03MCP stdio 协议（最小子集）

换行分隔 JSON-RPC 2.0

MCP stdio transport 的传输格式极其朴素：每条 JSON-RPC 2.0 消息独占一行（UTF-8），消息内不得含内嵌换行。client 写 server 的 stdin、读 server 的 stdout。这种格式有三个好处：

  • 不需要长度前缀或 Content-Length 头——只要 bufio.Scanner 按行扫即可；
  • 天然无背压死锁——读写各一根 goroutine，互不阻塞；
  • 本地通信、无网络面、无 SSRF 攻击面（§7 安全约束的天然满足）。

RPC 序列与握手

1
client → server: initialize

params 包含 protocolVersion（"2024-11-05"）、capabilities、clientInfo{Name,Version}。server 回 serverInfo / capabilities。

2
client → server: notifications/initialized

通知（无 id），告知 server 客户端就绪。

3
client → server: tools/list

server 回 {tools:[{name, description, inputSchema}]}。cogent 把每个工具包装为 remoteTool。

4
client → server: tools/call

params {name, arguments}，回 {content:[{type:"text",text}], isError}。cogent 把 text 串拼接为 ToolResult.Content、isError 直传。

internal/mcp/transport.go (核心片段)// 单 readLoop 按 id 把响应分发给等待中的 call；写入加锁串行化。
func (t *transport) readLoop(stdout io.Reader) {
    defer close(t.done)
    sc := bufio.NewScanner(stdout)
    sc.Buffer(make([]byte, 0, 64*1024), maxFrameBytes) // 防御异常超长行
    for sc.Scan() {
        line := bytes.TrimSpace(sc.Bytes())
        if len(line) == 0 { continue }
        var f frame
        if err := json.Unmarshal(line, &f); err != nil {
            continue // 跳过非法行，不毒化整条连接
        }
        if f.ID == nil { continue } // 服务端通知/请求：最小客户端不处理
        t.deliver(*f.ID, f) // 按 id 投递到 pending[id] channel
    }
    ...
}

04客户端实现：Connect → Tools → Close

internal/mcp/client.gotype MCPClient interface {
    Connect(ctx context.Context, cfg ServerConfig) error
    Tools() []tool.Tool
    Close() error
}

func (c *client) Connect(ctx context.Context, cfg ServerConfig) error {
    if err := cfg.validate(); err != nil { return err } // 名字白名单防注入
    cmd := exec.Command(cfg.Command, cfg.Args...)
    cmd.Env = mergeEnv(cfg.Env)
    stdin, _ := cmd.StdinPipe()
    stdout, _ := cmd.StdoutPipe()
    if err := cmd.Start(); err != nil { ... }
    c.tr = newTransport(stdout, stdin)
    return c.bootstrap(ctx) // initialize + initialized + tools/list
}

远端工具：fail-closed 默认 + 命名隔离

internal/mcp/remote_tool.gotype remoteTool struct {
    tool.Defaults // ★ 匿名嵌入：默认非并发安全、非只读、权限 ask
    client   *client
    origName string
    fullName string  // "mcp__<server>__<tool>"
    desc     string
    schema   json.RawMessage
}

func (t *remoteTool) Call(ctx context.Context, input json.RawMessage, _ tool.ProgressSink) (types.ToolResult, error) {
    ctx, end := t.client.tracer.Start(ctx, "mcp.call",
        observe.Attr{Key: "mcp.server", Value: t.client.name},
        observe.Attr{Key: "mcp.tool", Value: t.origName})
    res, err := t.client.callTool(ctx, t.origName, input)
    end(err)
    if err != nil { return types.ToolResult{Content: "mcp call failed: "+err.Error(), IsError: true}, nil }
    return res, nil
}
为什么 Call 不向上 return error？同 Phase 2 工具规范——传输/解码错误规范化为 IsError=true 的 ToolResult 回流，让模型自己理解工具失败并决定下一步（而不是把内核打挂）。这与内建工具的错误处理完全一致。

05工具池融合策略 ★（本阶段的安全核心）

核心问题：内建工具是可信的，外部 MCP 工具不可信。融合时必须同时解决三件事：

风险cogent 的对策
外部工具用同名（如 bash）劫持内建① 命名前缀 mcp__<server>__<tool> 命名隔离；② NewPool 已实现"先到先得"，把内建排在前面、MCP 排在后面注入即天然内建优先。
外部工具无人审计就放行所有 MCP 工具一律经 Guard 包裹——Defaults 已强制权限 ask，叠加静态 Policy 后默认走 HITL。
外部工具并发不安全导致竞态Defaults.IsConcurrencySafe 默认 false，Phase 3 的批分调度器会自动把它单独执行，不与其它工具混批。
cmd/cogent/commands.gofunc buildToolPool(workRoot string, prompter permission.Prompter, tracer observe.Tracer, mcpTools []tool.Tool) tool.Pool {
    policy := permission.StaticPolicy{}
    guard := func(t tool.Tool) tool.Tool { return tool.NewGuard(t, policy, prompter, tracer) }
    sb := sandbox.New(sandbox.Config{WorkRoot: workRoot, Enabled: true})
    tools := []tool.Tool{
        tool.NewReadFile(workRoot), tool.NewListDir(workRoot), tool.NewGrep(workRoot),
        guard(tool.NewWriteFile(workRoot)),
        guard(tool.NewEditFile(workRoot)),
        guard(tool.NewBash(sb, tracer)),
    }
    for _, mt := range mcpTools {
        tools = append(tools, guard(mt)) // ★ 外部不可信：统一过 permission/HITL
    }
    return tool.NewPool(tools...) // ★ 先到先得 ⇒ 内建优先
}
工程杠杆点："你怎么防止恶意 MCP server？"——三层闭环：命名前缀使其无法冒用内建名；池装配先到先得使同名时内建必胜；Guard + Defaults 让所有 MCP 工具默认走 HITL 询问。三者任一失守都不致命，构成纵深防御。

09设计要点回顾（六宫格）

① 接口隔离

MCPClient 是唯一缝隙；engine 只看见 tool.Tool，对 MCP 一无所知。换协议、换实现都不破坏内核。

② fail-closed 默认

tool.Defaults 匿名嵌入让外部工具默认非并发安全 + ask；忘记覆写 = 退化为最安全行为，把约定写进语言层。

③ 命名前缀 = 命名空间

mcp__<server>__<tool> 物理上隔绝同名劫持；server 名走字符白名单防注入。

④ 池装配先到先得

复用 Phase 1 的 NewPool 同名去重；只需把 MCP 排在内建之后传入即天然内建优先——零新增机制。

⑤ Guard 装饰器复用

外部工具走与写/执行类内建工具完全相同的权限闸门；不发明新机制 = 不引入新攻击面。

⑥ 协议契约可测

把"连接→列工具→调用→错误结果"沉淀成内联一致性套件 + 协议兼容 fake server（测试二进制自执行充当），手写实现是否合规一目了然；全在主模块内、零额外依赖。

STAR 要点

S（Situation）：Agent 工具池如果只能编进进程，能力扩展靠改源码重发版，与 MCP 生态绝缘。

T（Task）：接入 MCP 客户端，让 cogent 能动态接入任意 stdio MCP server 的工具，但不能为此牺牲零依赖、安全默认、内核稳定三条底线。

A（Action）：① 自实现最小 stdio JSON-RPC 客户端约 400 行，主 go.mod 零新增依赖；② 远端工具匿名嵌入 tool.Defaults + 命名前缀 mcp__server__tool，经 Guard 统一过权限闸门后置于内建之后入池，借 NewPool 先到先得实现内建优先；③ 把协议契约沉淀成内联一致性套件 + 协议兼容 fake server（测试二进制自执行充当 server），对手写客户端逐条断言「连接→列工具→调用→错误结果」，全程零外部进程、零额外依赖。

R（Result）：主模块 gofmt / go build / go vet / go test -race ./... 全绿，go.mod 一字未改；CLI cogent mcp 自检 + run/resume 自动装配 MCP 工具；纵深防御（命名隔离 + 池先到先得 + Guard）三层闭环防御恶意 server。
