cogent Phase 4：让 Agent 跑得动「长任务」

前三阶段的 cogent 能对话、能动手、能并发安全地干活，但长任务的对话历史会无情地撑爆模型上下文窗口。Phase 4 引入自动压缩（auto-compact）+ 分层记忆（MEMORY.md）：历史超窗时压成摘要并重注入关键状态，跨会话从 .cogent/MEMORY.md 加载项目约定——压缩后模型仍"站在工作台旁"，而不是失忆重来。

01为什么需要上下文工程

LLM 的上下文窗口是有限且昂贵的资源。一个自主 Agent 在长任务里会不断累积：用户任务、每轮模型回复、每次工具调用的 tool_use 与回流的 tool_result（文件内容、命令输出动辄上千 token）。轮数一多，历史就会突破窗口上限，请求直接报错。

朴素做法的问题

简单截断头部历史 → 模型"失忆"：忘了在读哪个文件、计划进行到哪一步；更糟的是可能切断 tool_use ↔ tool_result 配对，让 API 直接报"孤立 tool_result"错误。

cogent 的做法

阈值触发时把旧历史压成摘要（调 LLM），保留尾部最近原文，并重注入关键状态；切点向头部平移以绝不拆散配对；连续失败则熔断止损。

DeepSeek 的特殊性：deepseek-chat 窗口约 128K、单次输出上限 ~8K，仍比 Claude 的 200K 紧张——这让压缩策略是"长任务能否跑通"的关键。所以 cogent 的窗口常量以 128K 调参。

03窗口模型与 ShouldCompact

核心公式：有效窗口 = 总窗口 − 摘要预留。再留一个缓冲提前触发，避免"刚好到顶才压缩"时连摘要请求都发不出去。

internal/contextmgr/manager.go// 以 DeepSeek 128K 为基准，均可经 env 覆盖。
const (
    ContextWindowDefault   = 131072 // 总窗口：DeepSeek 128K
    ReservedForSummary     = 8000   // 摘要预留（同时作摘要调用 MaxTokens 上限）
    AutoCompactBufferToken = 8000   // 提前触发压缩的缓冲
    MaxConsecutiveFailures = 3      // 压缩连续失败熔断阈值
    KeepRecentTokens       = 24000  // 尾部保留下限（随窗口翻倍）
)

func (m *Manager) effectiveWindow() int { return m.window - m.reserved }

// ShouldCompact 依据已用 token 与有效窗口判断是否压缩；熔断后恒 false。
func (m *Manager) ShouldCompact(usedTokens int, _ string) bool {
    if m.failures >= m.maxFail {
        return false // 熔断
    }
    return usedTokens+m.buffer >= m.effectiveWindow()
}
env 可覆盖：COGENT_CONTEXT_WINDOWCOGENT_CONTEXT_RESERVECOGENT_CONTEXT_BUFFERCOGENT_CONTEXT_KEEP—— 换模型（如 Claude 200K）只需调环境变量，不改代码；单测也靠它把窗口收到极小值来稳定触发压缩。

熔断机制：为什么重要

如果压缩本身一直失败（如摘要请求超时、模型抽风），还反复触发压缩，就会陷入"越压越失败、越失败越压"的死循环，白白烧钱。所以 failures 累加到阈值后，ShouldCompact 直接返回 false 永久止损（成功一次则归零）。

工程杠杆：蓝本（Claude Code auto-compact）的公开数据是熔断每天省下约 25 万次"死锁"API 调用。这是"生产级而非玩具"的成本意识体现。

04Compact 四步流程

Compact 被拆成四个 < 80 行的小函数，职责单一、便于单测：

1
findCutPoint

从尾部向头累加 token，直到达到 KeepRecentTokens，该索引即切点 cut：msgs[cut:] 为保留段。切点不越过系统提示（cut ≥ 1）。

2
adjustForPairing

把切点向头部平移，避免保留段以孤立 tool_result 开头（详见下节）。

3
summarize

把被丢弃段 msgs[1:cut] 序列化为文本，调 LLM 生成一条状态摘要；range channel 收齐全部增量。失败则累加 failures。

4
rebuild

重建为 [系统提示] + [摘要(RoleUser)] + [尾部最近消息]。成功后 failures 归零。

internal/contextmgr/manager.gofunc (m *Manager) Compact(ctx context.Context, msgs []types.Message, llmc llm.Client) ([]types.Message, error) {
    if len(msgs) < 2 {
        return msgs, nil // 仅有系统提示，无可压缩历史
    }
    cut := adjustForPairing(msgs, m.findCutPoint(msgs))
    if cut <= 1 {
        return msgs, nil // 尾部已覆盖全部历史，无需压缩
    }
    summary, err := summarize(ctx, llmc, msgs[1:cut])
    if err != nil {
        return m.recordFailure(msgs, err) // 累加失败；达阈值返回 ErrCompactGiveUp
    }
    m.failures = 0
    return rebuild(msgs, cut, summary), nil
}
fail-safe 不变量：任何失败路径都返回原始消息列表，绝不丢历史——压缩是"优化"，失败只是"没优化成"，决不能因优化失败而破坏任务。

切点平移：保住 function calling 配对

这是整个 Phase 最微妙、也最值钱的不变量。OpenAI 兼容协议要求：每个 tool（结果）消息，必须紧跟在携带对应 tool_calls 的 assistant 消息之后。若切点恰好落在配对中间，保留段就会以一个"没有爹"的 tool_result 开头，llm/client.go 的 toOpenAIMessage 会把它发出去，API 直接报错。

配对平移示意  index:  0       1        2                    3              4
        [system] [user] [assistant(tool_calls c1)] [tool c1] [user]
                                                    ▲
                          findCutPoint 落在 index 3 ─┘（tool_result）
                          ✗ 保留段 = [tool c1, user] → 孤立 tool_result！

  adjustForPairing：cut 是 tool → 向头部平移 cut-- 直到非 tool
                                            ▼
                          cut = 2（assistant 含 tool_calls）
                          ✓ 保留段 = [assistant(c1), tool c1, user] → 配对完整
internal/contextmgr/manager.go// adjustForPairing 把切点向头部平移，避免保留段以孤立 tool_result 开头。
func adjustForPairing(msgs []types.Message, cut int) int {
    for cut > 1 && msgs[cut].Role == types.RoleTool {
        cut--
    }
    return cut
}
对应单测：TestAdjustForPairing 直接断言"切点落在 tool_result 上 → 平移到 assistant"；TestManager_Compact_PreservesPairing 端到端断言重建后不存在孤立 tool_result。

05状态重注入：压缩后仍"站在工作台旁"

压缩不是简单丢弃，而是把旧历史的关键状态蒸馏进摘要。cogent v1 通过精心设计的摘要 system prompt 实现"软重注入"——明确要求模型保留正在读/改的文件、进行中的计划、关键决策与待办：

internal/contextmgr/manager.go · summaryPrompt"You are compacting an autonomous coding agent's conversation history... "
"Preserve: the user's overall goal, files currently being read or edited, "
"the in-progress plan, key decisions and findings, and any pending next steps. "
"Omit chit-chat. Output only the summary text."
为什么工具声明无需重注入？因为 engine 每轮请求都会重新下发 toolSchemas()（运行期只读的工具池），工具声明天然"常驻"，不在历史里，自然不受压缩影响。

06分层 Memory：读写双向 + 跨会话沉淀又不臃肿

记忆若无限注入会再次撑爆上下文。cogent 的分层记忆同时覆盖读与写——借鉴 CodeBuddy 的双层设计：

读 · Loader.Build

从 .cogent/MEMORY.md 读取长期记忆入口，注入系统提示。行数(200)+字节(25000)双重硬截断取严，缺失返空不报错。

写 · Writer

让 agent 运行中把决策/教训持久化。两个动作：AppendDaily（追加到 daily/YYYY-MM-DD.md，append-only 保审计链）+ UpdateMemory（覆写 MEMORY.md，curated 长期记忆）。

设计借鉴：CodeBuddy（IDE Agent 运行时）自身在 .codebuddy/memory/ 落了双层记忆——daily（append-only + 纠正条目，保审计可追溯）+ MEMORY.md（curated 长期）。cogent 的 Writer 对齐这一设计：daily 追加不重写，MEMORY 覆写由调用方合并内容。future 可在此基础上做 30 天 distill（日志蒸馏进长期记忆）。

行截断

最多 200 行（MaxEntrypointLines），bufio.Scanner 逐行计数。

字节截断

最多 25000 字节（MaxEntrypointBytes），且不切断 UTF-8 字符。

路径防护

filepath.Clean + 前缀校验，读写均防 .. 穿越出工作根。

07engine 集成：最小侵入

遵循向后兼容原则——Deps.Context 与 Deps.Memory 都是可选的，为 nil 时跳过，不破坏 Phase 0–3 的既有测试。

① 记忆注入系统提示（New 时一次性）

internal/engine/engine.go// buildSystemPrompt 把基础系统提示与项目记忆（若有）拼成最终系统提示。
func buildSystemPrompt(mem memory.Loader, workRoot string) string {
    if mem == nil {
        return systemPrompt
    }
    text, err := mem.Build(context.Background(), workRoot)
    if err != nil {
        slog.Warn("load memory", "err", err) // 记忆失败仅告警，不阻塞内核
        return systemPrompt
    }
    if strings.TrimSpace(text) == "" {
        return systemPrompt
    }
    return systemPrompt + "\n\n# Project memory (.cogent/MEMORY.md)\n" + text
}

② 主循环每轮判断压缩

// maybeCompact 在触达阈值时压缩历史，成功后上抛 EventCompacted。
func (e *engine) maybeCompact(ctx context.Context, out chan<- types.StreamEvent) {
    if e.ctxmgr == nil {
        return
    }
    used := e.used
    if used == 0 {
        used = contextmgr.EstimateTokens(e.msgs) // 无真实 usage 时回退估算
    }
    if !e.ctxmgr.ShouldCompact(used, e.model) {
        return
    }
    cctx, end := e.tracer.Start(ctx, "ctx.compact") // 可观测 span
    compacted, err := e.ctxmgr.Compact(cctx, e.msgs, e.llm)
    end(err)
    if err != nil {
        slog.Warn("context compact failed", "err", err)
        return
    }
    e.msgs = compacted
    e.used = contextmgr.EstimateTokens(e.msgs)
    emitEvent(ctx, out, types.StreamEvent{Type: types.EventCompacted})
}

③ token 估算与校准

used 优先取自 LLM 返回的真实 usage（streamAssistant 里把 Delta.Usage.PromptTokens + CompletionTokens 写回 e.used）；fake/无 usage 时回退 len([]rune)/4 的粗估。压缩后用估算重置 e.used，反映新的（更小的）上下文规模。

为什么 used 是"设置"而非"累加"？因为 PromptTokens 已经包含了完整发送的历史规模——它本身就是"当前上下文有多大"的最佳信号；累加会无界增长、误判。

09设计要点回顾

① 有效窗口 + 缓冲

总窗口 − 摘要预留 − 缓冲提前触发，给摘要请求留出发送空间，避免"到顶才压"的死局。

② 配对不变量优先

切点向头部平移，绝不拆散 tool_use ↔ tool_result——这是 function calling 协议的硬约束，由专门单测守护。

③ 熔断止损

连续失败达阈值即停手，把"越压越败"的烧钱风暴掐死，体现生产级成本意识。

④ fail-safe

任何失败都返回原历史，压缩是优化而非义务，决不因优化失败破坏任务。

⑤ 状态重注入

摘要 prompt 显式要求保留在读文件/计划/决策，让模型压缩后仍"站在工作台旁"。

⑥ 可选 + 向后兼容

Context/Memory 为 nil 即跳过，旧测试零改动通过；env 可覆盖窗口参数，换模型零代码改动。
