图 1 · Git 四大存储区域与它们之间的命令流

工作区（Working Directory）

就是你电脑上的文件夹——你用编辑器打开、能看到、能修改的那些文件。

类比：你的书桌，正在写的草稿就摊在上面。

暂存区（Staging Area / Index）

一个「待发货清单」。git add 把文件放进暂存区，意思是「这些改动我确认了，准备提交」。

类比：打包台。你把写好的草稿放到打包台上，准备寄出去。

本地仓库（Local Repository）

git commit 把暂存区的内容永久记录到仓库历史中。每次 commit 是一个不可篡改的快照。

类比：已寄出的快递。包裹已经发走，有物流单号（commit SHA），随时可以追溯。

Stash（临时保险箱）

把当前改动「暂存到一边」，工作区恢复干净。需要时再取出来。详见 §2。

类比：抽屉。桌面太乱了，先把草稿塞进抽屉，等会儿再拿出来继续写。

§2 git stash 到底是什么

Stash 是 git 提供的「临时保险箱」——把当前工作区的改动打成一个包存起来，工作区恢复干净。需要时再把包取出来。

普通 stash vs cogent 用的 stash create

git stash（普通）git stash create（cogent 用的）
效果保存改动 + 清空工作区只创建快照对象，不清空工作区
stash 列表出现在 git stash list不出现（只返回 SHA）
恢复方式git stash popgit stash apply <SHA>
用途人工暂存：切分支前保存进度程序化快照：Undo 用
为什么 cogent 用 stash create 而不是 stash push？
因为 stash create 是「只拍照不挪东西」——工作区文件原封不动，Agent 可以继续干活。 普通 stash 会把桌面清空，Agent 就没东西可改了。

§3 git worktree 到底是什么

一个 git 仓库通常只有一个工作区（你 cd 进去看到的那些文件）。但 git worktree add 可以在另一个目录创建第二个、第三个工作区——它们共享同一个 .git 仓库，但各自检出不同的分支，互不干扰。

图 3 · 一个 .git 仓库 + 多个独立工作区，各自有自己的分支和文件

普通分支切换
git checkout feature

在同一个目录里切换分支——你的文件会被覆盖。

问题：Agent 正在改 feature-A 的代码时，你不能同时改 feature-B。

Worktree
git worktree add /tmp/wt feature

在另一个目录创建一个独立工作区，检出 feature 分支。

好处：主目录完全不受影响，两个 Agent 可以并行在不同 worktree 里改代码。

cogent 为什么需要 worktree？

核心目的：物理隔离，通过才合并。

没有 worktree 时：Maker 直接改你的代码 → 审查不通过 → 得用 git checkout -- . && git clean -fd 回滚。
风险：回滚可能不彻底（比如改了 .gitignore 里的文件）。

有了 worktree：Maker 在 /tmp/cogent-wt-xxx/ 里改 → 审查不通过 → 直接删掉整个目录。
你的代码从头到尾没被碰过——零风险。

关键设计

  • stash create 不入 stash 列表、不清空工作区——Agent 无感知，不影响正常工作
  • 快照后立即 git reset 撤销 add 副作用——暂存区保持干净

§8 安全设计

安全
Sandbox 保护（worktree / review）

Worktree 和 Review 的所有 git 命令都经 sandbox.Exec() 执行——sandbox 有危险命令白名单拦截和工作目录约束。

Agent 不可能通过工具调用执行 git push --force 或 rm -rf /。

直接执行（snapshot）

Undo 的快照操作使用 exec.Command 直接执行——因为快照属于引擎内部可信控制面，不是 Agent 的工具调用。

Agent 无法触发或篡改快照命令。

bare repo 逃逸防御

Sandbox 在每次命令执行后检测工作根是否新出现了 bare git repo 的签名文件（HEAD/objects/refs），如果有就清理——防止恶意 Agent 通过伪造 git 仓库逃逸沙箱。

Worktree 命名隔离

所有临时分支以 cogent/wt- 开头，目录以 cogent-wt- 开头——便于识别和批量清理，不会与用户的分支冲突。
