0. 先搞清楚：数据从哪来、到哪去？

整个可观测模块的数据流可以用一句话概括：

内核各层调用 tracer.Start(ctx, "span.name") 产出 span → 经 OTel SDK 异步批量导出 → 落地到文件/Jaeger

1. 启动阶段：从环境变量到 Provider

cogent 启动时，commands.go 按环境变量构造 observe.Config，然后调用 observe.New(cfg) 得到 Provider。

⚡ 关键设计：默认 COGENT_OBSERVE_ENABLED=false，返回 noopProvider。
所有 tracer.Start() 调用点恒定存在于代码中，但 no-op 时零开销——没有 if 分支污染业务代码。

2. 埋点全景：9 个 span 分布在哪些文件

cogent 的 9 个 span 埋点分布在 5 个内核文件中，每个 span 对应 ReAct 循环的一个关键步骤：

#Span 名文件:行号触发时机关键 Attributes
①cogent.sessionengine.go:Run()任务开始session.id, task, model
②react.stepengine.go:197每轮 ReAct 循环step.index
③llm.streamengine.go:267每次 LLM 流式调用llm.model, tokens, ttft_ms
④tool.batchorchestrate.go:86每个调度批batch.concurrent, batch.size
⑤tool.callexec.go:33单次工具调用tool.name, is_error, duration_ms
⑥permission.checkguard.go:65权限判定permission.behavior
⑦sandbox.execbash.go:60沙箱命令执行command.first, exit_code
⑧agent.spawnspawner.go:58子 Agent 派发—
⑨ctx.compactengine.go:231上下文压缩compact.triggered
⑩mcp.callremote_tool.go:63MCP 远端工具调用mcp.server, mcp.tool
🎯 核心认知：这些埋点在 Phase 0~7 就以 no-op 形式写好了。Phase 8 没有新增任何调用点，只是把 observe 包内部的 noopTracer 换成了 otelTracer——这就是"换内胆不换接口"。

3. 核心机制：ctx 如何串联父子 span

这是 Phase 8 最值钱的认知点。tracer.Start(ctx, name) 返回一个派生 ctx，把新 span 绑进去；继续往下传这个 ctx，子 span 自动认父——无需手动传 trace_id。

// 代码中的统一模式（所有 9 个埋点都长这样）：
ctx, end := e.tracer.Start(ctx, "span.name", observe.Attr{Key: "k", Value: v})
defer func() { end(err) }()  // 出错自动 RecordError + SetStatus(Error)
// ... 业务逻辑，继续把 ctx 往下传 ...

5. Exporter 三态：数据最终去哪

exporter.go 是一个工厂，按 Config.Exporter 字符串选择不同的导出后端：

// 使用示例：
// 本地文件（零依赖）
COGENT_OBSERVE_ENABLED=true COGENT_TRACE_EXPORTER=file go run ./cmd/cogent run "修复 bug"
// → data/traces/traces-20260618-165132.jsonl

// 接 Jaeger 看火焰图
docker run -d --name jaeger -p 16686:16686 -p 4317:4317 jaegertracing/all-in-one:latest
COGENT_OBSERVE_ENABLED=true COGENT_TRACE_EXPORTER=otlp go run ./cmd/cogent run "修复 bug"
// → 打开 http://localhost:16686 看 span 树

6. Meter 指标：计数器与直方图

除了 trace（调用树），observe 还提供 Meter（指标），用于统计性数据。

7. slog 对齐：日志如何关联 trace

光有 trace 还不够——排查时常要在"看 span"和"看日志"之间跳。traceHandler 让每条日志自动带上 trace_id/span_id。

⚠️ 易踩的坑：WithAttrs/WithGroup 必须重新包一层 traceHandler（而不是直接返回 inner 的结果），否则 slog.With(...) 之后注入能力就丢了——这是包装 slog.Handler 的经典陷阱。

8. Shutdown：优雅退出保证数据不丢

BatchSpanProcessor 是异步的——如果程序直接退出，队列里还没导出的 span 就丢了。所以必须在退出前调用 Shutdown。

9. no-op 降级：零开销的秘密

当 Enabled=false 时，observe.New() 返回 noopProvider。所有调用点的代码一行不变，但实际执行的是空函数。

🎓 最终总结（背下来）：

环境变量 → observeConfig() → observe.New(cfg) → Provider(Tracer+Meter)
→ 内核各层 tracer.Start(ctx, name) 产出 span → ctx 自动串联父子
→ BatchSpanProcessor 异步批量 → Exporter 落地(file/stdout/otlp)
→ Shutdown flush 保证不丢

整个模块就是一个 "把 ReAct 决策链变成 OTel span 树，经异步批量导出到文件或 Jaeger" 的可观测层。
核心设计：接口多态（no-op/OTel）+ ctx 双重职责（取消+trace）+ 换内胆不换接口。
