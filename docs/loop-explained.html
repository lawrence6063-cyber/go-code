LOOP ENGINEERING · 循环工程

什么是 Loop？

不再「你一句一句指挥 AI 干活」，而是设计一个循环，让循环替你指挥 AI，直到目标真正达成。

核心顿悟 Loop 公式 三个高度 对比例子 完整流程图 六原语 /goal 演示 护栏 怎么开 loop

1核心顿悟：人，从循环里走出来

过去我们用 AI，是人坐在循环里：你提问 → AI 回答 → 你看不满意 → 你再提问……每一圈都要你亲自踩一脚油门。

Loop 的思想是：把「人踩油门」这件事本身自动化掉。你只定义「什么算成功」，然后让一个程序循环去反复驱动 AI，自己判断、自己重试，直到成功。人退到循环外面，只在最后看结果。

同样是「转圈」，区别在于谁在踩油门：是你，还是一段代码。
一句话：Loop = 把「不断 prompt AI 直到满意」这个动作，从你的手，搬到一段会自己判断的程序里。

2Loop = 闹钟 + 判官

一个最小的 Loop，其实只需要两样东西：

⏰ 触发器（何时干）

  • 定时：每 10 分钟 / 每天早上 9 点
  • 事件：有人提了新 issue
  • 监听：CI 变红了 / 文件被改了

⚖️ 判定器（干完没）

  • 客观脚本：go test ./... 全绿吗？
  • 独立模型：「这篇文档覆盖全要点了吗？」
  • 关键：不能让干活的 AI 自己给自己打分

没有判定器，AI 会「自我宣告完成」——嘴上说改好了，代码其实编译都过不了。判定器就是那个不讲情面、只认结果的「判官」。

3三个高度：你现在站在哪一层？

用 AI 写代码，能力有三层台阶。Loop 的目标是把你从「高度 2」推到「高度 3」。

高度谁在驱动什么时候停例子
1 手动你，每一步你说停在聊天框反复贴报错
2 单次任务Agent 自己跑一轮撞到步数上限 / 它觉得完了「帮我改这个函数」跑完就退
3 持续自治Loop 程序反复驱动判官说达标了（或撞预算）「修到测试全绿」，自己重试 5 轮

4同一个任务，两种活法

任务：「修复这个有 bug 的函数，要求所有单元测试通过。」

高度 2单次任务 Agent

  1. 你下指令
  2. Agent 改了代码
  3. Agent 说：「改好了！」✅
  4. 跑完，退出
  5. 你一跑测试 —— 还是红的 😤
  6. 你只好再贴报错，回到第 1 步……

问题：没人验收。Agent 的「我觉得好了」≠ 真的好了。

高度 3Loop 目标循环

  1. 你给目标 + 验收脚本 go test
  2. Agent 改代码
  3. 判官自动跑测试 → 还红 ❌
  4. 把失败原因喂回给 Agent，再来一轮
  5. 判官再跑 → 绿了 ✅ → 停！
  6. 你回来，已经是通过的代码 🎉

关键：判官认结果不认嘴，不达标自动重试。

用伪代码看清这个「循环」

// 高度 3 的本质：一个 while 循环 + 一个判官
task := "修复 bug，让所有测试通过"

for iter := 0; iter < maxIterations; iter++ {   // ← 预算护栏：最多转 N 圈
    agent.Run(task)                 // 让 AI 干一轮活

    report := judge.Verify()          // 判官跑 go test，看退出码

    if report.Passed {                  // ✅ 达标了
        return "成功"                    //    立刻停，这是唯一的成功出口
    }
    task = "上一轮没过，原因是：" + report.Detail  // ← 把失败反馈拼进下一轮
}
return "撞到预算上限，停"          // 兜底：绝不无限烧钱

看，所谓 Loop，去掉花哨概念，核心就是这段 for 循环：干活 → 验收 → 不行就带反馈再来。

6搭一个 Loop，需要的六块积木

把一个能跑的 Loop 系统拆开，由六个「原语」（基础能力）组成。你可以对照下面理解每一块各管什么：

Automations 触发

定时/事件让 Loop「自己醒来」干活——这是自治的起点。例：CI 一红就触发修复。

/goal 目标判定 ⭐

最关键。给「可验证的成功标准」，由独立判官裁决，达标才停。例：测试全绿。

Sub-agents 分工

「实现者」写代码，「审查者」挑毛病——不让一个人既写又自己批改。

State 进度看板

把「已做/待做/卡住」写进磁盘文件，重启后仍记得跑到哪了——「仓库不忘记」。

Worktrees 隔离

多个 Agent 并行时各占一个独立工作区，互不踩脚，干完再合并。

Skills 技能包

把「本项目怎么加限流」这类做法固化成文件，下次循环直接复用。

外加第六块 Connectors（连接器 / MCP）：让 Loop 能触达真实世界——读数据库、发 PR、查 issue。它是 Loop 伸向外部工具的「手」。

这六块里，最不能省的是 /goal（独立判定）。没有它，循环不知道何时该停，要么提前放弃，要么无限空转。

7慢动作：一次目标循环的 5 轮

目标：把测试通过率从 0 修到 100%。看判官如何一轮轮逼近：

第 3 轮没进步？判官如实反馈「这块还没动」，Agent 换思路，第 4、5 轮追上来。判官的诚实反馈，是循环能进步的燃料。

每一轮，判官到底说了什么

轮判官反馈（喂回给 Agent）结果
13/10 通过，TestParse 空指针崩溃❌ 继续
2修了崩溃，还有 4 个断言不符❌ 继续
3没新进展，改错了文件❌ 继续
48/10，边界值 0 和负数没处理❌ 继续
510/10 全绿✅ 达标，停

8最重要的安全带：预算护栏

让 AI 自己转圈很爽，但最大的风险是失控烧钱——目标永远达不到，循环停不下来，一晚上烧掉几百刀。所以任何 Loop 都必须带三重刹车：

🔢

最多 N 轮

比如最多重试 8 次，到了就停

💸

最多 $X

累计花费到 5 美元就刹车

⏱️

最多 T 分钟

跑满 15 分钟强制收手

两个出口，缺一不可：一个奖励成功，一个防止失控。这就是「无人值守」敢放手的底气。

9开发者怎么开一个 Loop？和传统 prompt 啥区别

最直观的区别：传统 prompt 你交付的是「一句话指令」，开 loop 你交付的是「一份合同」——目标是什么、怎么算验收、最多花多少预算。

传统 prompt 是「一次性对话」，loop 是「带验收和预算的托管任务」。

三种开 loop 的姿势（由轻到重）

姿势一 · 一次性目标循环（前台，最常用）

「我盯着，但别让我一句句催」。给目标 + 验收脚本，它自己反复干到达标。

# 给目标、给验收脚本、给预算护栏，回车后就自动转圈
cogent goal \
  --intent="修复 parser 的 bug，让所有单元测试通过" \
  --verify=./verify.sh \         # ← 判官：退出码 0 = 达标
  --max-iterations=8 \           # ← 护栏：最多重试 8 轮
  --max-cost=5 --max-wallclock=15m

其中 verify.sh 可以简单到一行：go test ./... （测试过就退出 0，loop 据此判定达标）。

姿势二 · 用配置文件描述目标

目标复杂时，写进文件而非一长串命令行参数。

# .cogent/goal.md
目标：给 user 服务加上限流中间件，QPS 上限 100
验收：bash ./verify_ratelimit.sh   # 压测脚本，超限返回 429 才算过
预算：最多 10 轮 / $8 / 30 分钟

# 然后：
cogent goal --goal-file=.cogent/goal.md

姿势三 · 守护进程（后台 / 无人值守）

「我走了，你自己醒来找活干」。定时或监听事件触发，这才是完整的「自治 loop」。

# 每 10 分钟醒一次，检查有没有失败测试，有就自动修
cogent loop --interval=10m --goal-file=.cogent/goal.md \
            --max-iterations=8 --max-cost=5

# 或：盯着文件变化，CI 一红就触发
cogent loop --watch=. --verify=./verify.sh

# Ctrl-C 随时优雅停下，进度已写进 .cogent/progress.md

本质区别一览

维度传统 Prompt开 Loop
你交付什么一句指令目标 + 验收标准 + 预算
谁判断「做完了」你，肉眼独立判官（脚本/模型）
不满意怎么办你手动再问一遍自动带反馈重试
什么时候停你说停达标停 / 撞预算停
你要不要盯着全程盯可以走开，甚至无人值守
能否定时自动跑不能能（守护进程）
失控风险低（你控制每一步）有 → 必须靠预算护栏兜底

从 0 开一个 loop，就 4 步

1写清目标

一句话说明「要达成什么」。越具体越好，别留模糊空间。

2定义验收（最关键）

写一个能客观判定成败的脚本/标准——退出码、测试、压测。这是 loop 和 prompt 最大的分水岭。

3设好预算护栏

最多几轮、几块钱、几分钟。没护栏不许启动——防失控烧钱。

4选触发方式

手动跑一次（goal），还是定时/监听自动跑（loop）。

记住这个心法转变：传统 prompt 你是「司机」，每一步都要打方向盘；开 loop 你是「导航员」——你设好目的地和油量上限，车自己开，到了或没油了才叫你。

✦三句话记住 Loop

1️⃣ 人走出循环：你定义「什么算成功」，不再亲手一步步指挥。

2️⃣ 判官认结果：独立验收，不达标就带反馈重来——AI 不能自己给自己打分。

3️⃣ 护栏兜底：达标停 + 撞预算停，永远不无限烧钱。

理解了这三点，你就明白了为什么 cogent 要新增一个 internal/loop 编排层，把现有的「单次任务引擎」包成「目标循环」——这正是 LOOP_SPEC.md 在设计的事。
