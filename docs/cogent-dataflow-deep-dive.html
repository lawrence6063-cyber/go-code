cogent 是什么

cogent（Coding aGENT in Go）是一个生产级自主编码 Agent 运行时。给定自然语言任务（如"修复这个 bug"），它自主完成： 探索代码 → 制定计划 → 调用工具读写文件/执行命令 → 跑测试验证 → 失败自我修正。全过程流式可见、可中断、可恢复。

① 前置知识 · 5 个核心数据类型

整个系统就是这几种数据结构在不断地变形、流动、配对。它们全部定义在最内层的 internal/types 包（零业务依赖，杜绝循环引用）。

types.Message —— 贯穿全系统的"单一真相源"

这是最重要的类型。用户输入、助手回复、工具结果，统统是 Message。Engine 持有的 []Message 切片就是整个对话的真相。

type Message struct {
    Role      Role           // system | user | assistant | tool
    Text      string         // 文本内容
    ToolCalls []ToolUseBlock  // 仅 assistant：模型发起的工具调用
    ToolUseID string         // 仅 tool：与 ToolUseBlock.ID 配对 ★关键
    ToolName  string         // 仅 tool：对应的工具名
}
配对关系（理解的关键）：当模型要调用工具时，产生一条 assistant 消息，其 ToolCalls 里每个 ToolUseBlock 有个 ID。工具执行后，产生一条 tool 消息，其 ToolUseID 必须等于那个 ID。这个配对一旦错乱，LLM 接口就会报错。全系统多处逻辑（压缩、恢复、并发）都在小心维护这个配对。

types.ToolUseBlock —— 一次工具调用请求

type ToolUseBlock struct {
    ID    string          // 调用 ID，用于和 ToolResult 配对
    Name  string          // 工具名，如 "read_file" / "bash"
    Input json.RawMessage // 工具入参（原始 JSON，延迟解析）
}

types.ToolResult —— 一次工具执行结果

type ToolResult struct {
    Content string // 结果文本（文件内容 / 命令输出 / 错误信息）
    IsError bool   // 是否失败；失败也回流给模型，让它自我修正
}
失败即数据，不是异常：工具出错不会中断主循环，而是被规范化为 ToolResult{IsError:true} 回流给模型当作普通反馈。这是 Agent 能"自我修正"的根基。

llm.Delta —— LLM 流式响应的增量片段

type Delta struct {
    Text         string              // 文本增量（逐 token）
    ToolCall     *types.ToolUseBlock // 累积完成的一次工具调用
    Usage        *Usage              // token 用量（流末尾）
    FinishReason string              // stop / tool_calls / length
    Err          error               // 流内错误
}

types.StreamEvent —— 内核向 UI 输出的统一事件

Engine 不直接操作终端。它只往一个 <-chan StreamEvent 里吐事件，UI 层（CLI）消费并渲染。这是内核与 UI 解耦的核心。

type StreamEvent struct {
    Type    EventType          // text|tool_start|tool_result|compacted|done|error
    Text    string             // EventText：文本增量
    ToolUse *ToolUseBlock      // EventToolStart：将要执行的工具
    Result  *ToolResult        // EventToolResult：执行结果
    Err     error              // EventError：错误
}
EventType含义UI 如何渲染
EventText文本增量到达逐字打印到终端
EventToolStart开始执行某工具显示"🔧 调用 read_file..."
EventToolResult工具执行完毕显示结果摘要
EventCompacted上下文被压缩显示"📦 历史已压缩"
EventDone本轮完成结束，等待下一轮输入
EventError发生错误红色错误提示
记住这句话：整个 cogent 的运行，本质上就是 []Message → Request → Delta 流 → 新 Message（含 ToolUseBlock）→ ToolResult → tool Message → 回到 []Message 的循环，每一步都顺带往外吐 StreamEvent 给 UI。

③ 分层依赖架构

20 个 internal 包按严格单向依赖分层：上层可依赖下层，下层永不反向依赖上层（杜绝循环引用）。最底层 types 被所有人依赖却不依赖任何人。

关键解耦技巧：3 处接口如何打断循环依赖

分层最难的是避免循环依赖。cogent 用了三个"接口定义在消费侧"的技巧：

  • tool.Spawner：task 工具要派发子 Agent，但 tool 不能依赖 agent（否则 agent→engine→tool→agent 成环）。解法：在 tool 包定义 Spawner 接口，agent.SubAgent 隐式满足它，由 cmd 层注入。
  • loop.Pipeline：外层循环要驱动 maker/reviewer 流水线，但接口定义在消费侧 loop，cmd 用 pipelineAdapter 桥接到 agent。
  • loop.engineRunner：循环只需要 Run() 方法，于是只声明窄接口，不引入完整 Engine 依赖。

记忆口诀："谁用谁定义接口"——依赖倒置，把具体实现的注入推迟到最顶层的 cmd。

④ 路径 1 · 主交互流逐步拆解

这是最核心的一条路径。我们跟踪一次完整的 cogent run "fix the bug"，看数据如何一步步变形。

1

CLI 入口 → 装配阶段

Cobra 解析到 run 子命令（cmd/cogent/commands.go），调用 buildEngine() 装配全部依赖：LLM 客户端、工具池（buildToolPool()）、会话存储、上下文管理器、可观测 Provider。装配产物是一个 engine.Engine 实例，其内部已持有一条初始消息：

// engine.New 末尾
e.msgs = []types.Message{{
    Role: types.RoleSystem,
    Text: buildSystemPrompt(deps.Memory, e.workRoot), // 系统提示 + 项目记忆
}}

输入：CLI 参数字符串。输出：Engine（内含 []Message，长度=1，仅 system）。

2

Engine.Run() → 消息初始化

Run(ctx, task) 把任务文本包装成 user 消息追加进历史，落盘记录，然后启动 goroutine 跑 step()，立即返回一个 <-chan StreamEvent 给 CLI。

func (e *engine) Run(ctx, task string) (<-chan types.StreamEvent, error) {
    out := make(chan types.StreamEvent, 16)
    go func() {
        defer close(out)
        e.takeTurnSnapshot(ctx)              // 为 Undo 拍快照
        userMsg := types.Message{Role: RoleUser, Text: task}
        e.msgs = append(e.msgs, userMsg)        // 入历史
        e.record(ctx, userMsg)                // 落盘 JSONL
        e.step(ctx, out)                      // 进入 ReAct 循环
    }()
    return out, nil
}

变换：string → Message{Role:user} 追加进 []Message（现在长度=2）。

3

step() → ReAct 主循环

核心循环（engine.go）。每一轮：调 LLM → 看有没有工具调用 → 没有就结束，有就执行工具再循环。最多 maxSteps=16 步防失控。

for step := 0; step < e.maxSteps; step++ {
    reply, toolUses, err := e.streamAssistant(sctx, out) // ← 调 LLM
    e.appendAssistant(ctx, reply, toolUses)          // assistant 入历史
    if len(toolUses) == 0 {                          // 没有工具调用
        emitEvent(ctx, out, {Type: EventDone})         // → 本轮结束
        return
    }
    results := e.executeTools(sctx, toolUses, out)     // ← 执行工具
    e.appendResults(ctx, results)                     // tool 结果入历史
    e.maybeCompact(ctx, out)                          // 检查是否需压缩
}
4

streamAssistant() → 构建 Request 并流式调 LLM

把当前全部历史 e.msgs 和工具声明打包成 llm.Request，调 llm.Stream() 拿到 <-chan Delta，然后消费它：文本增量即时上抛 EventText，工具调用累积起来。

deltas, err := e.llm.Stream(ctx, llm.Request{
    Messages: e.msgs,            // ← 全部历史
    Tools:    e.toolSchemas(),  // ← 工具的 function calling 声明
    Model:    e.model,
})
for d := range deltas {
    if d.ToolCall != nil {
        toolUses = append(toolUses, *d.ToolCall)   // 累积工具调用
    }
    if d.Text != "" {
        sb.WriteString(d.Text)
        emitEvent(ctx, out, {Type: EventText, Text: d.Text}) // 即时上抛
    }
}
超时收紧：每次 LLM 调用施加独立超时（默认 120s），且取「单次超时」与「父 ctx 剩余墙钟」的较小值（effectiveLLMTimeout），防首 token 长挂起。
5

llm 层 → SSE 解析与 toolAcc 累积

在 llm/client.go 里，[]Message 先被 toOpenAIRequest() 转成 OpenAI 兼容请求，建立 SSE 流。pump() 循环读帧，processFrame() 逐帧解析：

  • 文本片段 → Delta{Text}
  • 工具调用分片（流式 function calling 是一个个碎片到达的）→ 按 Index 累积进 toolAcc；当 finish_reason=tool_calls 时 flush，组装成完整 ToolUseBlock（id + name + 拼接好的 arguments JSON）→ Delta{ToolCall}
  • 用量信息 → Delta{Usage}
为什么需要 toolAcc：OpenAI 协议下，一次工具调用的 arguments JSON 是分多个 SSE 帧流式传来的（如 {"pa、th":"a.go"}）。必须按 Index 把碎片拼起来才能得到完整入参。并行多工具调用时，不同 Index 对应不同工具，各自累积。
6

executeTools() → 工具分批编排

模型可能一次要调多个工具。executeTools（engine/exec.go）先用 orchestrate.PartitionBatches() 把它们切成并发批 + 串行批：连续的只读工具（如多个 read_file）合并并发执行，写/执行类工具各自独占串行。

batches := orchestrate.PartitionBatches(toolUses, e.tools)
return orchestrate.Run(ctx, batches, e.runOne, e.tracer)
fail-closed：工具未知或并发安全性不确定，一律按串行处理（最坏退化为全串行，绝不冒并发写竞态的险）。并发批用 errgroup + 索引槽位，无锁保序。
7

Guard.Call() → 权限闸门与 HITL 中断

每个工具执行前都先过 Guard（tool/guard.go）。它综合静态策略和工具内建判定，得出 allow / deny / ask 三态裁决：

switch dec.Behavior {
case BehaviorAllow: return g.inner.Call(...)            // 直接放行
case BehaviorDeny:  return {Content:"denied", IsError:true} // 回错误
default:           return g.askAndRun(...)              // ask → HITL
}

ask 时调 prompter.Ask() 打断，让人选择 Approve / Edit（改入参）/ Reject。无人值守模式下用 yesPrompter 自动批准。

fail-closed：如果 ask 了但没配 prompter，默认拒绝而非默认放行。安全优先。
8

工具实际执行 → 结果回流

放行后调用真实工具的 Call()（如 read_file 读文件、bash 经 sandbox 执行命令），返回 ToolResult。runOne 把它经 toolResultMessage() 规范化为 tool Message，关键是带上配对 ID：

return types.Message{
    Role:      types.RoleTool,
    Text:      content,          // ToolResult.Content
    ToolUseID: block.ID,         // ★ 与 ToolUseBlock.ID 配对
    ToolName:  block.Name,
}

同时往外吐 EventToolStart 和 EventToolResult 给 UI。

9

Session 持久化 → 上下文压缩检查

appendResults 把 tool 消息们追加进历史并逐条 record 落盘（append-only JSONL）。然后 maybeCompact 检查 token 是否触达阈值，必要时压缩历史（详见路径 5）。

10

循环判定 → 下一步 / 结束

回到 step 循环顶部，带着更长的历史（新增了 assistant + tool 消息）再调一次 LLM。如此往复，直到某轮模型不再要求工具调用 → 吐 EventDone。CLI 侧 runREPL 消费完整个 channel，渲染完毕，等待用户下一句输入。

一轮对话 = 多个 step。"fix the bug" 可能内部跑了：read_file → grep → edit_file → bash(test) → 报告，共 5 个 step，每个 step 一次 LLM 往返。用户看到的是连续的流式输出。

为什么 task 工具能调 SubAgent 却不形成循环依赖？

tool 包定义了窄接口 Spawner{ Spawn(ctx, prompt) (string, error) }。agent.SubAgent 的 Spawn 方法签名恰好满足它（隐式实现，无需 import）。cmd 层在装配时把 SubAgent 注入 task 工具。这样 tool 不依赖 agent，agent 依赖 engine，依赖图保持无环。

⑫ 分层记忆系统

Agent 重启后会"失忆"——上次学到的项目约定、踩过的坑全没了。internal/memory 用分层记忆解决：读侧把长期记忆注入系统提示，写侧让 Agent 运行中沉淀经验。

12.1 读侧（Loader）——记忆如何进入对话

还记得路径 1 第 1 步吗？engine.New() 构造时调用了 buildSystemPrompt()，记忆就是在这里被注入的：

func buildSystemPrompt(mem memory.Loader, workRoot string) string {
    base := systemPrompt                          // 基础系统提示
    base += Sprintf(workspaceHintTemplate, workRoot) // 工作根环境感知
    text, err := mem.Build(ctx, workRoot)        // 读 .cogent/MEMORY.md
    if err != nil { return base }              // 失败仅告警，不影响启动
    return base + "\n\n# Project memory (.cogent/MEMORY.md)\n" + text
}
记忆是"可选增强"，不是"正确性前提"：文件缺失返回空串 + nil error（os.ErrNotExist 被吞掉），加载失败只 slog.Warn，绝不阻断内核启动。这个设计原则贯穿全系统——增强功能失败，降级而非崩溃。

⑬ 会话持久化与 Resume

任务跑到一半被 Ctrl-C 了怎么办？cogent resume <session-id> 能从中断处接着干。这背后是"写时极简、读时修复"的设计哲学（DEV_SPEC §6.5）。

13.1 写路径——append-only 事件流

每条消息进历史时，record() 把它序列化成一个 session.Event 追加写入 JSONL 文件。事件靠 ParentUUID 串成链：

type Event struct {
    UUID       string          // 本事件唯一标识（crypto/rand 16字节hex）
    ParentUUID string          // 父事件 UUID → 串成链
    Type       string          // user/assistant/tool_result/undo/meta
    Payload    json.RawMessage // 序列化后的 Message（落盘前脱敏）
    Timestamp  int64           // Unix 纳秒
}
写时为什么极简？崩溃可能发生在任何时刻。如果写入逻辑复杂（比如要维护索引、更新状态），崩溃时容易写坏。append-only 纯追加是崩溃安全的——大不了最后一行写一半，读时跳过即可。复杂性全压到读路径。

14.3 第②层 · 路径越界与符号链接防护

ValidatePath 做两轮校验，第二轮专门堵 symlink 逃逸：

// 第一轮：Clean + 字符串前缀校验
abs = Clean(Join(root, target))
if abs != root && !HasPrefix(abs, root+sep) { return ErrPathEscape }
// 第二轮：解析符号链接后再校验真实落点（堵 symlink→区外）
realRoot   := evalSymlinks(root)
realTarget := evalSymlinks(abs)
if realTarget != realRoot && !HasPrefix(realTarget, realRoot+sep) { return ErrPathEscape }
为什么要解析 symlink？攻击者可能在工作区内建一个软链 link → /etc/passwd，第一轮校验看 link 在区内就放过了，但实际写的是区外文件。第二轮 EvalSymlinks 解析出真实落点 /etc/passwd，发现越界，拦截。连 root 本身也先解析（如 macOS /tmp → /private/tmp），避免前缀比对误判。

控制面写禁止

IsControlPlaneWrite 禁止写入 .cogent/ 和 .git/——防止"注入 → 持久化 → 长期控制"的攻击链（比如往 .cogent/MEMORY.md 塞恶意指令，让它每次启动都被注入）。

某些命令（如 git clone --bare .）会在工作根顶层撒下 HEAD/objects/refs/config... 一堆文件，污染工作区。清理逻辑极其保守：只有当 {HEAD,objects,refs} 三件套全是本次执行新建（执行前做 snapshotTopLevel 快照比对）才清理，且只删本次新增的，既有的 .git 和用户文件绝不触碰。这体现了"清理也要 fail-safe"——宁可不清理，也不能误删用户文件。

⑮ 可观测体系

Agent 是个黑盒——它为什么这么慢？花了多少钱？卡在哪一步？internal/observe 用 OpenTelemetry 把每条数据流变成可观测的 span 和指标，且默认零开销。

15.1 三个核心接口

type Tracer interface {  // 开 span，返回派生 ctx + 结束函数
    Start(ctx, name string, attrs ...Attr) (context.Context, EndFunc)
}
type Meter interface {   // 计量指标
    Count(name string, delta int64, attrs ...Attr)   // 计数器
    Record(name string, value float64, attrs ...Attr) // 直方图(耗时分布)
}
type Provider interface {  // 装配 Tracer/Meter + exporter
    Tracer() Tracer; Meter() Meter; Shutdown(ctx) error
}

内核里到处是这个惯用法：ctx, end := tracer.Start(ctx, "react.step"); defer end(err)。EndFunc 巧妙之处——结束时才知道的属性（token 数、耗时、结果）可以延迟补写：end(err, Attr{Key:"tokens", Value:n})。

15.2 零开销原则（no-op 模式）

关键设计：未开启可观测时（Enabled=false 或 Exporter=none），New() 返回 noopProvider——它的 Start 原样返回 ctx + 空函数，Count/Record 啥也不干。所以埋点代码永远在，但关闭时零运行时开销。生产可关、排查可开，代码无需改动。
func New(cfg Config) (Provider, error) {
    if !cfg.Enabled { return noopProvider{}, nil }       // 零开销
    if lower(cfg.Exporter) == "none" { return noopProvider{}, nil }
    return newOTelProvider(cfg)                          // file/stdout/otlp
}

路径 1 · 主交互流（核心）

用户在终端输入 cogent run 'fix the bug'，Cobra 解析到 run 子命令，调用 buildEngine() 装配 LLM 客户端、工具池、会话存储、上下文管理器。

Engine.Run() 把任务包装为 Message{Role: user} 追加进 e.msgs（此前已有一条 system 消息），落盘后启动 goroutine 跑 step() 循环。

每个 step：streamAssistant() 把全部历史打包成 llm.Request，调 Stream() 拿到 Delta 流；文本增量即时上抛 EventText，工具调用累积成 ToolUseBlock（llm 层靠 toolAcc 把流式分片按 Index 拼起来）。

若本轮无工具调用 → 吐 EventDone 结束；否则 executeTools() 经 PartitionBatches 切成并发/串行批，每个工具过 Guard 权限闸门（allow/ask/deny 三态，ask 触发 HITL），执行得 ToolResult，规范化为带 ToolUseID 配对的 tool Message 回流历史。

然后 maybeCompact() 检查是否需压缩，带着更长的历史进入下一个 step，如此往复直到完成。

路径 3 · 目标驱动循环

cogent goal 调 RunGoal，每轮跑一次 Engine.Run，然后用 Verifier 经 sandbox 执行 verify.sh；退出码 0 则达标结束，否则把 stderr 包成 feedbackPrompt 喂下一轮，直到达标或撞 三重预算（轮数/成本/墙钟）。

一句话总纲

cogent 的运行本质是 []Message → Request → Delta流 → 含 ToolUseBlock 的新 Message → ToolResult → tool Message → 回到 []Message 的循环，每步顺带外吐 StreamEvent 给 UI 渲染，由 maxSteps 兜底防失控。
