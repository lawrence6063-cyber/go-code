§1 全景概览：什么是 Loop Engineering

Loop Engineering（循环工程）是 2026 年 6 月兴起的 AI Agent 工程范式。 其核心思想是：你不再亲自提示 Agent，而是设计一个能自动提示 Agent 的系统—— 把「发现任务 → 启动 Agent → 验证结果 → 记录状态 → 进入下一轮」设计成持续运行的工程循环。

cogent 对 Loop Engineering 的实现：在 internal/loop/ 包中， cogent 把自身的单次 ReAct 执行引擎（engine.Engine）包装进一个 目标驱动的外层自治循环——定义目标与验收条件后，系统自动完成 「执行 → 判定 → 未达标带反馈续跑 → 再判定」直到目标达成或撞预算。

§3 核心数据结构

📄 internal/loop/loop.go

Goal — 目标定义

一次目标驱动循环的输入：自然语言意图 + 可验证的终止条件 + 预算护栏。

type Goal struct {
    Intent   string          // 自然语言目标，如「修复 X 并保证全部测试通过」
    WorkRoot string          // 工作根目录，传给 Verifier
    Verifier verify.Verifier // 独立终止条件判定器；nil 时 fail-closed
    Budget   Budget          // 预算护栏；零值时用保守默认
}

Budget — 三重失控护栏

任一触顶即终止循环，防止无限烧钱。

type Budget struct {
    MaxIterations int           // 外层循环最大轮数（默认 8）
    MaxCostUSD    float64       // 累计 LLM 成本上限（默认 $5）
    MaxWallClock  time.Duration // 端到端墙钟上限（默认 15m）
}

Outcome — 结局枚举

结局值含义
OutcomeAchieved0✅ 判定器确认目标达成（唯一成功结局）
OutcomeBudgetSpent1⏱️ 撞预算护栏（轮数 / 成本 / 墙钟）
OutcomeCanceled2🚫 ctx 被上游取消（如 Ctrl-C）
OutcomeFatal3💀 内层不可恢复错误

LoopEvent — 事件流

外层循环向 UI 单向上抛的统一事件（事件单向上抛不变量）。

事件类型含义携带数据
LoopIterationStart新一轮开始轮序号
LoopInner透传内层 engine 事件*types.StreamEvent
LoopVerify一次判定完成*verify.Report
LoopFinished循环结束*LoopResult

关键方法解析

loop() — 外层循环主体

func (o *orchestrator) loop(ctx context.Context, goal Goal, budget Budget, ...) LoopResult {
    deadline, cancel := budgetContext(ctx, budget) // 墙钟→派生ctx超时
    defer cancel()

    task := goal.Intent
    for iter := 0; iter < budget.MaxIterations; iter++ {
        if deadline.Err() != nil { return ... }   // 墙钟检查
        report, stopped := o.iterate(deadline, ...)  // 执行+判定
        if report.Passed { return OutcomeAchieved }   // 达标即停
        if o.overCost(budget) { return BudgetSpent }  // 成本护栏
        task = feedbackPrompt(goal.Intent, report)    // 带反馈续跑
    }
    return BudgetSpent // 撞轮数上限
}

feedbackPrompt() — 反馈续跑

把未达标的判定报告拼为下一轮 user task，注入原始目标 + 未通过的原因 + 修复建议， 让 Agent 在下一轮带着反馈继续迭代。

// 输出示例：
"The goal is not yet achieved. Keep working until verification passes.

Original goal: 修复所有测试用例
Verification feedback (not passed):
test TestFoo: expected 42, got 0

Details:
--- FAIL: TestFoo (0.00s)

Please fix the remaining issues and try again."

iterate() — 两种执行路径

路径 A：Pipeline（双角色）

当 Pipeline != nil 时走「制造-审查」双角色执行体。 Maker 改代码 → Reviewer 独立审查 → 客观 Verify 为最终硬闸门。

路径 B：Engine（单引擎）

当 Pipeline 为 nil 时回退到单 Engine 执行 → 外层独立调用 Verifier 判定。 这是最简模式，适合不需要双角色审查的场景。

§5 Trigger：触发源机制

核心组件

📄 internal/loop/trigger.go

Trigger 把「何时该干活」与「干什么」（Goal）解耦。 它返回一个事件 channel，每次有元素到达即表示应触发一次目标循环。

type Trigger interface {
    Fire(ctx context.Context) (<-chan TriggerSignal, error)
}

type TriggerSignal struct {
    Source  string // "cron" | "fswatch" | "manual" | ...
    Payload string // 触发上下文（如变更文件路径）
}

内置触发器

🕐 CronTrigger — 定时触发

按固定间隔触发（基于 time.Ticker，零第三方依赖）。

CronTrigger{Interval: 10 * time.Minute}

默认心跳间隔 10 分钟，可通过 --interval CLI 参数配置。

📁 FSWatchTrigger — 文件变更触发

监听 workRoot 下的文件变更（零依赖：周期轮询 fs.WalkDir 取最近修改时间 + 抖动合并）。

FSWatchTrigger{WorkRoot: "./src", Debounce: 2*time.Second}

自动跳过 .git、.cogent、data 等目录，避免自身写入触发死循环。

§6 Daemon：守护进程

核心组件

📄 internal/loop/daemon.go

Daemon 是 Loop 的最外层——它把 Trigger 与 Orchestrator 接起来： 每次触发跑一次 RunGoal，把结局写入 progress.md 看板，并尊重全局预算与 ctx 取消。 

type Daemon struct {
    Trigger Trigger        // 触发源
    Orch    Orchestrator   // 目标循环编排器
    Board   progress.Board // 跨 run 待办看板
    Tracer  observe.Tracer // loop.daemon span
}

核心运行逻辑

func (d *Daemon) Run(ctx context.Context, goalOf func(TriggerSignal) Goal) error {
    signals, err := d.Trigger.Fire(ctx)   // 启动触发源
    for {
        select {
        case <-ctx.Done():                    // Ctrl-C 优雅停
            return ctx.Err()
        case sig, ok := <-signals:             // 收到触发信号
            d.runOnce(ctx, goalOf(sig))        // 跑一轮目标循环
        }
    }
}

裁决解析 — fail-closed

Reviewer 被要求在最后一行输出 VERDICT: APPROVED 或 VERDICT: REJECTED。 解析逻辑全文扫描结构化裁决标记，取最后一个为准（兼容思维链型模型先推理后结论的输出模式）。 fail-closed：无明确 APPROVED 标记即判拒，防止自评虚高。

改动丢弃

审查未通过时，Discarder 接口负责回滚 maker 的改动（典型实现经沙箱跑 git 命令回滚工作区）。 配合 --worktree 选项，每轮在独立 git worktree 中操作，通过才 Merge 回基线，物理隔离主工作区。

§8 Verifier：独立判定器

安全设计 LOOP_SPEC §4.1

📄 internal/verify/verify.go

判定器与执行体（写代码的 Agent）完全解耦：执行体无法篡改判定结果 （独立判定不变量）。本包是依赖图的叶子，仅依赖 sandbox / 标准库。 

type Verifier interface {
    Verify(ctx context.Context, workRoot, goalIntent string) (Report, error)
}

type Report struct {
    Passed  bool   // fail-closed：判定异常一律视为 false
    Summary string // 人类可读结论；未通过时注入下一轮反馈
    Detail  string // 原始证据（脚本 stdout/stderr）
}

ScriptVerifier — 脚本判定器

经沙箱执行开发者提供的验收脚本：

  • 退出码 0 = 通过
  • stdout/stderr 收敛进 Report.Detail
  • 脚本是开发者提供的可信控制面，与不可信的模型输出隔离
  • 支持 NewSandbox 按 workRoot 构造受限沙箱（让验收脚本跑在 worktree 改动目录）
⚠️ 客观判据必须能被触达：自循环实验（§13）最大教训是 verify.sh 设计得再严谨， 只要它在 reviewer 闸门之后，就可能永远跑不到。cogent 已修复此问题： 客观 verify 成为不可被主观闸门短路的最终硬闸门—— 即使 reviewer 拒绝，只要 verify 通过就视为达标。

§11 安全护栏与不变量

安全设计

cogent 的 Loop 实现围绕以下系统不变量设计，确保自治循环安全可控：

🔒 独立判定不变量

执行体（写代码的 Agent）无法篡改判定结果。Verifier 与 Engine 在依赖图中完全解耦， 判定器是开发者提供的可信控制面。

💰 预算先行不变量

无 Budget 即用保守默认（16轮/$10/60分钟），绝不无限循环。 三重护栏（轮数 / 成本 / 墙钟）任一触顶即终止。

📤 事件单向上抛不变量

外层事件通过 chan LoopEvent 单向推送到 UI 层， UI 无法反向干预循环决策，保持执行的确定性。

🧪 客观判据为硬闸门

客观 verify（脚本退出码）为最终不可被短路的硬闸门。 即使 reviewer 主观拒绝，verify 通过仍视为达标——防止主观闸门屏蔽客观判据。

🔀 单向依赖

loop → engine 单向，engine 对 loop 零反向依赖。 Pipeline 接口在消费侧（loop）定义，agent 包经 cmd 层薄适配满足之。

🌿 Worktree 物理隔离

--worktree 模式下每轮在独立 git worktree 内操作， 通过才 Merge 回基线，物理隔离主工作区，防止未审查的改动污染代码。

§13 Dogfood 实验：自循环报告

实战验证

📄 docs/selfloop-report.md

cogent 用自身的 Loop 循环来优化自己的真实代码（dogfood）， 是 Loop Engineering「吃自己的狗粮」的最佳验证。以下是关键实验结论：

📋 实验配置

模型: deepseek-reasoner ｜隔离: --worktree ｜预算: 16轮 / $10 / 60分钟

目标: 补全 llm.finish_reason 端到端透出

六大关键发现

发现②（已修复）：reviewer 双闸门在 verify 之前拦截，客观判据从未运行。 worktree 流水线的「maker改→reviewer审→通过才Merge→再跑verify.sh」双闸门中， reviewer 3/3 拒绝 → verify.sh 一次都没执行。

修复：客观 verify 升级为最终硬闸门（VerifyAtFunc）， 执行体在落盘决策前必须先跑 verify，verify 通过即视为达标，reviewer 裁决降级为建议性反馈。
#发现状态改进措施
① 无人值守 loop 缺自动批准开关 ✅ 已修复 新增 COGENT_YES=1 环境变量门控的 yesPrompter
② 主观闸门短路客观判据 ✅ 已修复 客观 verify 升级为不可短路的硬闸门（VerifyAtFunc）
③ deepseek-reasoner 裁决格式不合规 ✅ 已修复 parseVerdict 升级为全文扫描 + 结构化裁决标记提取
④ reasoner 的 cwd 幻觉与高延迟 📝 记录 建议 maker 用 deepseek-chat、reviewer 用 strong 模型
⑤ $5 成本护栏未真正生效 📝 待做 需接入基于 token 指标 + 模型单价的 CostMeter
⑥ 墙钟取消有滞后 📝 记录 属可接受范围（受 120s LLM 超时上界约束）
核心结论：循环机制（worktree 隔离 / maker-reviewer / 预算护栏 / 反馈续跑 / 全链路 trace） 端到端跑通且可观测。暴露的真实问题（主观闸门短路客观判据、reviewer 模型选型、成本护栏未闭环） 比一次侥幸成功更有价值。
