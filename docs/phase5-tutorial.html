Phase 5 · Session Persistence & Resume

cogent Phase 5：让长任务「断了能接着干」

前四阶段的 cogent 能对话、能动手、能并发安全、能跑长任务，但任务一旦被 Ctrl-C 中断或进程崩溃，所有上下文就灰飞烟灭、只能从头再来。Phase 5 给它装上可中断可恢复能力：每一步交互以 append-only JSONL 事件流落盘，cogent resume <id> 时重放重建上下文、修复 function calling 配对、注入"从中断处继续"提示后交回同一主循环——无损接着干。

01为什么自主 Agent 必须「可恢复」

自主编码任务往往是长链路、多轮、耗时的：探索代码、改文件、跑测试、自我修正，可能要几十轮 LLM 调用。这条链路上有太多中断点——用户主动 Ctrl-C、网络抖动、进程被杀、机器重启。如果中断就意味着"从头再来"，那么"自主跑长任务"就失去了实用价值（既费钱又费时）。

没有持久化

上下文只活在内存的 e.msgs 里。进程一退，几十轮积累的探索成果、已做的改动决策全部蒸发，下次只能重跑——重复烧 token、重复踩坑。

cogent 的做法

每一步交互即时追加到 ./data/<id>.jsonl。中断后 cogent resume <id> 重放事件重建消息列表，注入"继续"提示，交回同一 ReAct 主循环续跑。

核心哲学：写入路径做简单，复杂性压到恢复路径

这是蓝本 Claude Code sessionStorage.ts 的关键工程取舍，cogent 照搬其精神：

路径特性设计取舍
写入（热路径）每轮都跑、要快要稳只做最简单的事：顺序 append 一行 JSON。无锁、无事务、append-only 天然崩溃安全（写到一半崩了，最多丢最后一行，已写的行完好）。
恢复（冷路径）只在 resume 时跑一次可以承担复杂度：去重、链路修复、剥离悬空 tool_use、丢弃孤立 tool_result。复杂性集中在这里，不污染热路径。
工程杠杆点："为什么写简单、恢复复杂？"——因为写在每轮的热路径上，必须快且崩溃安全；恢复是冷路径，只跑一次，可承担修复成本。append-only 是这套取舍的载体：单向追加既最快又最抗崩溃。

03事件模型与 JSONL 落盘

transcript（会话记录）是一条 append-only 的事件流，每行一个 JSON 事件。事件按 ParentUUID 串成链，按 UUID 去重——这让"同一事件被重复写入"（如重试）在恢复时被自动幂等处理。

internal/session/store.go · Eventtype Event struct {
    UUID       string          `json:"uuid"`        // 唯一标识，用于去重
    ParentUUID string          `json:"parent_uuid"` // 父事件 UUID，串成会话链
    Type       string          `json:"type"`        // user/assistant/tool_result/summary/meta
    Payload    json.RawMessage `json:"payload"`     // 序列化后的消息
    Timestamp  int64           `json:"ts"`          // Unix 纳秒
}
为什么 Payload 直接存整条 types.Message？恢复时只需 json.Unmarshal 回 Message，无需任何字段映射——序列化即"消息快照"，反序列化即重建。Type 由 Role 派生（RoleTool → "tool_result"），便于人读与按类型过滤。

Store 接口与 Append 落盘

internal/session/store.go · Store / Appendtype Store interface {
    Append(ctx context.Context, sessionID string, e Event) error
    Load(ctx context.Context, sessionID string) ([]Event, error)
    Resolve(sessionID string) string // transcript 文件路径
}

func (s *jsonlStore) Append(ctx context.Context, sessionID string, e Event) error {
    path, err := s.safePath(sessionID)        // 防 ../ 穿越
    // ... os.MkdirAll(dir, 0o700) ...
    e.Payload = redactSecrets(e.Payload)   // 落盘前脱敏
    line, _ := json.Marshal(e)
    f, _ := os.OpenFile(path, os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0o600)
    defer f.Close()
    _, err = f.Write(append(line, '\n'))  // 单行 JSON 追加，就是全部
    return err
}

Load 用 bufio.Scanner（放大 buffer 到 8MiB 容忍超长工具结果）单遍扫描，按 UUID 去重保序；坏行容错跳过（尽量恢复其余历史）；文件缺失返回 ErrSessionNotFound。

两道落盘安全：密钥脱敏 + 路径防护

T6 密钥脱敏（§7.5）

落盘前用正则把 sk-XXXX 令牌、"api_key"/"token"/"secret"/"password" 字段值替换为 [REDACTED]，杜绝密钥泄露进 transcript。

T1 路径穿越防护

safePath 用字符白名单 ^[A-Za-z0-9_-]+$ 校验 sessionID，拒绝 ../evil、a/b、空串等，再拼 <dataDir>/<id>.jsonl。

UUID / sessionID 生成（零依赖）// 不引入 uuid 库，用标准库 crypto/rand 生成 hex
func genUUID() string {
    var buf [16]byte
    if _, err := rand.Read(buf[:]); err != nil { /* 时间戳兜底 */ }
    return hex.EncodeToString(buf[:])
}
// 会话 ID 带时间前缀（可读 + 唯一）：20260617-171500-a1b2c3d4e5f6
func NewSessionID() string { return ts + "-" + hex }

04内核如何记录事件（最小侵入）

关键设计：Session 是可选依赖（Deps.Session 为 nil 时 record 直接跳过）。这保证了既有组件测试（注入 Fake LLM、不带 Session）和纯对话场景完全不受影响——向后兼容。

engine/session_event.go · recordfunc (e *engine) record(ctx context.Context, msg types.Message) {
    if e.session == nil || e.sessionID == "" {
        return // 未配置会话：纯对话 / 单测，零侵入
    }
    payload, _ := json.Marshal(msg)
    ev := session.Event{
        UUID: genUUID(), ParentUUID: e.lastUUID, // 链到上一条
        Type: eventType(msg.Role), Payload: payload, Timestamp: time.Now().UnixNano(),
    }
    if err := e.session.Append(ctx, e.sessionID, ev); err != nil {
        slog.Warn("append session event", "err", err)
        return // 落盘失败仅告警，不中断主循环
    }
    e.lastUUID = ev.UUID // 刷新链尾
}

记录点嵌在三处（step 主循环里），覆盖一轮交互的全部消息：

1
Run 追加用户任务

进入 step 前 append(user) + record(user)。

2
appendAssistant 记录助手回复

含 tool_calls 的 assistant 消息追加后立即 record，保证调用与回复一同落盘。

3
appendResults 逐条记录工具结果

本批 tool_result 保请求序逐条 record，与 tool_use 配对完整。

落盘失败不致命：持久化是增强而非内核正确性前提。Append 出错只 slog.Warn，主循环继续跑——磁盘满了也不该让正在进行的任务崩掉，最多失去 resume 能力。

05Resume：重建上下文并续跑

守住单一真相源不变量（§4.4）：Resume 不分叉主循环，只在 bootstrap 阶段做"重放重建"，随后交回与 Run 共用的 step。这正是把 loop 拆成"不追加 task 的 step"的意义。

engine/engine.go · Resumefunc (e *engine) Resume(ctx context.Context, sessionID string) (<-chan types.StreamEvent, error) {
    if e.session == nil { return nil, errors.New("resume requires a configured session store") }
    events, err := e.session.Load(ctx, sessionID)   // 读 transcript
    if err != nil { return nil, fmt.Errorf("load session: %w", err) }

    e.sessionID = sessionID
    rebuilt, lastUUID := rebuildMessages(events)        // 事件 → 消息
    rebuilt = filterUnresolvedToolUses(rebuilt)        // ★ 修复配对
    e.msgs = append(e.msgs[:1:1], rebuilt...)        // 保留 system，接上历史
    e.lastUUID = lastUUID                            // 续上事件链，不重复记录历史

    out := make(chan types.StreamEvent, 16)
    go func() {
        defer close(out)
        cont := types.Message{Role: types.RoleUser, Text: continuePrompt}
        e.msgs = append(e.msgs, cont)
        e.record(ctx, cont)                            // 仅记录新的 continue，不重放旧事件
        e.step(ctx, out)                              // 交回同一主循环
    }()
    return out, nil
}
两个易错点都已规避：① e.msgs[:1:1] 保留内核重新注入的 system 提示（含最新 MEMORY.md），再接重建历史——system 不入 transcript，始终用最新的。② Resume 不重新 record 重建的历史（它们已在文件里），只把 lastUUID 续到链尾、仅记录新的 continue 消息——否则会产生重复事件。

★ filterUnresolvedToolUses：恢复路径的核心

中断最危险的后果是留下"半截"的工具调用：模型已请求 tool_use 但结果还没回流，或反过来有孤立的 tool_result。直接把这种历史交给 LLM，toOpenAIMessage 会生成"孤立 tool_result"，DeepSeek API 直接报错。所以重建后必须修复 function calling 配对不变量——这与 Phase 4 的 compact 切点修复同等重要。

中断现场（坏）user:      "重构 X 模块"
assistant: tool_use(c1=read) tool_use(c2=edit)
tool:      c1 → "文件内容"
           ✗ c2 的结果还没回来，进程就被 Ctrl-C 了
           ────────── 中断 ──────────
直接 resume → c2 是悬空 tool_use → API 报错
engine/session_event.go · 三步修复func filterUnresolvedToolUses(msgs []types.Message) []types.Message {
    // ① 收集所有"有结果"的 ToolUseID
    resolved := collectResolvedIDs(msgs)
    kept := make(map[string]bool)
    out := make([]types.Message, 0, len(msgs))
    for _, m := range msgs {
        if m.Role == types.RoleAssistant && len(m.ToolCalls) > 0 {
            // ② 只保留有结果的调用；记下被保留的 ID
            m.ToolCalls = keepResolvedCalls(m.ToolCalls, resolved, kept)
            if m.Text == "" && len(m.ToolCalls) == 0 { continue } // 空壳丢弃
        }
        // ③ 丢弃指向已剥离调用的孤立 tool_result
        if m.Role == types.RoleTool && !kept[m.ToolUseID] { continue }
        out = append(out, m)
    }
    return out
}
修复后保证：每个保留下来的 assistant tool_call 都有配对的 tool_result，且没有任何孤立 tool_result。交回 llm.toOpenAIMessage 时协议完整，API 不报错。续跑时模型看到的是一个干净、自洽的历史。

07Session Transcript 与 Trace 正交（§8.5）

容易混淆的两条流，目的与约束完全不同，用 session_id 关联：

Session Transcript（本阶段）Trace（Phase 8）
目的业务恢复：可中断可 resume可观测：性能/成本/调用链分析
内容消息事件（user/assistant/tool_result）span 树（耗时/token/父子/错误）
不变量强 function calling 配对完整弱 尽力而为，丢 span 不影响正确性
落盘data/<id>.jsonl（append-only）data/traces/<id>.jsonl 或 OTLP
排障姿势：定位坏 case 时，先用 trace 看"哪轮哪工具慢/错"，再用 transcript 看"那轮模型说了什么、给了什么 tool_result"——一物一职，互为补充。

09设计要点回顾

① 写简单恢复复杂

热路径只 append 一行 JSON（崩溃安全），把去重/配对修复压到冷的恢复路径。

② append-only JSONL

单向追加、按 UUID 去重、ParentUUID 串链，人可读、可审计、可增量。

③ 单一真相源

Resume 不分叉主循环，仅 bootstrap 重建，交回与 Run 共用的 step。

④ 配对不变量

重建必经 filterUnresolvedToolUses，剥离悬空 tool_use、丢孤立 result，否则 API 报错。

⑤ 落盘安全

密钥脱敏 + 路径白名单 + 0o600/0o700；密钥绝不进 transcript。

⑥ 零新依赖

UUID 用 crypto/rand 自生成，持久化仅靠标准库 os/encoding/json。

STAR（可中断可恢复）

维度要点
S 情境自主编码任务长链路多轮，中途 Ctrl-C / 崩溃就得从头再来，重复烧钱踩坑。
T 任务让长任务可中断、可无损恢复。
A 行动"写入简单、恢复复杂"——会话以 append-only JSONL 事件流落盘；signal.NotifyContext 把 Ctrl-C 变成 ctx 取消、全链路优雅收尾；resume 时重放事件、按 UUID 去重、filterUnresolvedToolUses 修复配对、注入"继续"提示再交回同一主循环。
R 结果中断后 resume 无损接着干；事件流还一物多用——既是恢复依据，又是可观测/回放数据源（与 trace 用 session_id 关联）。
