把 Agent 变成会自我审查、能自己醒来的团队

上一阶段（L0+L1）让 cogent 学会「达目标才停」。这一阶段更进一步：L2 把执行体拆成「写代码的 maker」+「独立审查的 reviewer」，实现「不批改自己的作业、通过才落盘」；L3 给循环装上「心跳」——可被定时或文件变更唤醒，跨多次运行用 progress.md 记住「还剩什么没做」。从「你点一下跑一次」迈向「无人值守的自治团队」。

0这一步在做什么

Loop Engineering 的「六原语」里，这一阶段补齐两块关键拼图：

L2 · Sub-agents 制造-审查分离

写代码的不能给自己的代码打分。引入两个独立上下文的子 Agent：maker 负责改、reviewer 负责审，审查通过才真正落盘。

L3 · Automations + State 心跳 + 看板

让循环能自己「醒来」（定时 / 文件变更触发），并把跨多次运行的「已做/待做/阻塞」写进磁盘 progress.md，仓库不忘记。

延续铁律：叠加而非重写。engine / tool / sandbox 一行没改——L2 复用 v1 的「隔离子 Engine」模式扩展 agent，L3 新增 progress 叶子包 + loop 的 Trigger/Daemon。

1为什么必须双角色：别让学生自己改卷

如果让同一个 Agent 既写代码又自评，它有强烈的「确认偏置」——刚写完的代码，它会本能地觉得「挺好的」。这等于让学生改自己的卷子。

reviewer 用独立子 Engine + 独立上下文（甚至可指不同模型），没看过 maker 的「内心戏」，只看实际代码——审查才客观。
成本逻辑：maker 可用便宜快的模型（迭代次数多），reviewer 用更强的模型（质量闸门）——通过 COGENT_REVIEWER_MODEL 指定，把钱花在刀刃上。

2maker / reviewer 流水线：通过才落盘

一轮双角色流水线分四步，核心是「通过才落盘」——reviewer 不通过就丢弃 maker 的本轮改动，带着审查意见重做。

reviewer 输出首行 APPROVED / REJECTED: 理由，parseVerdict 解析——fail-closed：解析不出明确 APPROVED 一律视为未通过。
internal/agent/maker_reviewer.gofunc (m *MakerReviewer) Iterate(ctx context.Context, task string) (PipelineResult, error) {
	makerSummary, _ := m.runRole(ctx, RoleMaker, m.maker, task)            // ① maker 改（可写池）
	reviewOut, _   := m.runRole(ctx, RoleReviewer, m.reviewer,
	                       reviewPrompt(task, makerSummary))               // ② reviewer 审（只读池）
	verdict := parseVerdict(reviewOut)                                 // ③ fail-closed 解析裁决
	if !verdict.Approved {
		m.discard(ctx)                                            // ④ 未通过 → 回滚改动
	}
	return PipelineResult{MakerSummary: makerSummary, Verdict: verdict}, nil
}
一个工程约束：engine 的 system prompt 是固定常量，无法给 reviewer 注入「你是审查者」的系统提示。解法：把评审 rubric 放进任务 prompt，reviewer 以只读档位读代码后据此输出裁决——不动 engine 一行。

3双闸门：主观审查 + 客观验收

L2 的 reviewer 接进 L1 的目标循环后，形成两道闸门：reviewer 的「主观代码审查」+ Verifier 的「客观测试验收」，两道都过才算达标。

reviewer 拒 → 直接判未达标、带审查意见续跑；reviewer 通过 → 再跑验收脚本（go test 退出码）做客观确认。
为什么这样设计？主观审查能抓「客观测试测不到」的问题（命名、可维护性、潜在坑），客观验收能抓「reviewer 看走眼」的回归。两者互补，比单一判据更可靠。

4L3 · 心跳 Trigger：让循环自己醒来

到目前为止，循环还是「你敲一次 cogent goal 才跑」。L3 加上 Trigger（触发源）——把「何时干活」与「干什么」解耦，让循环能被定时或事件唤醒。

⏰ CronTrigger（定时）

基于 time.Ticker，每隔 --interval（如 10m）醒一次。零第三方依赖。适合「每 10 分钟检查有没有失败测试」。

👀 FSWatchTrigger（文件变更）

零依赖轮询扫描目录最近修改时间 + 抖动合并；检测到改动就触发。适合「CI 一红/文件一变就修」。跳过 .git/.cogent/data 防自触发。

internal/loop/trigger.go// Trigger 把「何时该干活」与「干什么」（Goal）解耦。
type Trigger interface {
	// Fire 返回事件 channel；每个元素=触发一次目标循环。ctx 取消即关闭 channel（无泄漏）。
	Fire(ctx context.Context) (<-chan TriggerSignal, error)
}

5Daemon + progress.md：守护进程与跨 run 看板

Daemon 把 Trigger 和目标循环接起来：串行消费触发信号（一次只跑一个 RunGoal，防并发膨胀），每轮结局写进 progress.md 看板。

守护进程 O(触发次数) 串行执行 + 预算护栏兜底，绝不并发膨胀、不失控烧钱；Ctrl-C 经 signal.NotifyContext 全链路收手。

三层状态各司其职（别搞混）

session（v1）memory（v1）progress（L3 新增）
粒度单任务消息事件项目长期约定跨 run 的待办进度
生命周期一次任务（可 resume）长期沉淀一条自治循环全程
读者engine 重建上下文注入系统提示Daemon 决定「下一步干什么」
语义重放事实/约定当前还剩什么
progress.md 是控制面：它落在写禁止区 .cogent/，只许经 progress.Board 这条受控通道写，模型工具不能直接改——防「注入 → 篡改进度 → 长期跑偏」。看板是人可读、可手改、可 diff 的 Markdown 表格。

6怎么跑起来

terminal# 双角色模式：maker 改 + reviewer 独立审，通过才落盘
cogent goal "给 user 服务加限流中间件" --review \
  --verify=./verify_ratelimit.sh --max-iterations=8

# 守护进程：每 10 分钟醒一次，自动跑目标循环，进度写 progress.md
cogent loop --interval=10m --goal-file=.cogent/goal.md \
  --verify=./verify.sh --review --max-iterations=8

# 或：盯着目录文件变更触发（CI 一红就修）
cogent loop --watch=. --verify=./verify.sh

# Ctrl-C 随时优雅停，进度已写进 .cogent/progress.md

守护进程跑完每轮后，.cogent/progress.md 大致长这样（人可读、可手改）：

.cogent/progress.md# cogent progress

| ID | Status | Title | Note | Updated |

7收尾：这一步把 cogent 推到了哪

① 不批改自己的作业

maker / reviewer 独立上下文分离，通过才落盘，双闸门把关。

② 会自己醒来

Cron / FSWatch 心跳 + Daemon 守护，从命令式迈向持续自治。

③ 仓库不忘记

progress.md 跨 run 看板，重启后知道「还剩什么没做」。

六原语已落地五项（Connectors / State / Sub-agents / Automations / goal），cogent 正式成为一个会自我审查、能无人值守的自治循环系统。配合 L0+L1 的「达目标才停」，它已是一台完整的「高度 3」Loop 引擎。

  • Skills（SKILL.md）：把「怎么给本项目加限流中间件」这类流程固化为可复用技能包。
