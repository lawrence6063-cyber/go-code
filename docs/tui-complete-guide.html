cogent 的 终端交互层：一个包，十四项能力

cogent 的 TUI 已聚合到单一 internal/tui 包。它逐字节接管键盘，实现 @ 文件补全下拉、/ 斜杠命令、命令历史与反向搜索、单选菜单、常驻状态栏与成本计量、权限 HITL 交互，以及类 Claude Code 的流式渲染（工具调用折叠成一行摘要、正文与工具视觉分区）。本文逐一拆解当前 TUI 的全部功能及其实现原理——SVG 展示终端观感，Mermaid 描绘时序。

1Cooked vs Raw 模式与 termios

默认的 Cooked（规范）模式下，内核终端行规程替你回显、处理退格、把 Ctrl-C 翻成信号，直到回车才把整行交给程序——做不了「边打字边弹框」。要实时交互，必须切到 Raw 模式，让程序自己收字节、自己画界面。

图 1 · 行规程「代劳」的 Cooked 模式 vs 程序自管的 Raw 模式。

enterRaw 用 golang.org/x/sys/unix 读改 termios：关闭 ECHO/ICANON/ISIG/IEXTEN、关 ICRNL/IXON/OPOST，并设 VMIN=0 / VTIME=1（0.1s 轮询超时），为「可取消阻塞读」铺路；返回恢复闭包由 defer 保证退出复位。请求号按平台分文件：darwin 用 TIOCGETA/SETA，linux 用 TCGETS/SETS，其它平台 editor_stub.go 一律返回非终端 → 走逐行回退。

editor_raw.go（darwin||linux）raw.Lflag &^= unix.ECHO | unix.ECHONL | unix.ICANON | unix.ISIG | unix.IEXTEN
raw.Iflag &^= unix.ICRNL | unix.IXON | unix.BRKINT | ...
raw.Oflag &^= unix.OPOST
raw.Cc[unix.VMIN] = 0; raw.Cc[unix.VTIME] = 1   // 无数据 0.1s 返回，支持轮询取消
关 ISIG 的关键含义：Raw 期 Ctrl-C 不再产生 SIGINT，而是以字节 0x03 抵达，由编辑器自行决定语义（清行 / 双击退出）——见 §8。

2按键解码状态机 & 可取消阻塞读

Raw 下拿到的是原始字节流。decodeKey 把它翻译成语义 keyEvent：单字节控制符直接映射，ESC 引导的转义序列（方向键 ESC[A/B/C/D、Home/End、ESC[n~）分支解析，>=0x80 的多字节按 UTF-8 首字节长度续读。

flowchart TD B["readByte()"] --> C{首字节?} C -->|"0x1b ESC"| E["decodeEscape：读 '[' / 'O'
→ 方向键/Home/End；孤立 ESC=keyEsc"] C -->|"\r \n"| K1["keyEnter"] C -->|"0x03 / 0x04"| K2["keyCtrlC / keyCtrlD"] C -->|"0x12 / 0x15 / 0x10 / 0x0e"| K3["Ctrl-R / Ctrl-U / ↑(Ctrl-P) / ↓(Ctrl-N)"] C -->|"< 0x80 可打印"| K4["runeEvent(b)"] C -->|">= 0x80"| U["decodeUTF8：按首字节续读 2~4 字节"]

可取消阻塞读：rawInput 内部缓冲 + 一字节回退（unread，供 ESC 歧义判定归还）。readByte 循环里每次先查 ctx.Err() 再 fill()；因 VTIME 超时返回 0 字节时 Go 会附带 io.EOF——这是伪 EOF，fill 特判忽略（只有真实错误才冒泡），从而实现「阻塞读也能被 ctx 取消」。

editor_keys.go · fill()m, err := r.f.Read(r.buf[:])
if m > 0 { r.n, r.i = m, 0; return true, nil }
if err == io.EOF || err == nil { return false, nil } // VTIME 超时，非真 EOF
return false, err

4@ 文件补全：候选来源与模糊排序

5/ 斜杠命令补全

行首输入 / 触发命令补全（优先级高于 @）。命令来自单一注册表 builtinCommands——它同时是补全下拉与 /help 的唯一事实来源。ParseSlashToken 仅在「行以 / 开头且光标在首个空格之前」时激活；光标进入参数区即停止补全命令名。

6命令历史与 reverse-i-search

history.Store 内存维护有序条目（最早在前），并落盘到 .cogent/history（0600 权限、上限 1000 条、忽略空行与紧邻重复）。↑/↓ 浏览历史（首次上翻前暂存当前编辑行，翻到底回到暂存行）；Ctrl-R 进入 reverse-i-search 子模式，边打边从末尾向前增量匹配，再按 Ctrl-R 跳更早命中。

(reverse-i-search)`test': go test ./...

反向搜索由 SearchBackward(query, from) 支撑：从自末尾第 from 条起向更早找首个包含 query 的条目，返回命中行与索引。Enter 采纳命中行、Esc/Ctrl-G 取消并恢复原行。

7下拉框渲染：相对光标移动

下拉画在输入行下方。早期用 DECSC/DECRC（保存/恢复绝对光标）在终端底部滚动时会失效导致残留。现改为相对光标移动：lineEditor 记录上次渲染的下拉行数 drawnDropdownLines，每次重绘先 ESC[nA 上移回输入行、\r + 跳过提示符宽度、ESC[0J 清到屏末，再从行首重绘输入 + 下拉（每行加 ESC[2K 防残留），最后把光标按显示宽度放回。可见窗口 maxVisibleSuggest=5，选中项超窗自动滚动。

editor_render.go · drawLineif le.drawnDropdownLines > 0 { fmt.Fprintf(&b, "\x1b[%dA", le.drawnDropdownLines) } // 上移回输入行
b.WriteString("\r"); fmt.Fprintf(&b, "\x1b[%dC", le.promptWidth)          // 跳过 "you> "
b.WriteString("\x1b[0J")                                             // 清到屏末，保留提示符
提示符感知：重绘跳过 promptWidth（"you> "=4 列）再清屏，不会擦掉提示符前缀。

8双击 Ctrl-C 退出、Ctrl-D 与单选菜单

空行双击 Ctrl-C 才退出

参考 readline/bash：有文本或下拉激活时 Ctrl-C 只清行 + 关下拉；空行首次 Ctrl-C 进入待确认态（下方显示灰色 (press Ctrl-C again to exit)），ctrlCTimeout=800ms 内二次才真正 statusInterrupt。任何非 Ctrl-C 按键惰性清除待确认态。空行 Ctrl-D 返回 EOF。

9状态栏与成本计量

每次输入提示前打印一行常驻状态栏：当前模型、累计输入/输出 token、估算美元成本。

[deepseek-v4-pro | tok in:12040 out:3110 | $0.0342] you>

数据来自 costMeter——它装饰 observe.Meter：拦截 cogent.tokens 计数，按模型单价表（可用 COGENT_PRICE_<MODEL>_{INPUT,OUTPUT} 环境变量覆盖）累计美元，其余指标原样转发；同时实现 loop.CostMeter(SpentUSD)，驱动 goal/loop 的 --max-cost 护栏。CostProvider 装饰 observe.Provider 使 Meter() 返回它，装配阶段 prov = NewCostProvider(prov) 织入整条链。

10流式渲染：从「一股脑」到「分区折叠」

这是本轮改造的重点。重构前，工具的几百行结果、入参 JSON、子 Agent 事件顺序刷屏；改造后每次工具调用只占两行——● 工具名 参数 + ⎿ 状态，正文与工具块留白分区。

图 10 · 同一轮回复：左「一股脑」vs 右「分区折叠」。

inTool 状态机

streamRenderer 维护 inTool 标记「工具已开始、结果未到」区间：EventToolStart flush 正文 + 空行分区 + 打印 ● Summarize(name,input) 并置位；区间内 EventText（进度/子 Agent 文本）抑制不刷屏；EventToolResult 打印单行 ⎿ 状态 并复位。首个可见输出前有 thinking… spinner；正文经 StreamMarkdown 逐行渲染（标题加粗、围栏变暗、行内 code 着色）。

stream.gofunc (r *streamRenderer) writeText(text string) {
    if text == "" { return }
    if r.inTool { return }              // ← 抑制工具区间内的进度/子 Agent 刷屏
    r.beginOutput(); fmt.Fprint(r.w, r.md.Write(text))
}

11权限 HITL（human-in-the-loop）

写/执行类工具（write_file/edit_file/bash + 全部 MCP 外部工具）经 Guard 触发权限中断，由 cliPrompter 读 stdin 决策：

[permission] write_file internal/x.go approve / always / edit / reject? [a/A/e/r]
输入动作
a / approve / y本次批准
A / always本会话内该工具自动批准（存入 allowlist，不再询问）
e / edit输入修正后 JSON 入参（非法 JSON 降级为拒绝）
其他 / reject拒绝并可附一句给模型的改道指引

提示复用 Summarize——不再打印完整入参 JSON（降低噪音与泄露面）。无人值守场景（COGENT_YES=1）用 yesPrompter 一律批准，危险命令仍由 sandbox 确定性拦截兜底。cliPrompter 的 allowlist 有互斥锁保护，支持并发 Ask（-race 验证）。

12REPL 循环与 per-turn SIGINT

repl.go 的 inputLoop 是多轮对话的心脏：打印状态栏 + you>、读一行、路由（exit/斜杠命令/普通输入）、流式渲染本轮回复。它独占管理 SIGINT：先 signal.Reset(SIGINT) 解除 main.go 顶层 NotifyContext 的监听（防止其 cancel 顶层 ctx 导致整体退出），再用本地 channel 接管——模型执行时 SIGINT 仅 cancel 当前 turn（打印 [interrupted] 后 REPL 继续），等待输入时由 raw 模式双击 Ctrl-C 决定退出。

13非 TTY 回退与外层循环渲染

NewTTYInputReader 先判 IsTerminal：是终端才启用 raw 行编辑器；否则（管道/脚本/测试）用后台 bufio.Scanner 逐行读取。即便进了终端却发现 raw 不可用（受限 pty），readLine 返回三态 usable=false，next 会一次性懒回退到逐行读取，而非让 REPL 秒退。这保证脚本与交互场景行为一致。

loopview.go 的 RunLoopView 服务 goal/loop：透传内层 engine 的文本/工具事件（复用同一套分区折叠渲染），单独呈现轮次标题、[verify] 判定与终局归因；内层错误只记 stderr、不打断外层带反馈续跑。

三态返回的意义：readLine 区分「用户主动退出（ok=false, usable=true）」与「环境不支持 raw（usable=false）」，避免把「不支持 raw」误判成 EOF 而秒退——这是此前 cogent run 一秒退出 bug 的根因修复。
