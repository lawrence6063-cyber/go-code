从「回车才知道你输了啥」到「边打字边弹下拉框」

普通的 bufio.Scanner 只能在你按下回车后一次拿到整行——它对「你正在输入什么」一无所知。要实现输入 @ 就实时弹出文件下拉框、↑↓ 导航、Tab 选中，就必须让程序逐字节接管键盘，自己画界面、自己管光标。这正是终端 Raw 模式 的用武之地。本文拆解 cogent 这套 @ 补全从终端底层到状态机的完整实现原理，并回答两个延伸问题：这算不算 TUI？一个 AI CLI 还差哪些关键能力？

0一句话顿悟：程序要「抢」键盘

你在终端敲字，默认情况下有个「隐形的中间人」在替你处理：内核里的终端行规程（line discipline）。你打的每个字符先进它的缓冲区，它负责回显到屏幕、处理退格、把 Ctrl-C 翻译成信号……直到你按回车，它才把一整行交给程序。程序全程蒙在鼓里。

这套「贴心代劳」叫 Cooked 模式（规范模式），写脚本很省事，但做不了实时交互——因为在回车之前，程序根本收不到任何字符，谈何「输入 @ 立即弹框」。

要实现实时下拉框，只有一条路：让程序告诉内核「别管了，键盘我自己收」——切到 Raw 模式，然后自己承担起原本中间人做的全部脏活：逐字节读取、自己回显、自己处理退格与方向键、自己画下拉框、自己恢复终端。cogent 的 lineeditor.go 就是这个「接管者」。

1Cooked vs Raw：内核到底替你做了什么

「模式」本质是内核终端驱动上的一堆开关（termios flags）。Cooked 模式下这些开关默认开着，Raw 模式就是把关键的几个关掉。理解这张表，就理解了 Raw 模式的全部：

行为Cooked（默认）Raw（我们要的）关掉的 flag
回显：按键自动显示在屏上内核负责回显关闭，由程序自己决定怎么画ECHO
行缓冲：攒够一行才交付攒到回车才给程序关闭，逐字节立即交付ICANON
Ctrl-C / Ctrl-Z 转信号自动发 SIGINT/SIGTSTP关闭，Ctrl-C 变成普通字节 0x03，程序自己判断ISIG
回车/换行转换（CR↔LF）自动转换关闭，避免污染我们对回车的判断ICRNL/OPOST
Ctrl-S/Ctrl-Q 软件流控暂停/恢复输出关闭，避免误吞按键IXON
直觉记忆：Cooked 模式是「内核当你的秘书，帮你整理好一整段话再转达」；Raw 模式是「内核当你的话筒，你说一个字它传一个字」。做实时 UI 必须用话筒。

但 Raw 模式有个「昂贵」的代价：你关掉的每一样，都得自己补回来。关了回显？那得自己把字符打到屏上。关了行缓冲？那退格、光标移动全得自己算。关了 ISIG？那 Ctrl-C 得自己识别并优雅退出。cogent 后面几节做的就是这些「补课」。

2进出 Raw：termios 与「恢复现场」的纪律

切换模式靠 ioctl 系统调用读写终端的 termios 结构体。cogent 在 lineeditor_raw.go 里用 golang.org/x/sys/unix 封装了 enterRaw：

cmd/cogent/lineeditor_raw.go// enterRaw 将终端置为原始模式，返回用于恢复原状态的闭包。
func enterRaw(fd uintptr) (func() error, error) {
    old, err := unix.IoctlGetTermios(int(fd), termiosGetReq) // ① 先存原始设置
    if err != nil { return nil, err }
    raw := *old                                          // ② 拷贝一份来改
    raw.Iflag &^= unix.IGNBRK | unix.BRKINT | unix.ISTRIP | unix.INLCR |
        unix.IGNCR | unix.ICRNL | unix.IXON              // 输入：关 CR↔LF 转换、软件流控
    raw.Oflag &^= unix.OPOST                             // 输出：关自动后处理（\n 不再自动变 \r\n）
    raw.Lflag &^= unix.ECHO | unix.ICANON | unix.ISIG | unix.IEXTEN
                                                         // 本地：关回显 / 行缓冲 / 信号 / 扩展
    raw.Cflag |= unix.CS8                                // 8 位字符，UTF-8 才不被截断
    raw.Cc[unix.VMIN] = 0                           // ③ 关键：见下文「可取消读」
    raw.Cc[unix.VTIME] = 1
    if err := unix.IoctlSetTermios(int(fd), termiosSetReq, &raw); err != nil {
        return nil, err
    }
    fdInt := int(fd)
    return func() error {                              // ④ 返回恢复闭包
        return unix.IoctlSetTermios(fdInt, termiosSetReq, old)
    }, nil
}
最容易踩的坑：忘了恢复现场。如果程序崩溃或忘记还原 termios，你的终端会「坏掉」——打字不回显、回车不换行，只能盲敲 reset 抢救。所以 enterRaw 的设计强制返回一个 restore 闭包，调用方必须 defer 它。 
cmd/cogent/lineeditor.go · readLinerestore, err := enterRaw(le.in.Fd())
if err != nil { return "", false }   // 非终端/无权限 → 优雅退回
defer func() { _ = restore() }()          // 无论正常返回、panic、中断都恢复

为什么用 defer 而不是手动调用？

Go 的 defer 保证函数任意路径退出（正常 return、panic、提前 return）都会执行恢复。这是「进入 Raw 一定能出来」的纪律保证，也是终端类代码最重要的安全网。

为什么只在「读一行」期间进 Raw？

cogent 只在等待/编辑输入时进 Raw，读到整行立刻 restore 回 Cooked。模型流式输出、日志打印阶段始终是 Cooked——这样 Ctrl-C 能正常走 SIGINT 中断长任务，作用域最小、风险最低。

2.1　平台差异：build tag 分文件

termios 的 ioctl 请求号在 darwin 和 linux 上不同（TIOCGETA vs TCGETS），非 unix 平台压根没有。cogent 用 Go 的 build tag 拆成多个文件，让「平台细节」与「主逻辑」彻底解耦：

文件build 约束职责
lineeditor_raw.godarwin || linuxenterRaw / isTerminalFD 主逻辑
lineeditor_ioctl_darwin.godarwintermios 请求号常量（macOS）
lineeditor_ioctl_linux.golinuxtermios 请求号常量（Linux）
lineeditor_stub.go!darwin && !linux桩：isTerminalFD 恒 false，自动退回非 TTY 路径

3ANSI 转义序列：程序如何「画」下拉框

关了回显后，屏幕上的一切都得程序自己写。终端不认识「把光标移到第 3 列」这种指令，只认识一串转义序列（escape sequence）——以 ESC（0x1b，写作 \x1b）开头的特殊字节串。程序把它们混在普通文本里输出，终端负责解释执行。cogent 用到的几条：

序列助记作用
\x1b7DECSC保存当前光标位置（记住「输入起点」）
\x1b8DECRC恢复到上次保存的光标位置
\x1b[0JED清除从光标到屏幕末尾（擦掉旧的下拉框）
\x1b[nCCUF光标右移 n 列（定位到编辑光标）
\x1b[7m … \x1b[0mSGR反显（高亮选中项）… 复位样式

3.1　核心套路：保存 → 重绘 → 复位

每按一次键，界面都要整体重绘一次。为了不闪、不错位，cogent 用了一个经典套路：打印提示符后先 DECSC 记住「输入起点」，之后每次重绘都回到这个起点、擦干净、重新画行和下拉框、再把光标精确移回编辑位置。

cmd/cogent/lineeditor.go · renderfunc render(w io.Writer, core *editorCore) {
    var b strings.Builder
    b.WriteString("\x1b8\x1b[0J")          // ① 复位到输入起点 + 擦掉旧内容
    b.WriteString(string(core.line))       // ② 画当前整行
    if core.active && len(core.sugg) > 0 {
        renderDropdown(&b, core)              // ③ 行下方画下拉框
    }
    b.WriteString("\x1b8")                  // ④ 再次复位到输入起点
    if core.cursor > 0 {
        fmt.Fprintf(&b, "\x1b[%dC", core.cursor) // ⑤ 右移到真实光标列
    }
    io.WriteString(w, b.String())          // ⑥ 一次性写出，减少闪烁
}
为什么用 strings.Builder 攒完再一次写？终端渲染最忌讳「写一点刷一点」——多次 write 会让用户看到中间态导致闪烁。把一帧的所有转义序列拼成一个字符串原子性输出，视觉上就是「瞬间刷新」。这跟游戏里的双缓冲是同一个道理。

3.2　下拉框长什么样

renderDropdown 在当前行下方逐条打印候选，用 \x1b[7m（反显）高亮选中项，并只渲染「可见窗口」内的条目（maxVisibleSuggest = 8，通过 offset 滚动）：

you> 帮我看下 @inte
> internal/tool/grep.go
  internal/tool/findfiles.go
  internal/completion/token.go
  internal/engine/snapshot.go

↑ @inte 触发，第一项反显为当前选中；↑↓ 移动高亮，Tab/Enter 把 @inte 替换为 @internal/tool/grep.go。

4按键解码：把字节流翻译成「按键事件」

Raw 模式下我们拿到的是裸字节。一次按键可能是 1 个字节（a）、也可能是控制字符（Ctrl-C = 0x03）、还可能是一串转义序列（↑ = ESC [ A 三字节），中文更是多字节 UTF-8。decodeKey 就是这台「翻译机」，把字节流规整成统一的 keyEvent：

cmd/cogent/lineeditor.go · decodeKey（节选）switch {
case b == 0x1b:                   return decodeEscape(ctx, src)  // ESC 引导 → 方向键等
case b == '\r' || b == '\n':  return ev(keyEnter), nil
case b == '\t':                 return ev(keyTab), nil
case b == 0x7f || b == 0x08:   return ev(keyBackspace), nil
case b == 0x03:                 return ev(keyCtrlC), nil   // Ctrl-C（因为关了 ISIG）
case b == 0x04:                 return ev(keyCtrlD), nil
case b == 0x10:                 return ev(keyUp), nil     // Ctrl-P，emacs 习惯
case b == 0x0e:                 return ev(keyDown), nil   // Ctrl-N
case b < 0x20:                  return ev(keyUnknown), nil // 其他控制字符忽略
case b < 0x80:                  return runeEvent(rune(b)), nil // ASCII 可打印
default:                       return decodeUTF8(ctx, src, b)  // 多字节 rune（中文）
}

4.1　方向键的「歧义」：孤立 ESC vs 转义序列

方向键 ↑ 发送 ESC [ A，但用户单独按 Esc 只发一个 ESC。程序读到 ESC 后必须判断：后面还有没有字节？decodeEscape 用一次非阻塞轮询（readByteNow）来窥探：

  • 轮询立刻拿到 [ 或 O → 是转义序列，继续读末字母（A=↑ / B=↓ / C=→ / D=←）。
  • 轮询拿不到任何字节 → 判定为孤立 Esc，返回 keyEsc（用来关闭下拉框）。
  • 拿到的不是 [/O → 用 unread 把这个字节「塞回去」，本次当孤立 Esc 处理。
这里的 unread（一字节回退）是解析器里很实用的小技巧：多读了一个不属于自己的字节，就还回缓冲区，交给下一轮解码。rawInput 用一个 ungot int 字段实现了它。

4.2　为什么要拆出 editorCore？——为了可测

真实终端没法在单测里跑。cogent 把状态机（editorCore）与终端 I/O（lineEditor）彻底分离：editorCore 是纯逻辑——喂进一串 keyEvent，就能断言行缓冲、光标、下拉状态如何变化，完全不碰终端。

纯逻辑单测示意core := newEditorCore(ctx, provider)
core.handleKey(runeEvent('@'))   // 输入 @
core.handleKey(runeEvent('a'))   // 输入 a → 触发过滤
core.handleKey(ev(keyDown))       // ↓ 选下一项
status := core.handleKey(ev(keyEnter)) // Enter → accept，写回 @路径
// 断言 core.line / core.cursor / status，无需真实 TTY
设计要点：把「难测的副作用（终端）」推到最外层薄薄一圈，把「核心逻辑」做成纯函数/纯状态机。这是所有 CLI/TUI 可测试性的通用法门。

5@ 补全的三层实现

@ 补全的逻辑分三层，各司其职：Token 解析（我在补全什么？）、候选来源（有哪些文件？）、状态机编排（怎么随按键变化）。前两层放在纯逻辑包 internal/completion，与终端零耦合。

图 5-1　@ 补全的三层职责

5.1　第一层：Token 解析（我在补全什么）

每次按键后，需要知道「光标前那段 @ 片段是什么」。ParseAtToken 从光标向左回溯，遇到 @ 就成功、遇到空白就放弃（说明不在补全上下文）：

internal/completion/token.gofunc ParseAtToken(line []rune, cursor int) AtToken {
    // ...夹取 cursor 到 [0, len]...
    for i := cursor - 1; i >= 0; i-- {
        r := line[i]
        if r == atTrigger {   // 找到 @ → 上下文成立
            return AtToken{Start: i, Partial: string(line[i+1 : cursor]), Active: true}
        }
        if unicode.IsSpace(r) {  // 先遇空白 → 不在补全中
            return AtToken{Active: false}
        }
    }
    return AtToken{Active: false}
}

选中时用 ApplyChoice 把 @partial 整段替换成 @完整相对路径，并把光标移到路径末尾。保留 @ 前缀是刻意设计——让后续处理仍能识别「这是一个文件引用」。

5.2　第二层：候选来源（有哪些文件）

Provider 负责提供候选文件清单，策略对齐 Claude Code：

  1. git 优先：git ls-files（已跟踪）+ git ls-files --others --exclude-standard（未跟踪但没被 gitignore）。天然尊重 .gitignore，不会把 node_modules 塞进来。
  2. 非 git 回退：filepath.WalkDir 遍历，手动跳过 .git / .cogent / node_modules，带 2 万条上限防止巨型仓库卡死。
  3. TTL 缓存：清单缓存 5 秒（cacheTTL）。这点至关重要——每次击键都 spawn 一个 git 进程或遍历磁盘会卡到没法用，缓存后每次击键只在内存里过滤。
性能红线：补全是「每个字符都要响应」的高频路径。所以清单只加载一次并缓存，过滤纯在内存做。这是把「昂贵的 I/O」和「高频的过滤」解耦的典型手法。

模糊排序：不是简单 contains

rankMatches 给每个候选打一个匹配等级，越小越靠前，做到「越符合直觉的越靠上」：

等级含义例（查询 grep）
0 前缀整条路径以 query 开头grep.go
1 段前缀文件名（basename）以 query 开头internal/tool/grep.go
2 子串路径任意位置包含 queryx/mygrep_test.go
3 子序列字符按序出现（不连续）g...r...e...p

同等级内再按路径长度、最后按字典序稳定排序——短路径通常是你更想要的那个。

5.3　第三层：状态机编排（随按键联动）

editorCore 把前两层串起来。关键是每次编辑动作后调用 refresh 重算候选：

cmd/cogent/lineeditor.go · refreshfunc (e *editorCore) refresh() {
    tok := completion.ParseAtToken(e.line, e.cursor)
    if !tok.Active {          // 光标已不在 @ 上下文
        e.closeDropdown()      // → 关闭下拉框
        return
    }
    e.sugg = e.provider.Filter(e.ctx, tok.Partial, maxSuggest) // 重新过滤
    e.active = len(e.sugg) > 0
    // ...修正选中索引、调整可见窗口...
}

于是完整链路是：按键 → handleKey 改行/光标 → refresh 重算候选 → render 重绘，每一帧都自洽。Tab/Enter 命中候选时走 accept（调 ApplyChoice 写回并关框），否则 Enter 提交整行。

6细节彩蛋：如何让「阻塞读」变得可取消

有个隐蔽难题：readLine 阻塞等键盘时，如果上层 ctx 被取消（比如收到 SIGINT），怎么让它及时醒来？普通阻塞 Read 会一直卡着，永远看不到 ctx。

cogent 的巧解是 VMIN=0, VTIME=1：这让终端的 Read 变成「最多等 100ms，没数据就返回 0 字节」的轮询式读。于是 readByte 可以在每次超时后检查 ctx.Err()：

cmd/cogent/lineeditor.go · rawInput.readBytefor {
    if r.i < r.n { /* 缓冲区还有字节，直接返回 */ }
    if err := ctx.Err(); err != nil { return 0, err }  // ★ 每轮检查取消
    if _, err := r.fill(); err != nil { return 0, err } // VTIME 超时返回 0 字节，循环继续
}
本质：用「短超时轮询」把一个不可中断的系统调用，变成一个「每 100ms 有机会响应取消」的可协作循环。代价是极小的空转，收益是 Ctrl-C 和 ctx 取消都能及时生效。

7这些算 TUI 吗？——概念澄清

简短回答：是 TUI 技术，但只是「行级 TUI」，还不是「全屏 TUI」。这里有个常被混淆的谱系：

CLI（命令行）

一问一答、读整行、顺序输出。bufio.Scanner 那套。无光标控制、无实时反馈。

git commit

早期 cogent REPL

行级 TUI（本文所在）

Raw 模式 + ANSI，在当前行及其附近做实时交互：行编辑、补全下拉、高亮。不接管整屏。

readline / fzf 输入行

cogent @ 补全

全屏 TUI

接管整个屏幕（alternate screen），有布局、面板、窗口、鼠标事件，自建事件循环。

vim / htop

lazygit / k9s

cogent 的 @ 补全用到的底层技术（Raw 模式、termios、ANSI 转义、按键解码、自绘光标）与全屏 TUI 完全同源——Go 生态里 bubbletea、Rust 的 ratatui、C 的 ncurses 内部干的就是这些。区别只在「作用域」：

维度cogent（行级）全屏 TUI 框架
屏幕范围提示符行 + 下方几行下拉整屏 alternate screen buffer
Raw 模式✅ 只在读一行期间✅ 全程
ANSI 转义✅ 少量（DECSC/清屏/反显）✅ 大量（定位/颜色/边框）
事件循环读一行的 for 循环常驻主循环 + 消息队列
布局系统无有（弹性盒/网格/窗口）
所以准确说法是：cogent 自己手写了一个轻量的行级 TUI 组件（相当于 readline + 补全菜单），它是 TUI 技术的应用，但没有升级成「全屏 TUI 应用」。对一个 AI CLI 来说，这个粒度恰到好处——既有现代交互体验，又不必背上全屏框架的复杂度与渲染负担。

8顺带一说：AI CLI 还有哪些关键需求

@ 文件补全只是「输入体验」这一块。一个成熟的 AI 编码 CLI（如 Claude Code、cogent）通常还需要以下能力，简要过一遍：

① 流式输出（Streaming）

模型 token 边生成边打印，而不是等整段完成。需要处理增量渲染、光标控制、以及「流式中途 Ctrl-C 打断」。是「感觉快」的关键。

② 工具调用 + 人在环（HITL）

模型请求读文件/改文件/跑命令时，危险操作要弹出权限确认（允许/拒绝/编辑）。cogent 的 cliPrompter 就干这个——@ 补全也顺带覆盖了这些输入点。

③ 上下文 / 会话管理

多轮对话历史、token 预算控制、超长时压缩（compact）、会话持久化与 resume 续跑。上下文是 AI CLI 的「内存管理」。

④ 斜杠命令 & 快捷输入

/help、/clear、/compact 等命令，以及 @文件、# 标签等快捷引用。补全菜单是它们共同的载体。

⑤ 权限与安全边界

命令白/黑名单、路径沙箱、避免任意命令执行（RCE）、密钥只走环境变量不落盘。AI 会「自作主张」，边界必须硬。

⑥ 成本与可观测

token 用量/费用实时展示、请求日志、trace。让用户对「烧了多少钱、模型在干嘛」心里有数。

⑦ 错误恢复 & 中断安全

网络抖动重试、工具失败回喂模型自愈、Ctrl-C 干净退出并恢复终端（正是本文 Raw 模式恢复纪律的意义）。

⑧ 可扩展集成（MCP / 插件）

通过 MCP 等协议接入外部工具、知识库、数据源，让 CLI 的能力可插拔地增长。

一句话归纳这 8 点：输入要顺手（补全/命令）、输出要即时（流式）、行为要可控（HITL/权限）、上下文要经济（压缩/持久化）、过程要透明（成本/日志）、异常要稳健（重试/中断恢复）、能力要可长（MCP）。@ 补全解决的是第一条「输入顺手」里的一小块。
