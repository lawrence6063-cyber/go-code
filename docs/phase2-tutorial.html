01架构全景

Phase 2 的依赖方向严格单向，engine 只依赖 tool.Pool 抽象，权限与沙箱被「装配」进需要它们的工具里，而不是侵入内核。这是整个设计的骨架：

依赖方向          ┌─────────────┐
          │     CLI     │  装配一切：建工具池、注入 Prompter/Policy/Tracer、解析 --mode
          └──────┬──────┘
                 │ 注入 Deps{ LLM, Tools, Mode, ... }
                 ▼
          ┌─────────────┐      ┌──────────────┐
          │   engine    │─────▶│  tool.Pool   │  仅依赖抽象，不认识具体工具
          │ (ReAct loop)│      └──────┬───────┘
          └──────┬──────┘             │
                 │ Stream()           ├── readfile / listdir / grep  (只读, 直接入池)
                 ▼                    └── Guard( writefile/editfile/bash )
          ┌─────────────┐                      │ 包裹
          │   llm 层    │                      ▼
          │ function    │              ┌──────────────┐   ┌─────────────┐
          │ calling     │              │  permission  │   │   sandbox   │
          └─────────────┘              │  (三态裁决)   │   │ (路径/危险) │
                                       └──────────────┘   └─────────────┘
关键取舍：本阶段工具执行先做串行，bash 做最小安全版（ask + 路径校验 + 危险命令拦截 + 超时）。并发分批调度与完整命令沙箱留给 Phase 3。

什么是 function calling 链路？

「function calling」是让 LLM 在回答时不直接输出文本，而是输出「我想调用某个工具 + 参数」的结构化请求。完整一轮闭环是：

1
声明工具

请求里带上每个工具的 name / description / 入参 JSON Schema，告诉模型「你有这些能力」。

2
模型发起调用

模型流式返回 tool_calls（分片到达），cogent 累积成完整的 ToolUseBlock{ID, Name, Input}。

3
本地执行工具

engine 查池、过权限、跑工具，拿到结果。

4
回流 tool_result

把结果按 ID 配对成 role=tool 消息塞回历史，再次请求模型。模型据此继续或收尾。

不变量（最易踩的坑）：assistant 的 tool_calls 与 role=tool 的 tool_result 必须按 ID 一一配对完整。少一个、多一个或顺序错乱，OpenAI 兼容接口都会直接报 400。

02LLM 层：把流式分片拼回完整工具调用

难点在于：OpenAI 流式协议里，一次 tool_calls 的 arguments（JSON 字符串）是跨多个 SSE 帧分片到达的——首帧带 id/name，后续帧只带 arguments 的一截。我们需要一个按 index 累积的「拼接器」。

累积器 toolAcc

internal/llm/client.go// toolAcc 按 index 累积一次流式响应里的多个并行工具调用，保序刷出。
type toolAcc struct {
    bufs  map[int]*toolCallBuf // index → 累积缓冲
    order []int                // 首次出现顺序，保证刷出与模型请求一致
}

// add 把一帧的 tool_calls 分片并入累积器；Index 标识所属调用。
func (a *toolAcc) add(calls []openai.ToolCall) {
    for _, c := range calls {
        idx := 0
        if c.Index != nil { idx = *c.Index }
        b, ok := a.bufs[idx]
        if !ok {                          // 首次见到这个 index
            b = &toolCallBuf{}
            a.bufs[idx] = b
            a.order = append(a.order, idx)
        }
        if c.ID != ""            { b.id = c.ID }            // 首片带 id
        if c.Function.Name != "" { b.name = c.Function.Name } // 首片带 name
        b.args.WriteString(c.Function.Arguments)              // 后续片拼 arguments
    }
}

何时「刷出」一个完整调用？

当某帧的 finish_reason == tool_calls 时，说明工具调用已集齐，把缓冲组装成 ToolUseBlock 上抛；流结束（EOF）时再兜底刷一次，防止漏掉。

internal/llm/client.go · processFrameacc.add(choice.Delta.ToolCalls)
if choice.FinishReason == openai.FinishReasonToolCalls && !acc.flush(ctx, out) {
    return false
}
细节兜底：flush 里若 arguments 为空字符串，会规整为合法空对象 "{}"——这样无参工具调用（如 list_dir 列当前目录）也能产出可被 json.Unmarshal 解析的入参。

请求侧：声明工具 + 配对消息

请求映射要做两件事：① 把工具 schema 转成 OpenAI 的 tools 字段；② 让 assistant 消息携带 tool_calls、tool 消息携带 ToolCallID，保证配对完整。

internal/llm/client.go · toOpenAIMessagefunc toOpenAIMessage(m types.Message) openai.ChatCompletionMessage {
    msg := openai.ChatCompletionMessage{Role: mapRole(m.Role), Content: m.Text}
    if m.Role == types.RoleTool {        // tool_result 必须带 ID 才能配对
        msg.ToolCallID = m.ToolUseID
        msg.Name = m.ToolName
    }
    if len(m.ToolCalls) > 0 {            // assistant 携带它发起的 tool_calls
        msg.ToolCalls = toOpenAIToolCalls(m.ToolCalls)
    }
    return msg
}

03tool 层：统一协议、工具池与 fail-closed 默认

所有工具——内建的、未来 MCP 接入的外部工具——都实现同一个 Tool 接口、进同一个 Pool。新增能力 = 实现接口 + 注册，内核零改动。

internal/tool/tool.gotype Tool interface {
    Name() string                          // 唯一标识（function calling 名）
    Description() string                   // 给模型看的用途描述
    InputSchema() json.RawMessage          // 入参 JSON Schema

    IsConcurrencySafe(input json.RawMessage) bool // 默认 false
    IsReadOnly(input json.RawMessage) bool        // 默认 false

    CheckPermission(ctx, input) (permission.Decision, error)
    Call(ctx, input, sink ProgressSink) (types.ToolResult, error)
}

fail-closed：把「安全」从约定变成语言层兜底

这是本层最精妙的设计。Defaults 提供一组最保守的默认实现：非并发安全、非只读、权限 ask。工具匿名嵌入它，只覆写自己需要「放开」的方法。忘记覆写 = 自动退化为最安全行为，而不是最危险行为。

// 保守默认
type Defaults struct{}
func (Defaults) IsReadOnly(...) bool {
    return false   // 默认不是只读
}
func (Defaults) CheckPermission(...) {
    return ask     // 默认要问
}
// 只读工具只覆写需要放开的
type ReadFile struct {
    Defaults  // 匿名嵌入
    ...
}
func (ReadFile) IsReadOnly(...) bool {
    return true   // 显式放开
}
工具池 Pool 的并发模型：启动期一次性装配（NewPool），运行期只读——切片 + map 索引，无锁、无竞态。同名工具保留先出现者（内建优先于 MCP）。

04permission：权限三态

权限模型只有三个状态，简单但够用：

allow

直接放行执行。只读工具走这条路。

ask

中断，征询人类（HITL）：Approve / Edit / Reject。

deny

拒绝，把原因规整为错误结果回流给模型改道。

裁决由 StaticPolicy（静态策略，Deny 优先）与工具内建 CheckPermission 共同决定：策略命中 allow/deny 即采纳，否则回落到工具自身判定。

Guard：把权限 + HITL 收敛到一处

不在每个工具里散落权限逻辑，而是用一个装饰器 Guard 包裹写/执行类工具。Guard 自身也实现 Tool（元数据全透传内层），只在 Call 时插入权限闸门：

internal/tool/guard.go · Callfunc (g *Guard) Call(ctx, input, sink) (types.ToolResult, error) {
    dec := g.decide(ctx, input)   // 在 permission.check span 内裁决
    switch dec.Behavior {
    case permission.BehaviorAllow:
        return g.inner.Call(ctx, pickInput(input, dec.UpdatedInput), sink)
    case permission.BehaviorDeny:
        return types.ToolResult{Content: denyMessage(dec.Reason), IsError: true}, nil
    default: // ask → 走 HITL
        return g.askAndRun(ctx, input, dec, sink)
    }
}
fail-closed 再兜底：若 ask 但没有配置 Prompter（比如非交互环境），Guard 直接返回错误结果而非放行。安全永远是默认值。

05sandbox：路径与危险命令边界

所有文件操作和命令执行都先过 sandbox 校验，三道防线：

A
ValidatePath — 防越界

把目标路径解析为绝对路径后，校验它仍位于 workRoot 之内。../../etc/passwd 这类穿越会被 ErrPathEscape 拒绝。

B
控制面写禁止

.cogent/、.git/ 等控制面目录的写入被禁止——防止 Agent 改坏自己的配置或仓库元数据。

C
危险命令识别

bash 命令按复合分隔符逐子命令检查破坏性片段（rm -rf /、mkfs、dd if=…），并单独整命令匹配 fork bomb :(){:|:&};: 与管道执行 curl ... | sh。

设计原则：宁可误拦，不可漏放。危险命令用小写子串匹配，复合命令任一子命令危险即整体判危。这是保守的「最小安全版」，Phase 3 会换成真正的命令沙箱。

06engine：工具化 ReAct 主循环

这是 Phase 2 的「心脏」。Phase 1 的 loop 在没有工具调用时就结束；Phase 2 改为：有工具调用 → 串行执行 → 回流 tool_result → 进入下一轮，maxSteps 作为防失控空转的护栏。

1
streamAssistant

流式调 LLM，文本即时上抛为 EventText，同时累积模型发起的 toolUses。

2
appendAssistant

把助手回复（含 tool_calls）落入消息历史——tool_calls 必须和 assistant 消息一起入历史，否则下一轮配对断裂。

3
无工具 → 结束

len(toolUses)==0 时发 EventDone，本次任务收尾。

4
executeTools（串行）

逐个执行，每个工具发成对的 ToolStart/ToolResult 事件，结果规整为 role=tool 消息按 ID 配对回流。

internal/engine/exec.go · 核心不变量// 工具失败规范化为 tool_result(IsError) 让模型自我修正，不中断主循环
func toolResultMessage(block types.ToolUseBlock, res types.ToolResult) types.Message {
    return types.Message{
        Role:      types.RoleTool,
        Text:      content,
        ToolUseID: block.ID,   // ← 与 assistant 的 tool_call.ID 配对
        ToolName:  block.Name,
    }
}
容错哲学：工具报错不抛异常中断循环，而是把错误写进 tool_result 回给模型，让模型「看到报错后自己改方法重试」。这正是 Agent 自我修正能力的来源。

07CLI：人在环交互与运行档位

三种运行档位 --mode

档位暴露的工具典型用途
auto（默认）全部工具，可写放手让 Agent 自主完成改造
plan仅只读工具探索仓库、产出改造计划，不落盘
ask仅只读工具只读问答

档位过滤在两处生效：toolsForMode 决定声明给模型哪些工具；toolAllowedInMode 在执行前二次兜底，即便模型硬调了写工具也会被挡回——再次体现 fail-closed。

HITL 中断交互

写/执行类工具触发 ask 时，cliPrompter 在终端暂停征询：

交互示例[permission] tool "write_file" requests execution:
  input: {"path":"main.go","content":"..."}
  approve / edit / reject? [a/e/r]
  • a (Approve)：原样批准执行。
  • e (Edit)：输入修正后的 JSON 入参，再执行修正版。
  • r (Reject)：拒绝，可附一段指引文本回流给模型，让它改道。
并发读 stdin 的坑：REPL 主循环和 HITL 都要读标准输入。这里用单一共享 inputReader，避免两个 goroutine 同时读 stdin 产生竞争——go test -race 能发现这类问题。

设计要点回顾

① fail-closed 贯穿全程

Defaults 保守默认 + 嵌入覆写、ask 无 Prompter 即拒、档位执行前二次兜底——「忘了配置」永远退化到最安全行为。

② 配对完整是硬约束

tool_calls 与 tool_result 按 ID 一一配对，assistant 消息必须携带它发起的 tool_calls。

③ 装饰器收敛横切关注点

Guard 把权限 + HITL + 埋点收敛到一处，工具本身只管「做事」，不掺权限逻辑。

④ 错误即上下文

工具失败不中断循环，而是回流 IsError 结果让模型自我修正——这是 Agent 韧性的核心。

⑤ 依赖单向、内核纯净

engine 只认 tool.Pool 抽象；permission/sandbox 随工具装配注入，不污染内核 Deps。

⑥ 启动装配、运行只读

Pool 一次装配、运行期无锁只读，从结构上消除并发竞态。
