Phase 8 · Observability with OpenTelemetry

cogent Phase 8：把 ReAct 决策链变成可视化调用树

Agent 任务是长链路、多轮、并发的——一个任务跑了 20 轮、烧了 5 万 token、最后失败，黑盒化后你无法回答"慢在哪、钱花在哪、错在哪"。Phase 8 用 OpenTelemetry SDK 把整条决策链建成一棵 span 树（cogent.session → react.step → llm.stream / tool.call / agent.spawn …），落本地文件或一键推到 Jaeger 看火焰图；并复用 context.Context 让 span 自动父子串联——与取消传播共用同一条管线。这是把后端最硬的"分布式链路追踪"直接迁移到 Agent 内核。

01为什么 trace 对 Agent 尤其关键

维度没有 Trace有 Trace
性能定位只知道"任务花了 90s"80s 耗在第 7 轮某个 bash，12s 在 LLM 首字延迟
成本归因只知道"总共 5 万 token"哪一轮把上下文撑大、compact 是否生效
行为审计只知道"失败了"回放完整决策链：第几轮选了哪个工具、为什么
并发调试并发批某工具卡死难复现span 树清晰展示并发批内每个工具的起止与父子关系
核心洞察：ReAct 循环天然就是一棵 span 树，与 OpenTelemetry 的 trace 模型同构。把它建成 OTel span 树，是顺水推舟，而非生搬硬套。

02为何这次改用 OTel SDK（一次有意的反向取舍）★

Phase 6（MCP）与 Phase 7（SubAgent）我们都坚持"手写核心、零新增依赖"。但 Phase 8 我们有意打破这条原则，引入 go.opentelemetry.io/otel 全家桶。为什么？因为取舍的天平这次倒向了另一边：

若自实现 trace

能保住零依赖，但要自己定义 span 数据结构、序列化格式、采样、批量导出——而且接不进 Jaeger/Tempo 生态，火焰图得自己写。可视化价值大打折扣。

引入 OTel SDK ✅

业界事实标准；file/stdout/otlp exporter 现成；一行 env 切到 Jaeger 看火焰图；"上来就是 OTel"比自造结构更有说服力。代价：主 go.mod 多一批传递依赖。

取舍逻辑（可讲）：MCP 那次自实现，是因为"亲手写协议"能讲清原理、且生态收益有限；Trace 这次用 SDK，是因为可观测的价值恰恰在于接入标准生态（Jaeger 火焰图）——自造一套谁都不认识的格式，反而丢掉了 trace 最大的价值。同一个工程师，两次相反的决定，背后是同一套"按收益权衡"的判断框架，而不是"凡事都手写"的教条。这本身就是成熟度的体现。

03span 树模型与字段约定

一次任务（session）对应一棵 trace，span 层级如下——这些埋点 Phase 0~7 早已写好（以 no-op 形式），Phase 8 只是给它们换上真 exporter：

ReAct span 树cogent.session                       # 根 span：一次任务（trace_id 贯穿全程）
└── react.step (step=0..N)
    ├── llm.stream                   # model / tokens / 首字延迟 / finish_reason
    ├── tool.batch (concurrent=bool)
    │   └── tool.call (name=read_file)   # tool.name / is_error / duration
    │       ├── permission.check        # allow/ask/deny
    │       └── agent.spawn             # SubAgent 子树（Phase 7）
    │           └── react.step ...       #   子 Agent 自己的循环，经 ctx 自动挂接
    └── ctx.compact (triggered=bool)   # 上下文压缩

所有 span 出错时统一 RecordError(err) + SetStatus(codes.Error)，便于在可视化里红色高亮失败节点。敏感字段（task 原文、文件内容、命令明文）落 span 前先脱敏（§7.5）。

04context 的双重职责：取消 + trace 共用一条管线 ★

这是 Phase 8 最值钱的认知点，也是 Go 做 Agent 的天然优势。Tracer.Start 返回一个派生 ctx（绑入新 span）；把它继续往下传，子 span 自动认这个父亲——无需手动传 trace_id。而这个 ctx 同时还是取消信号的载体。

internal/observe/otel.gofunc (t *otelTracer) Start(ctx context.Context, name string, attrs ...Attr) (context.Context, EndFunc) {
    ctx, span := t.tracer.Start(ctx, name)   // 派生 ctx 携带新 span
    if len(attrs) > 0 { span.SetAttributes(toKeyValues(attrs)...) }
    return ctx, func(err error) {
        if err != nil {
            span.RecordError(err)
            span.SetStatus(codes.Error, err.Error())
        }
        span.End()
    }
}
父子自动串联

派生 ctx 沿 engine→exec→tool→sandbox 一路传，每层 Start 自己的子 span，OTel 自动认父。

取消即终止

Ctrl-C 取消同一个 ctx，进行中的 span 也带 cancelled 状态，链路与生命周期一致。

并发不串味

并发批每个 goroutine 各从父 ctx 派生独立子 span，OTel span 上下文值传递、协程安全。

05换内胆不换接口

cogent 从 Phase 0 起就把可观测抽象成不暴露任何 OTel 类型的薄接口。Phase 8 的全部改动收敛在 observe 包内——内核各层（engine/exec/tool/orchestrate/agent/mcp）一个 import 都不用改。

internal/observe —— 导出接口签名一字未改type Tracer interface { Start(ctx, name, ...Attr) (context.Context, EndFunc) }
type Meter  interface { Count(name, delta int64, ...Attr); Record(name, value float64, ...Attr) }
type Provider interface { Tracer() Tracer; Meter() Meter; Shutdown(ctx) error }

// New：关闭 → 零开销 no-op；开启 → OTel SDK 支撑的真实现
func New(cfg Config) (Provider, error) {
    if !cfg.Enabled || cfg.Exporter == "none" { return noopProvider{}, nil }
    return newOTelProvider(cfg)
}
解耦红利：OTel 类型被严格封在 observe 包内（otel.go/exporter.go）。这意味着——① 内核可单测（注入 no-op）；② 未来想换 trace 后端只改一个包；③ Enabled=false 时 Start 调用点恒定存在但零开销、无 if 分支污染业务代码。这正是 §5.11 的设计意图落地。

exporter 三态 + 接 Jaeger

exporter.go 按 Config.Exporter 装配底层 OTel exporter；TracerProvider 用 BatchSpanProcessor（异步批量、热路径低开销），Shutdown 时强制 flush 保证最后一批不丢。

06slog 对齐 trace_id

光有 trace 还不够——排查时常要在"看 span"和"看日志"之间跳。我们用一个 slog.Handler 包装器，让每条日志自动带上当前 span 的 trace_id/span_id，两边就能互相对照。

internal/observe/slog.gofunc (h traceHandler) Handle(ctx context.Context, r slog.Record) error {
    if sc := trace.SpanContextFromContext(ctx); sc.IsValid() {
        r.AddAttrs(
            slog.String("trace_id", sc.TraceID().String()),
            slog.String("span_id", sc.SpanID().String()),
        )
    }
    return h.inner.Handle(ctx, r)
}
易踩的坑：WithAttrs/WithGroup 必须重新包一层 traceHandler（而不是直接返回 inner 的结果），否则 slog.With(...) 之后注入能力就丢了——这是包装 slog.Handler 的经典陷阱。

08测试策略

可观测代码也要测，重点是验证"span 父子关系、错误状态、属性"真的被正确导出。用 file exporter 写临时目录 → 读回 JSON 断言，零外部依赖。

no-op 降级

Enabled=false / none → 类型断言为 noopProvider；调用零开销不 panic。未知 exporter 返回 error。

file exporter 父子 span 往返 ★

起 parent→child，child 标错误；Shutdown 后用 json.Decoder 串联解码 jsonl，断言 trace_id 相同、child.parent=parent.span、Status=Error、attr k=v。

Attr 转换

表驱动覆盖 string/bool/int/int64/float64 + 结构体兜底，断言 Key 与 Emit 输出。

goleak + race

BatchSpanProcessor/PeriodicReader 的后台 goroutine 须随 Shutdown 退出，goleak.VerifyTestMain 把关。

09设计要点回顾

trace = Agent 的 X 光

把长链路黑盒变成可视化调用树，回答慢/贵/错在哪。

按收益取舍

MCP 自实现、trace 用 SDK——同一套权衡框架，不是教条。

ctx 双重职责

取消与 trace 共用一条 ctx 管线，父子 span 零成本自动串联。

换内胆不换接口

OTel 类型不外泄，内核零改动、可单测、可替换。

零开销降级

默认 no-op，调用点恒在但无 if 分支污染。

接入标准生态

一行 env 切 Jaeger 火焰图，可观测价值最大化。

STAR（工程要点）：

S：Agent 任务长链路、多轮、并发，跑完一个 90s 的任务只知道"慢且贵"，定位不到瓶颈与坏 case。

T：用后端最成熟的可观测手段把决策链变成可视化调用树。

A：把 ReAct 循环建模为 OTel span 树（cogent.session→react.step→llm.stream/tool.call/agent.spawn/ctx.compact）；复用 context.Context 让 span 自动父子串联、与取消传播共用一条管线；observe 包封住 OTel 类型、对内核零暴露，Enabled=false 时 no-op 零开销；exporter 支持 file/stdout/otlp，一行 env 切 Jaeger 看火焰图；slog 注入 trace_id 与 span 对齐；goleak+race 验证后台导出 goroutine 随 Shutdown 退出。

R：一次任务即得完整 span 树，能精确定位"80s 耗在第 7 轮某 bash、12s 在首 token 延迟"；token/成本按轮归因；这套把后端可观测经验直接迁移到 Agent，是区别于"框架调用 Demo"的硬工程信号。

诚实边界 / 取舍说明：本阶段有意打破前序"零新增依赖"原则引入 OTel SDK（主 go.mod 多一批传递依赖）——理由是 trace 的价值在于接入标准可视化生态（Jaeger），自造格式得不偿失。指标侧 v1 落最小集（轮数/工具调用/token），未接 Prometheus/Grafana 看板；OTLP 默认明文本地连接，生产需配 TLS。
