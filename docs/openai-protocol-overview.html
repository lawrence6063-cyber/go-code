OpenAI Chat Completions 协议

流式（SSE）vs 非流式 · 请求 & 响应结构梳理

概览对比 非流式 流式 工具调用分片 finish_reason 取值 工具调用完整往返 messages 字段 & tools 区别 字段速查表

非流式 stream: false

一次性等待模型生成完毕，返回完整 JSON。适合后台批处理、简单调用。

Client ──POST──▶ OpenAI ──等待生成完毕──▶ 完整 JSON

流式 stream: true

SSE 逐 token 推送，client 边收边拼接。适合实时打字效果 + 可中断的 Agent 循环。

Client ──POST──▶ OpenAI ──多帧 SSE──▶ 增量拼接
① 非流式请求 & 响应

请求（Request）

{
  "model": "gpt-4",
  "messages": [
    { "role": "user", "content": "帮我读一下 a.go" }
  ],
  "tools": [ /* function schema，可选 */ ],
  "stream": false  // 默认值，可省略
}

响应（Response）— 单个完整 JSON，一次性返回

{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1677858242,
  "model": "gpt-4",
  "choices": [
    {
      "index": 0,
      "message": {  // 注意：是完整 "message"，不是 "delta"
        "role": "assistant",
        "content": "好的，我来读取这个文件",
        "tool_calls": [
          {
            "id": "call_abc",
            "type": "function",
            "function": {
              "name": "read_file",
              "arguments": "{\"path\":\"a.go\"}"  // 已是完整 JSON
            }
          }
        ]
      },
      "finish_reason": "tool_calls"
    }
  ],
  "usage": { "prompt_tokens": 100, "completion_tokens": 20, "total_tokens": 120 }
}
特点说明
关键字段choice.message（完整消息对象，含 content + tool_calls）
拿到结果一次 HTTP 响应 body 即完整答案，无需拼接
中途取消只能整体取消 HTTP 请求，无法"看到一半就停"
适用场景后台自动化任务、无 UI 展示需求、逻辑简单的单轮调用
② 流式请求 & 响应（SSE）

请求（Request）— 仅多了两个字段

{
  "model": "gpt-4",
  "messages": [ ... ],
  "tools": [ ... ],
  "stream": true,                          // ★ 开启流式
  "stream_options": { "include_usage": true }  // ★ 末尾帧附带 usage
}

响应（Response）— SSE（Server-Sent Events）协议

流式响应不是一个 JSON，而是一个 HTTP 长连接，body 遵循 SSE 规范持续推送。

① 响应头（Response Headers）

HTTP/1.1 200 OK
Content-Type: text/event-stream       // ★ 标识这是 SSE 流，不是普通 JSON
Cache-Control: no-cache
Connection: keep-alive            // ★ 保持连接不关闭，持续推送
Transfer-Encoding: chunked         // 分块传输，服务端边生成边发送

② SSE 原始报文格式规则

规则说明
事件前缀每条事件以 data: （英文冒号+一个空格）开头
事件分隔每条事件后跟 两个换行符（\n\n）表示一条事件结束
payload 格式data: 后面是一整行 JSON 字符串（本身不能包含未转义的换行）
结束标志最后一条固定为 data: [DONE]\n\n（不是 JSON，是纯文本字面量）
其他字段（可选）SSE 规范还支持 event: / id: / retry: 字段，但 OpenAI 只用 data:

③ 原始报文示例（TCP 层面实际收到的字节流）

data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"index":0,"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"index":0,"delta":{"content":"好"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"index":0,"delta":{"content":"的"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc","object":"chat.completion.chunk","choices":[{"index":0,"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":10,"completion_tokens":2,"total_tokens":12}}

data: [DONE]

Client 侧按行读取，遇到 data: 前缀就取出后面的内容：如果是 [DONE] 就结束流；否则 json.Unmarshal 成一个 ChatCompletionChunk 对象，再从 choices[0].delta 里取增量。

④ 语义化拆解（下面把关键帧的 delta 字段单独列出，方便理解每帧携带了什么）

帧 1
delta: {"role":"assistant"} 首帧标识角色，通常无内容
帧 2
delta: {"content":"好"} TEXT
帧 3
delta: {"content":"的，我来"} TEXT
帧 4
delta: {"tool_calls":[{"index":0,"id":"call_abc","function":{"name":"read_file","arguments":""}}]} TOOL 首片
帧 5
delta: {"tool_calls":[{"index":0,"function":{"arguments":"{\"pa"}}]} TOOL 分片
帧 6
delta: {"tool_calls":[{"index":0,"function":{"arguments":"th\":\"a.go\"}"}}]} TOOL 分片
帧 7
delta: {}, finish_reason: "tool_calls" 结束标志：本轮请求工具调用
帧 8
usage: {"prompt_tokens":100,"completion_tokens":20} 末尾附带用量（因 include_usage:true）
EOF
data: [DONE] 流结束标志（原始 SSE 协议约定）
特点说明
关键字段choice.delta（增量片段，需 client 自己拼接）
拿到结果多次网络帧，text 逐 token 拼接，tool_calls 按 index 累积后组装
中途取消可以随时关闭连接（ctx cancel），已花费的 token 立即停止
适用场景CLI/Web 实时打字效果、Agent 循环（需可中断 + 低首字延迟）
③ 工具调用（tool_calls）分片累积逻辑

流式下 tool_calls 的 arguments 是逐片 JSON 字符串拼接的，中间态可能是不完整的 JSON（如 {"pa）， 必须等 finish_reason == "tool_calls" 才能安全地把所有分片拼成完整对象再解析。

帧4: id + name ＋ 帧5: args 分片 ＋ 帧6: args 分片 ═══finish_reason═══▶ 完整工具调用对象 
// 累积后组装出的完整调用（可安全 json.Unmarshal arguments）
{
  "id": "call_abc",
  "name": "read_file",
  "arguments": "{\"path\":\"a.go\"}"
}
④ finish_reason 取值一览
值触发条件
"stop"模型自然结束（遇到停止符或正常说完），无工具调用
"tool_calls"模型决定调用一个或多个工具，本轮已生成完整调用
"length"达到 max_tokens 限制被截断，回复可能不完整
"content_filter"内容被安全过滤器拦截/截断
"function_call"已废弃的旧版单函数调用格式（现统一用 tool_calls）
null流式传输中间帧：本轮尚未结束（只有末帧才带值）
⑤ 工具调用完整往返（两轮 HTTP 请求）

工具调用不是一次请求搞定：模型只能"请求"调用工具，真正执行是 client（agent）自己做的。 完整流程需要 两轮 HTTP 请求：

Req①: user 消息 + tools 声明 ──▶ Resp①: tool_calls ──client 本地执行工具──▶ Req②: 追加 assistant(tool_calls) + tool(result) ──▶ Resp②: 最终文本

Req① — 首次请求（声明 tools）

{
  "model": "gpt-4",
  "messages": [
    { "role": "user", "content": "帮我读一下 a.go" }
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "description": "读取指定路径文件内容",
        "parameters": {
          "type": "object",
          "properties": { "path": { "type": "string" } },
          "required": ["path"]
        }
      }
    }
  ]
}

Resp① — 模型请求调用工具（非流式简化展示）

{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc",
        "type": "function",
        "function": { "name": "read_file", "arguments": "{\"path\":\"a.go\"}" }
      }]
    },
    "finish_reason": "tool_calls"
  }]
}

▸ client 本地执行 read_file({"path":"a.go"})，得到结果 "package main..."

Req② — 携带 assistant(tool_calls) + tool(result) 追加历史，再次请求

{
  "model": "gpt-4",
  "messages": [
    { "role": "user", "content": "帮我读一下 a.go" },
    {  // ★ 把 Resp① 的 assistant 消息原样放回去
      "role": "assistant",
      "content": null,
      "tool_calls": [{ "id": "call_abc", "type": "function", "function": { "name": "read_file", "arguments": "{\"path\":\"a.go\"}" } }]
    },
    {  // ★ 新增：工具执行结果，role 必须是 "tool"
      "role": "tool",
      "tool_call_id": "call_abc",  // 必须与 Resp① 里的 id 对应
      "content": "package main\n\nfunc main() {...}"
    }
  ],
  "tools": [ /* 同 Req①，通常继续携带，方便模型再次调用 */ ]
}

Resp② — 模型基于工具结果给出最终回复

{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "文件 a.go 的内容是一个简单的 main 函数，仅打印..."
    },
    "finish_reason": "stop"
  }]
}
关键约束说明
配对完整性每个 assistant.tool_calls[i].id 必须有且仅有一条对应 role:"tool" 消息，tool_call_id 精确匹配
顺序要求tool 消息必须紧跟在对应的 assistant(tool_calls) 消息之后
并行调用一次 Resp① 可能同时返回多个 tool_calls（不同 index），Req② 需为每个都追加对应的 tool 结果消息
可能多轮Resp② 也可能再次是 finish_reason:"tool_calls"（继续调用工具），如此循环直至 "stop"
⑥ messages 对象字段 & tools 与 tool_calls 的区别

请求体是扁平结构：model / messages / tools / stream 等都是 顶层平级字段，tools 不属于 messages。二者常被混淆，但语义完全不同： tools 是"这轮对话模型可以使用哪些工具"的能力声明；messages[].tool_calls 是 "模型这轮实际决定调用了哪个工具"的调用记录，只会出现在 role: "assistant" 的消息里。 

{
  "model": "gpt-4",          // 顶层字段
  "messages": [ ... ],  // 顶层字段：对话历史数组
  "tools": [ ... ],     // 顶层字段：与 messages 平级，声明"有哪些工具可用"
  "tool_choice": "auto",  // 顶层字段：auto / none / required / 指定某个工具
  "stream": false          // 顶层字段
}

上面只是与 messages/tools 关系最相关的几个，请求顶层字段远不止这些，按用途完整分类如下：

分类字段说明
必需model / messages模型名 + 对话历史
工具调用tools / tool_choice / parallel_tool_calls可用工具声明 / 调用策略（auto/none/required/指定） / 是否允许一次并行返回多个调用（默认 true）
采样与生成控制temperature / top_p / max_tokens（新版 max_completion_tokens） / n / stop / presence_penalty / frequency_penalty / seed / logit_bias控制随机性、输出长度、候选数量、重复惩罚、可复现性等
流式相关stream / stream_options是否 SSE 推送 / 流式附加选项（如 include_usage）
输出格式response_format强制输出合法 JSON（json_object）或按 JSON Schema 严格约束（json_schema）
可观测/审计user / metadata / store / logprobs / top_logprobs终端用户标识、附加元数据、是否留存用于评测蒸馏、返回 token 概率
特殊模型专属reasoning_effort / modalities / audio / predictiono 系列推理强度、多模态语音输出、预测输出加速
已废弃functions / function_call被 tools / tool_choice 取代，勿再使用

messages[] 数组中每个消息对象的字段：

字段出现在哪种 role说明
role全部system / user / assistant / tool（旧版还有已废弃的 function）
content全部（tool_calls 场景下 assistant 可为 null）文本内容，也可以是多模态内容数组（文本+图片）
name可选，user/system标识消息发送者名称，较少使用
tool_calls仅 assistant本轮模型发起的工具调用列表（id / type / function.name / function.arguments）
tool_call_id仅 tool对应 assistant.tool_calls[].id，标明这是哪次调用的结果
refusal较新，仅 assistant模型拒答时的说明文本（安全策略触发）
tools（顶层字段）messages[].tool_calls（消息内字段）
位置请求顶层，与 messages 平级某条 role:"assistant" 消息内部
方向Client → Model：告知模型"有哪些工具可用"Model → Client：模型返回"我要调用哪个工具"
内容工具的 JSON Schema 定义（name/description/parameters）具体调用实例（id + name + 已填好的 arguments）
出现时机每次请求都可携带（尤其 Req①、Req②通常都带）只在模型决定调用工具时的响应里出现

role 字段只存在于 messages[] 数组的每个对象里——它不是请求顶层字段， tools/tool_choice 等顶层字段本身没有 role。响应里 choices[0].message 其实也是同一种 "消息对象"结构（只是永远固定为 role:"assistant"）。完整对话共有 4 种角色，下面分别给出每种角色单独的 Req / Resp 长什么样：

role出现在 Req？出现在 Resp？由谁产生 / 何时出现
system✅ 会❌ 不会开发者预设的人设/规则，每次请求固定携带，模型永远不会"回复"一条 system 消息
user✅ 会❌ 不会终端用户的输入，API 只会"消费"它，不会在 Resp 里回传
assistant✅ 会（历史回填）✅ 会（本轮结果）模型生成；出现在 Resp 后，client 若要继续对话，需把它原样放回下一次 Req 的 messages[] 尾部
tool✅ 会（工具结果）❌ 不会client 在本地执行完 tool_calls 后自己构造并放进下一次 Req，API 从不会在 Resp 里直接返回 role:"tool" 的消息

role: system

只会出现在 Req 里，Resp 永远不会返回它

Req 片段

{ "role": "system",
  "content": "你是一个专业的代码助手" }

Resp：（无——不存在该角色）

role: user

只会出现在 Req 里，Resp 永远不会返回它

Req 片段

{ "role": "user",
  "content": "北京今天天气怎么样" }

Resp：（无——不存在该角色）

role: assistant

Resp 唯一会返回的角色；下一轮需回填进 Req

Resp 片段（choices[0].message）

{ "role": "assistant",
  "content": "北京今天晴，25℃" }

Req 片段（原样回填到 messages[]）

{ "role": "assistant",
  "content": "北京今天晴，25℃" }

role: tool

只会出现在 Req 里，由 client 自己构造，Resp 永远不会返回它

Req 片段（工具执行结果）

{ "role": "tool",
  "tool_call_id": "call_xyz",
  "content": "{\"temp\":25,\"cond\":\"晴\"}" }

Resp：（无——不存在该角色）

Req — 把上面 4 种角色拼在一起看：包含全部 4 种 role 的完整 messages[]（工具调用已执行完毕，正在请求最终回复）

{
  "model": "gpt-4",
  "messages": [
    { "role": "system", "content": "你是一个代码助手，可以读写文件" },  // ① 人设，开发者固定注入
    { "role": "user", "content": "帮我读一下 a.go" },  // ② 用户输入
    {  // ③ 上一轮 Resp 里模型返回的 assistant 消息，原样回填
      "role": "assistant",
      "content": null,
      "tool_calls": [{ "id": "call_abc", "type": "function", "function": { "name": "read_file", "arguments": "{\"path\":\"a.go\"}" } }]
    },
    {  // ④ client 本地执行工具后自己构造，从未由 API 直接返回
      "role": "tool",
      "tool_call_id": "call_abc",
      "content": "package main\n\nfunc main() {...}"
    }
  ]
}

Resp — 只会新增一条 role:"assistant"，绝不会出现 system/user/tool

{
  "choices": [{
    "message": {
      "role": "assistant",  // ⑤ Resp 里唯一可能出现的 role
      "content": "文件内容是一个简单的 main 包，包含 main 函数"
    },
    "finish_reason": "stop"
  }]
}
一句话总结： Req 的 messages[] 是"全量历史"，四种 role 都可能出现； Resp 永远只吐出一条 role:"assistant" 消息——它既是给用户看的最终答案，也是下一轮 Req 里要回填的历史片段。 system/user/tool 都只能由 client 端"喂"给模型，模型自己永远不会"扮演"这三种角色。
⑦ 字段速查表

delta 和 message 结构上长得几乎一样（都可能有 role/content/tool_calls）， 但语义完全不同：message 是"完整快照"，delta 是"本帧新增的增量"。二者不会同时出现—— 非流式响应的 choices[0] 只有 message；流式的每一帧 choices[0] 只有 delta。

message（非流式）

一次性给出完整值，直接能用

{
  "role": "assistant",
  "content": "好的，我来读取",
  "tool_calls": [{
    "id": "call_abc",
    "type": "function",
    "function": {
      "name": "read_file",
      "arguments": "{\"path\":\"a.go\"}"  // 完整JSON
    }
  }]
}

delta（流式，某一帧）

只带本帧新增的字段，其余字段直接不存在（不是 null）

{
  "tool_calls": [{
    "index": 0,  // ★ message 里没有，delta 独有
    "function": {
      "arguments": "th\":\"a.go\"}"  // 只是JSON字符串的一小段碎片
    }
    // 本帧没有 id/type/function.name，因为它们已在更早的帧出现过
  }]
}
维度message（非流式 / 流式收尾累积后）delta（流式，单帧）
本质完整快照，可直接用相对上一帧的增量，需要累积才能用
role必定存在只在第 1 帧出现一次，后续帧不再重复
content完整字符串（或 null）每帧是文本的一个片段，需按帧顺序拼接
tool_calls[].id/type/function.name始终完整存在只在该工具调用的首个分片帧出现，后续分片帧没有这些字段
tool_calls[].function.arguments完整、合法的 JSON 字符串JSON 字符串的碎片，中间态通常不是合法 JSON
tool_calls[].index没有这个字段（数组下标本身就是序号）message 没有的独有字段：多个工具并行调用时，用它标识"这个分片属于第几个调用"，是跨帧对齐拼接的关键
缺省表现字段值为空时是 null（如无工具调用时 tool_calls 不出现，但 content 显式为字符串）字段没有新增时直接不出现在这一帧 JSON 里（省流量），不是显式 null
何时可用拿到即用必须等 finish_reason 出现（尤其 tool_calls 场景）才能安全使用累积结果
字段路径非流式流式含义
choices[0].message✅ 完整消息对象❌ 不存在模型完整回复（content + tool_calls）
choices[0].delta❌ 不存在✅ 增量片段本帧新增的部分内容
delta.content / message.content完整文本逐 token 文本片段模型文本输出
tool_calls[].function.arguments完整 JSON 字符串JSON 字符串分片，需拼接工具调用入参
choices[0].finish_reason始终携带仅收尾帧携带stop / tool_calls / length / content_filter
usage始终携带需 stream_options.include_usage:true，仅末帧携带token 用量统计
核心要点回顾： 非流式响应用 choices[0].message 一次性给出完整消息；流式响应用 choices[0].delta 分帧给出增量片段，text 逐 token 拼接、tool_calls 按 index 累积到 finish_reason:"tool_calls" 才算完整。 工具调用永远是「模型只请求、client 自己执行」的两轮往返模式：Resp① 拿到 tool_calls → 本地执行 → Req② 把 assistant(tool_calls) 和 tool(result) 一起放回 messages → Resp② 得到最终回复， 如此循环直至 finish_reason:"stop"。
