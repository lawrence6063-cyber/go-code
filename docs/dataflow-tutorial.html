0. 先搞清楚：谁是谁？

整个 MCP 模块只有 两个角色，别想复杂了：

🧠 CLIENT（客户端）
cogent 主程序
你正在用的 AI 编码助手
它发起请求、消费工具
stdin (请求) →
⇄
← stdout (响应)
🔧 SERVER（服务端）
外部子进程
如 npx @some/mcp-server
它被启动、响应请求、提供工具
一句话记住：cogent 是老板（Client），MCP Server 是打工人（被 cogent 拉起来的子进程）。
老板通过 stdin/stdout 管道给打工人下指令，打工人干完活把结果写回来。

1. 全局鸟瞰：5 个文件各管什么

文件 职责（一句话） 类比
config.go 读 JSON 配置 → 告诉系统要连哪些 Server 📋 通讯录
manager.go 批量连接 Server、聚合工具、统一关闭 👔 部门经理
client.go 管理单个 Server 的生命周期（启动→握手→调用→关闭） 📞 专线电话
transport.go 底层 JSON-RPC 双工通信（发消息、收消息、配对） 📡 电话线路
remote_tool.go 把远端工具"伪装"成本地工具，融入 cogent 工具池 🔌 转接头
它们的调用关系（从上到下）： 
外部调用方 (cmd 层)
    │
    ▼
config.go  ──→  读文件，返回 []ServerConfig
    │
    ▼
manager.go ──→  拿着配置，逐个创建 client 并连接
    │
    ▼
client.go  ──→  exec.Command 启动子进程，用 transport 通信
    │
    ▼
transport.go ──→  最底层：往 stdin 写 JSON，从 stdout 读 JSON
    │
    ▼
remote_tool.go ──→  把拿到的工具信息包装成 tool.Tool 接口

2. 第一步：读配置（config.go）

输入：一个 JSON 文件
位置固定在项目根目录 .cogent/mcp.json
// .cogent/mcp.json 长这样：
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "xxx" }
    },
    "filesystem": {
      "command": "python",
      "args": ["fs_server.py"]
    }
  }
}
输出：一个 Go 结构体切片
按名字排序，保证每次启动顺序一致
[]ServerConfig{
    {Name: "filesystem", Command: "python", Args: ["fs_server.py"]},
    {Name: "github",     Command: "npx",    Args: ["-y", "@model..."]},
}
⚡ 关键设计：文件不存在 → 返回 (nil, nil)，不报错！
MCP 是"锦上添花"，没有配置文件 cogent 照样能跑。

3. 第二步：Manager 批量连接（manager.go）

Manager 是个"包工头"，拿着通讯录（配置列表），逐个去连接 Server：

Manager.Connect(ctx, configs) │ ├── connectOne(github) → 成功 ✓ 加入 clients 列表 │ ├── connectOne(filesystem) → 成功 ✓ 加入 clients 列表 │ └── connectOne(broken) → 失败 ✗ 打印警告，跳过，不崩溃！ Manager.Tools() → 把所有 client 的工具汇总成一个大列表 Manager.Close() → 逐个关闭，错误汇总返回
🛡️ 容错策略：一个 Server 挂了不影响其他 Server，也不影响 cogent 启动。
这就像你手机里装了 10 个 App，其中一个打不开，不影响你用其他的。

对应代码（精简版）：

func (m *Manager) connectOne(ctx context.Context, cfg ServerConfig) {
    cl, err := NewClient(m.transport, m.tracer)
    if err != nil {
        slog.Warn("mcp client init failed", ...)  // 警告，不 panic
        return
    }
    if err := cl.Connect(ctx, cfg); err != nil {
        slog.Warn("mcp connect failed", ...)      // 警告，不 panic
        _ = cl.Close()
        return
    }
    m.clients = append(m.clients, cl)              // 成功才加入
}

5. 第四步：Transport 双工通信（transport.go）

Transport 是最底层的"电话线路"，负责：发 JSON、收 JSON、把请求和响应配对。

🎯 核心机制：请求-响应配对

每个请求有唯一递增 ID，响应带着同样的 ID 回来，transport 按 ID 把响应交给正确的等待者。

┌─────────────────────────────────────────────────────────────────┐ │ transport 内部结构 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ 【写出方向】 │ │ │ │ call("tools/list", params) │ │ │ │ │ ├── 分配 ID (原子递增: 1, 2, 3...) │ │ ├── 注册到 pending map: pending[id] = chan frame │ │ ├── JSON 序列化: {"jsonrpc":"2.0","id":1,"method":...} │ │ ├── 追加换行符 \n │ │ └── 加锁写入 stdin ──────────────→ Server 子进程 │ │ │ │ 【读入方向】（独立 goroutine: readLoop） │ │ │ │ Server 子进程 ──────────────→ stdout │ │ │ │ │ ├── bufio.Scanner 逐行扫描 │ │ ├── JSON 反序列化为 frame │ │ ├── 取出 frame.ID │ │ └── deliver(id, frame): 从 pending 取出 chan，投递进去 │ │ │ │ 【等待机制】 │ │ │ │ call() 发完请求后 select: │ │ ├── case <-ctx.Done() → 超时/取消 │ │ ├── case <-t.done → 连接已断开 │ │ └── case f := <-ch → 收到响应！返回 f.Result │ │ │ └─────────────────────────────────────────────────────────────────┘

打个比方 🏪

想象一个取号排队的餐厅：

1. 你点餐时拿到一个号码牌（ID=7）
2. 你的订单被送进厨房（写入 stdin）
3. 你在座位上等着（select 阻塞）
4. 厨房做好了喊号："7号！"（readLoop 读到 ID=7 的响应）
5. 服务员把餐送到你桌上（deliver 投递到你的 channel）
6. 你拿到食物开吃（call 返回结果）

对应代码（精简版）：

func (t *transport) call(ctx context.Context, method string, params any) (json.RawMessage, error) {
    id := t.nextID.Add(1)                // ① 取号
    ch := make(chan frame, 1)
    t.pending[id] = ch                   // ② 注册等待

    t.write(rpcRequest{ID: id, ...})     // ③ 发送请求

    select {                              // ④ 等待响应
    case <-ctx.Done():  return nil, ctx.Err()
    case <-t.done:      return nil, errors.New("closed")
    case f := <-ch:     return f.Result, nil  // ⑤ 拿到结果
    }
}

6. 第五步：remoteTool 适配器（remote_tool.go）

这是整个模块的"最后一公里"——把远端工具伪装成本地工具。

🔌 问题：cogent 工具池只认 tool.Tool 接口

远端 Server 返回的只是一堆 JSON 描述（名字、描述、schema），不是 Go 接口。
remoteTool 就是那个"转接头"，让远端工具看起来和内建工具一模一样。

cogent 工具池要求每个工具实现： tool.Tool 接口 ├── Name() string → "mcp__github__search_code" ├── Description() string → "Search code in repos" ├── InputSchema() RawMessage → {"type":"object","properties":...} ├── IsConcurrencySafe() → false (默认保守) ├── IsReadOnly() → false (默认保守) ├── CheckPermission() → ask (默认要确认) └── Call(ctx, input, sink) → 通过 client.callTool() 远程调用

命名隔离策略

// 工具名格式：mcp__<server名>__<原始工具名>
// 例如：
ToolName("github", "search_code")  →  "mcp__github__search_code"
ToolName("fs",     "read_file")    →  "mcp__fs__read_file"
为什么要加前缀？
防止外部 Server 的工具名和 cogent 内建工具冲突。
比如 cogent 自己有个 read_file 工具，外部 Server 也叫 read_file，
加了前缀后变成 mcp__fs__read_file，就不会打架了。
而且工具池构造时"内建优先去重"，同名的内建工具永远优先。

错误处理策略（fail-closed）

func (t *remoteTool) Call(ctx, input, _) (types.ToolResult, error) {
    res, err := t.client.callTool(ctx, t.origName, input)
    if err != nil {
        // 注意！不向上抛 error，而是包装成"错误结果"
        return types.ToolResult{
            Content: "mcp call failed: " + err.Error(),
            IsError: true,
        }, nil  // ← error 返回 nil！
    }
    return res, nil
}
为什么错误不上抛？
如果返回 error，上层可能会中断整个对话流程。
包装成 ToolResult{IsError: true} 后，AI 模型会看到"这个工具调用失败了"，
然后它可以选择重试或换个方式——不会让整个系统崩溃。

7. 运行时：一次工具调用的完整旅程

假设 AI 模型决定调用 mcp__github__search_code，数据是怎么流动的？

AI 模型: "我要调用 mcp__github__search_code，参数是 {query: 'auth'}" │ ▼ cogent 工具池: pool.Get("mcp__github__search_code") → 找到 remoteTool │ ▼ remoteTool.Call(ctx, input, sink) │ │ 开启 trace span: "mcp.call" │ ▼ client.callTool(ctx, "search_code", input) │ │ 构造参数: {name: "search_code", arguments: {query: "auth"}} │ ▼ transport.call(ctx, "tools/call", params) │ │ 分配 ID=42 │ 序列化: {"jsonrpc":"2.0","id":42,"method":"tools/call","params":{...}} │ 写入 stdin │ ▼ ═══════════════ 跨进程边界 ═══════════════ │ ▼ MCP Server 子进程 │ 从 stdin 读到请求 │ 执行 search_code 逻辑（比如调 GitHub API） │ 构造响应: {"jsonrpc":"2.0","id":42,"result":{...}} │ 写入 stdout │ ▼ ═══════════════ 跨进程边界 ═══════════════ │ ▼ transport.readLoop │ 扫描到一行 JSON │ 解析出 ID=42 │ deliver(42, frame) → 投递到 pending[42] 的 channel │ ▼ transport.call 的 select 收到 channel 数据 │ │ 返回 frame.Result (json.RawMessage) │ ▼ client.callTool │ json.Unmarshal → callToolResult{Content: [...], IsError: false} │ 拼接文本内容 │ ▼ remoteTool.Call │ 结束 trace span │ 返回 types.ToolResult{Content: "搜索结果...", IsError: false} │ ▼ cogent 工具池 → 把结果交给 AI 模型继续对话

8. 关闭阶段：优雅退出

cogent 退出 │ ▼ Manager.Close() │ ├── client1.Close() │ │ │ ├── transport.closeStdin() 关闭写端 │ │ │ │ │ ▼ │ │ Server 收到 EOF → 自行退出 │ │ │ ├── waitProcess() 等待子进程退出 │ │ │ │ │ ├── 2秒内退出 → OK │ │ └── 超时 → Kill 强杀 │ │ │ └── transport.waitDone() 等待 readLoop 退出 │ ├── client2.Close() │ └── (同上) │ └── 汇总所有错误返回

9. 设计哲学总结

设计原则体现在哪里效果
Fail-Closed remoteTool 默认非并发安全、非只读、权限 ask 忘记配置 = 最安全行为，不会意外放开权限
容错降级 Manager 单 Server 失败仅 Warn 跳过 一个 Server 挂了不影响整体
错误不上抛 remoteTool.Call 把 error 包装为 ToolResult 单次调用失败不中断对话流
可插拔 配置文件不存在返回 nil，MCP 非硬依赖 没有 MCP 配置 cogent 照常运行
命名隔离 工具名 mcp__<server>__<tool> 前缀 外部工具无法冒用内建工具名
零依赖 自实现 JSON-RPC，不引入官方 SDK 生产二进制精简，无第三方依赖
🎓 最终总结（背下来）：

配置文件 → Manager 批量连接 → Client 拉起子进程 + stdio 握手
→ Transport 双工 JSON-RPC 通信 → remoteTool 适配为 tool.Tool
→ 融入工具池，运行时按需调用

整个模块就是一个 "把外部进程的能力，通过管道通信，伪装成本地工具" 的适配层。
