0这一步在做什么

Loop Engineering「六原语」里最后三块拼图，本阶段一次补齐：

L4-1/2 · Worktrees

用 git worktree 给每个 maker 一个独立目录 + 临时分支，物理隔离。审查通过才 Merge 回基线，否则整个 worktree Discard——「通过才落盘」从「写了再回滚」升级为「干净的分支合并」。

L4-3 · Skills

扫描 .cogent/skills/<name>/SKILL.md，按相关性召回注入目标意图。把「如何给本项目加限流中间件」这类隐性流程，固化为可版本化、可复用的技能包。

L4-4 · Connectors

调用外部 MCP server 时把当前 span 的 traceparent 经环境变量注入子进程，让支持 OTel 的 server 续接同一条 trace，不支持则优雅降级。

延续铁律：叠加而非重写。engine / tool / sandbox 一行没改——新增 worktree、skills 两个叶子包，observe 加一个传播 helper，mcp 在握手处注入 env，cmd 加一个 --worktree 装配分支。

1Worktrees：物理隔离 ≠ 路径约束

v1 的所有文件工具都直接在唯一的 WorkRoot 读写，sandbox.ValidatePath 只做越界防护（别写出仓库），却不做隔离。两个 maker 并行改同一目录，就会互相覆盖——后写的赢，先写的丢。

worktree 共享同一份 .git 对象库（磁盘开销小、切换快），但每个工作树是物理独立目录——天然成为各自的 sandbox.WorkRoot，越界防护照常生效。

Manager：三个动作，全经沙箱白名单 git

internal/worktree 是只依赖 sandbox 与标准库的叶子包。所有 git 操作都经注入的 sandbox.Sandbox 执行（白名单命令 + 危险命令拦截 + 工作目录约束），模型工具绝不能直接跑 git worktree。

internal/worktree/worktree.gotype Manager interface {
    // 从 baseRef 派生独立目录 + 临时分支
    Create(ctx context.Context, baseRef string) (Workspace, error)
    // 审查通过：提交 worktree 改动 → 合并回基线；冲突→ErrMergeConflict
    Merge(ctx context.Context, ws Workspace, baseRef string) error
    // 未通过/取消：清理目录 + 删临时分支
    Discard(ctx context.Context, ws Workspace) error
}
冲突即停，不强推。Merge 检测到 git merge 非零退出 → 执行 git merge --abort 并返回包装 ErrMergeConflict 的错误。上层据此把待办落为 progress 看板的 Blocked，交人介入——无人值守 ≠ 无人兜底。
为什么 sandbox 用 Enabled=false？受限环境的 PATH 只有 /usr/local/bin:/usr/bin:/bin，跑不到 git。git 工具链是开发者可信的控制面，需继承宿主 PATH——但危险命令拦截与 WorkRoot 约束仍然生效（与 L2 的 gitDiscarder 同构）。

2「通过才落盘」从 diff 回滚升级为 worktree 暂存

L2 的落盘策略是「maker 直接改工作区，reviewer 不通过就 git checkout -- . 回滚」——能用，但有「回滚不彻底」的风险（漏掉新建文件、untracked 残留）。L4 把它升级为更干净的方案 A：worktree 暂存。

主观审（reviewer）通过 → Merge 落盘 → 客观验（verify.sh）再把关。两道闸门都过才算「达标」；任一不过都带反馈续跑。合并冲突降级为「本轮未通过」。

cmd 层编排：为什么不动 agent 核心？

难点在于 engine.Deps.WorkRoot 在装配期就固定了，而 maker 必须在运行期才知道的 worktree 目录里工作。解法是在 cmd 层放一个 worktreePipeline 适配器（实现 loop.Pipeline），每轮先 Create、再用新目录重建 maker/reviewer：

cmd/cogent/review.go（简化）func (p *worktreePipeline) Iterate(ctx, task) (loop.PipelineResult, error) {
    ws, _ := p.mgr.Create(ctx, p.baseRef)        // 派生隔离 worktree
    r, _ := p.build(ws.Root).Iterate(ctx, task) // 按 wt 根重建双角色再跑
    if !r.Verdict.Approved {
        p.mgr.Discard(ctx, ws)                  // 未通过：物理丢弃，主区零残留
        return notApproved(r), nil
    }
    if err := p.mgr.Merge(ctx, ws, p.baseRef); err != nil {
        p.mgr.Discard(ctx, ws)
        if errors.Is(err, worktree.ErrMergeConflict) {
            return conflictFeedback(r, err), nil // 降级续跑 → Blocked
        }
        return loop.PipelineResult{}, err
    }
    return approved(r), nil                     // 已 Merge 落盘
}
向后兼容：diff 回滚仍是默认行为，worktree 暂存由 --worktree flag 显式开启（隐含 --review）。blast radius 受控，老路径零变化。

3Skills：把「怎么做」固化到磁盘

v1 只有 MEMORY.md——那是被动注入的记忆（事实/约定），不是可按任务召回的流程包。Skills 补上后者：一个目录一个 SKILL.md，描述「某类任务怎么做」。

MEMORY.md · 事实 / 约定

「本项目用 zap 做日志」「禁止提交 vendor」——被动注入到系统提示，每次都在。

SKILL.md · 流程 / 做法

「如何给本项目加限流中间件：1…2…3…」——按任务召回，命中才注入，不撑爆上下文。

两段式：先索引，后召回（沿用 memory 范式）

internal/skills/skills.gotype Loader interface {
    // 轻量索引：只读 Name + 一行 Description，不读正文
    List(ctx, workRoot) ([]Skill, error)
    // 按需召回：命中相关性时才加载 Body（按上限硬截断）
    Load(ctx, workRoot, name) (Skill, error)
}

// 最简关键词重叠召回，不引向量库（延续 v1 取舍）
func Relevant(skills []Skill, query string, max int) []Skill

cmd 层的 augmentWithSkills 把命中的 SKILL.md 正文拼进目标意图——不碰 engine 固定的 system prompt（它是 const，不可注入），走任务侧文本，与 reviewer rubric 同构。

控制面写禁止（防注入持久化）：.cogent/skills/ 与 .cogent/ 一样属控制面，由人维护，模型工具不可写。否则「注入 → 写技能包 → 长期劫持」就成立了。Load 的 name 还过白名单正则 ^[A-Za-z0-9_-]+$，杜绝 ../ 路径穿越。

4Connectors：让 Trace 跨过进程边界

v1 的 MCP 是六原语里最完整的一项，但有个缺口：调用外部 MCP server（独立子进程）时，trace 在进程边界断开——你看不到「子进程里那次工具调用花了多久 / 为何失败」。L4 用 W3C traceparent 把链路接上。

stdio MCP 的 JSON-RPC 没有标准 trace 字段，env 注入是跨语言子进程传播的通用手段（OTel 官方推荐）。是「尽力而为」——对方不认就忽略，cogent 侧不受影响。

关键不变量：只有 observe 碰 OTel

mcp 包绝不直接 import OTel（守 §4.4 横切不变量）。提取 traceparent 的活儿封在 observe 里，mcp 只用标准 map 载体委托它：

internal/observe/propagation.go + mcp/trace.go// observe：用 propagation.TraceContext 写出 traceparent（OTel 类型不外泄）
func InjectTraceContext(ctx, carrier map[string]string) {
    traceContextPropagator.Inject(ctx, propagation.MapCarrier(carrier))
}

// mcp：委托 observe，把 traceparent → TRACEPARENT 环境变量
func InjectTraceContext(ctx, env map[string]string) {
    carrier := map[string]string{}
    observe.InjectTraceContext(ctx, carrier)
    if tp := carrier["traceparent"]; tp != "" { env["TRACEPARENT"] = tp } // 无 span 不注入
}
安全：traceparent 只含 trace/span id，不含任何敏感信息。注入用的是 cfg.Env 的副本（cloneEnv），不污染调用方配置。

5怎么跑起来

cogent loop --goal-file=.cogent/goal.md --worktree --interval=10m --max-cost=5

6安全与依赖不变量（守住底线）

worktree 越权防护

每个 worktree 是独立 WorkRoot，maker 文件操作仍过 ValidatePath，跨不出自己的工作树；git 命令只许经 Manager 的白名单。

控制面写禁止

.cogent/skills、.git/config 模型不可写；技能包由人维护，杜绝「注入→持久化→劫持」。

冲突 = Blocked，不强推

合并冲突落 progress.md 的 Blocked 状态等人介入——无人值守 ≠ 无人兜底。

trace 尽力而为

无 span 不注入、对方不认就忽略；丢 span 不影响正确性。

依赖方向铁律：worktree / skills 是叶子包（只依赖 sandbox/标准库）；mcp 只依赖 observe 薄接口、不碰 OTel；cmd → loop → engine → … → types 仍无环。go.mod 零新增第三方依赖（traceparent 复用既有 OTel 全家桶）。
